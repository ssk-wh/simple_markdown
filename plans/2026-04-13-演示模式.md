# 演示模式（Presentation Mode）

## 背景

当前项目已经实现了 **Focus Mode（专注模式）**：按 F11 进入全屏，隐藏菜单栏、Tab 栏、TOC 面板、预览区，**只保留编辑器**并启用打字机模式（光标居中）。代码位于 `src/app/MainWindow.cpp` 约 1637-1714 行的 `enterFocusMode` / `exitFocusMode`。

用户反馈的新需求：

> 我希望 F11 触发的不是"全屏编辑器"，而是"全屏预览区"——把渲染后的 Markdown 内容作为幻灯片向观众演示。

即：**Focus Mode 是为写作者设计的（放大编辑器）**，而 **Presentation Mode 是为演讲者设计的（放大预览区）**。两者的用途正交，但都要全屏、都要隐藏干扰元素、都要可恢复布局。

核心差别：

| 维度 | Focus Mode（现有） | Presentation Mode（新需求） |
|------|--------------------|-----------------------------|
| 保留显示 | 编辑器 | 预览区 |
| 打字机模式 | 启用 | 不适用 |
| 键盘用途 | 编辑文本 | 翻页/滚动查看 |
| 禁用快捷键 | 无（Ctrl+S 仍有效） | 大部分编辑快捷键 |
| 目标用户 | 作者 | 演讲者/阅读者 |

本 Plan 讨论如何从 Focus Mode 迁移/扩展到 Presentation Mode，并解决决策点。

### 决策 1：Focus Mode 保留还是替换？

**选项 A：替换（推荐）**

- 直接把 Focus Mode 改造成 Presentation Mode
- 删除 `enterFocusMode` / `exitFocusMode`
- 重命名 `m_focusMode → m_presentationMode`
- 菜单项"Focus Mode"改名"Presentation Mode"（中文"演示模式"）
- F11 快捷键绑定到新 action

**优点**：
- 代码简洁，没有两套互斥的状态
- 快捷键 F11 语义清晰
- 维护成本最低

**缺点**：
- 失去"全屏编辑器"功能，若有作者用户习惯使用 Focus Mode 会被冲击
- 从 CHANGELOG 看 Focus Mode 刚在 P2 完成（TODO 里标记"✅ 已完成，待用户确认"），推翻它有心理阻力

**选项 B：并列（保留两个模式）**

- 保留 Focus Mode（全屏编辑器）
- 新增 Presentation Mode（全屏预览）
- 快捷键拆分：F11 给 Presentation（用户明确要求），Focus Mode 改为 Ctrl+Shift+F11 或无快捷键
- 菜单项并列两个

**优点**：
- 既满足新需求，又不破坏现有功能
- 不同用户可以各取所需

**缺点**：
- 两套互斥状态逻辑复杂，容易漏判
- `eventFilter` 等地方要同时处理两种模式
- 菜单项变多，视觉凌乱
- "Focus Mode 改成 Ctrl+Shift+F11"用户一定会抱怨"为啥我的 F11 不好使了"

**选项 C：统一为单一枚举**

```cpp
enum class FullscreenMode { Normal, Focus, Presentation };
FullscreenMode m_fullscreenMode = FullscreenMode::Normal;
```

- 两个菜单项分别切换到不同模式
- `eventFilter` 通过 switch 分发
- F11 默认绑定 Presentation，Focus 另绑快捷键（或无）

**优点**：
- 清晰的互斥语义
- 便于将来扩展第三种全屏模式

**缺点**：
- 改动面比 A 大
- 仍需要拆分快捷键

### 建议

**优先采用选项 A（替换）**，理由：

1. 用户明确说"希望改成演示模式"——言外之意是 Focus Mode 可以被替代
2. 项目处于早期阶段（spec 大部分是 draft 状态），推翻重来成本低
3. TODO 里的"专注模式 — ✅ 已完成，待用户确认"条目可以直接更新为"演示模式"，不算破坏既有功能
4. 代码最简洁

若本次讨论中明确用户想保留 Focus Mode，则退化到**选项 B**，并强制 Focus Mode 改为 Ctrl+Shift+F11。

### 决策 2：菜单文案

若采用选项 A，菜单文案选项：

- "演示模式"（中文）/ "Presentation Mode"（英文）
- "幻灯片模式" / "Slideshow Mode"
- "演讲模式" / "Talk Mode"

**建议**："演示模式" / "Presentation Mode"。直白、对应 F11 的传统语义（PowerPoint / Keynote 都是 F5 触发演示，但桌面应用中 F11 全屏更常见）。

### 决策 3：预览滚动位置的保留精度

见 Spec INV-13 和 陷阱 8.8：

- **最小契约**：保留像素 y 值（简单，但宽度变化后位置不精确）
- **增强契约**：保留"视口最顶部的段落索引"，进入演示模式后滚动到该段落

**建议**：先做最小契约（简单实现 + 覆盖 T-5），如有用户反馈再升级。

### 决策 4：是否放宽 INV-9（允许 Ctrl+Tab 切换 Tab）

- 严格（默认）：演示模式下锁定当前 Tab，拒绝所有 Tab 切换
- 宽松：允许 Ctrl+Tab 切换 Tab，方便演讲时对比多文档

**建议**：先严格，满足 INV-9，后续根据反馈再放宽。

## 风险与回滚

### 风险 1：取消 Focus Mode 引起已有用户反弹

- **影响**：若有用户在用全屏编辑器写作，突然变成全屏预览会感到迷惑
- **缓解**：CHANGELOG 明确说明"Focus Mode 升级为 Presentation Mode"；退出后仍保留原布局
- **回滚**：git revert 或保留 Focus Mode 代码分支作为备份

### 风险 2：多显示器全屏定位错误

- **影响**：副屏按 F11 后窗口跳到主屏
- **缓解**：INV-11 显式绑定 QScreen；T-18 强制验证
- **回滚**：短期可接受"演示模式强制主屏"，长期通过 Qt 版本升级或 platform-specific 代码修复

### 风险 3：编辑快捷键禁用不完整

- **影响**：某些 QAction 漏禁用，演讲中误触发
- **缓解**：采用"白名单"策略——只保留翻页和退出 action，其他所有 action 批量禁用
- **回滚**：退出时逐一恢复

### 风险 4：翻页事件与预览区原有键盘处理冲突

- **影响**：原 PreviewWidget 可能已有 Space/PageDown 处理，新增 eventFilter 可能重复触发
- **缓解**：Step 2 先调研现有处理；eventFilter 在 MainWindow 层处理完后 return true 吃掉事件
- **回滚**：改为在 PreviewWidget 层处理，MainWindow 不拦截

### 风险 5：演示模式下主题切换菜单不可用导致无法切深色主题

- **影响**：用户在昏暗演讲场景想临时切深色主题，但菜单栏隐藏
- **缓解**：白名单里保留主题切换 action 的快捷键（如果有）；或接受此限制，要求用户提前切好主题
- **决策**：Plan 决定暂不支持模式内切主题

### 回滚方案

- Spec 层：删除 `11-演示模式.md`，README 清单恢复
- 代码层：`git revert` 实施 commit 即可；原 Focus Mode 代码从 git 历史找回
- 用户层：提示"F11 改回全屏编辑"并在下次补丁中恢复
