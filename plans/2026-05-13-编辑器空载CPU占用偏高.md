# 编辑器空载 CPU 占用偏高

## 背景

用户报告（2026-05-13）：

> CPU 占用好像高了一些，之前正常是 0.5% 左右，现在变成 3% 左右了

时间点：A1 视口剪裁 + lineHeight 估算修复 + EditorPainter cursor Y 推进 三处改动落地后。

## 排查方向

可能原因（按可能性排序）：
1. **PreviewWidget::scrollContentsBy 的节流 rebuild 逻辑**——即使滚动距离小，也每帧调
   verticalScrollBar()->value() 和 viewport()->height()，可能本身有开销
2. **PreviewWidget::paintEvent 中的 updateMetrics**（行 220）每次 paint 都同步度量，
   wrap 模式下可能更频繁
3. **EditorLayout::lineHeight 改为返回估算**——其他调用方（搜索高亮、光标位置）原本
   依赖 ensureLayout 触发的精算路径，现在拿到估算值后重新调 ensureLayout，引起额外
   开销
4. **EditorPainter::paint 的 cursor Y 推进**——每行调 lineHeight 也是 dirty 时返回
   估算（O(1) 无问题），但 layoutForLine 触发 ensureLayout 是预期成本

## 动作

- [ ] 用 PerfProbe / SM_PERF_SCOPE 在 PreviewWidget::paintEvent / scrollContentsBy
  和 EditorWidget::paintEvent 加耗时埋点
- [ ] `--debug` 模式启动，闲置 30 秒，分析 stderr 输出
- [ ] 找到高频路径，加节流或缓存
- [ ] 验证：闲置 CPU 回到 < 1%

## 验收

- 应用打开后无操作状态下 CPU 占用 < 1%
- 拖动滚动条 / 拖 splitter / 输入等用户操作期间 CPU 短暂飙升可接受
- 切 Tab / 切主题瞬间 CPU 飙升可接受

## 优先级

P2（不影响功能，但属于 INV-1 空载内存/CPU 预算的精神延伸）
