# TOC 面板垂直对齐编辑/预览区（不占 Tab 栏横向空间）

## 背景

用户原话：

> 当前右侧目录（TOC）面板从 Tab 栏顶部开始渲染，占据了 Tab 栏的水平空间。
> 希望 TOC 面板**只占据编辑区 + 预览区的高度**，顶部与编辑区/预览区对齐，
> 不与 Tab 栏并排。这样 Tab 栏可以横跨整个窗口宽度，TOC 作为独立侧栏只影响下方内容区。

### 当前布局架构（调研结果）

`src/app/MainWindow.cpp:95-110` 中的 central widget 结构：

```cpp
m_tabWidget   = new QTabWidget(this);      // 内部存放 editor|preview 的 splitter
m_tocPanel    = new TocPanel(this);
m_mainSplitter = new QSplitter(Qt::Horizontal, this);
m_mainSplitter->addWidget(m_tabWidget);   // 左侧：整个 QTabWidget（含 tabBar + pages）
m_mainSplitter->addWidget(m_tocPanel);    // 右侧：TOC 面板（从窗口顶部开始）
setCentralWidget(m_mainSplitter);
```

每个 Tab 内部（`MainWindow.cpp:583-602` `createTab()`）是：

```cpp
tab.splitter = new QSplitter(Qt::Horizontal);   // editor | preview
tab.splitter->addWidget(tab.editor);
tab.splitter->addWidget(tab.preview);
// 该 splitter 被 addTab() 塞进 m_tabWidget 作为 page
```

因此**实际控件层级**：

```
QMainWindow::centralWidget = m_mainSplitter (H)
├─ m_tabWidget                 ← 包含 tabBar(顶) + 当前 page(editor|preview splitter)
└─ m_tocPanel                  ← 从 QTabWidget 顶部齐高开始渲染
```

由于 `m_tocPanel` 和 `m_tabWidget` 是 `m_mainSplitter` 的兄弟节点，TOC 的顶端与 `m_tabWidget` 顶端对齐——也就是 tabBar 的顶端。用户看到的现象是 TOC 和 tabBar 从同一行开始。

## 当前布局

```
+--menuBar--------------------------------------------+
+-tabBar-------------------------+--TOC 也在这行起头--+
|                                |                    |
|  editor          |  preview    |       TOC          |
|                                |                    |
+--------------------------------+--------------------+
```

## 目标布局

```
+--menuBar--------------------------------------------+
+--tabBar --------------------------------------------+  ← 横跨整个宽度
|                                |                    |
|  editor          |  preview    |       TOC          |
|                                |                    |
+--------------------------------+--------------------+
```

Tab 栏横跨整个宽度，TOC 顶端只与编辑区/预览区顶端对齐。

## 根因分析

`MainWindow.cpp:104-110` 把 `m_tabWidget`（含 tabBar）与 `m_tocPanel` 放成 `m_mainSplitter` 的**兄弟**，二者顶部齐头渲染——这是症状的直接原因。

要么把 TOC 下沉一层（进入每个 Tab 的 splitter 或进入 tabWidget 的 page 容器），要么把 tabBar 上浮一层（与 horizontal content splitter 分成 VBox 的上下两层）。

## 方案对比

### 方案 A：TOC 下沉到每个 Tab 的 splitter 内

改 `createTab()`（`MainWindow.cpp:583`），在 `editor | preview | toc` 三栏结构里内嵌独立的 TocPanel 实例。

- 优点：布局层级最"干净"，TOC 顶端自然与 editor/preview 顶端齐平
- 缺点：
  - 每个 Tab 要持有自己的 TocPanel 实例，**打破了当前"全局单例 TocPanel"的设计**（`MainWindow.cpp:101`、`611-620` 的信号桥接大量重写）
  - `m_tocPanel->setTheme()`（`MainWindow.cpp:832`）、演示模式保存可见性（`1936`、`1984`）、设置持久化（`1410`）全部要重构为"遍历所有 tab"
  - Splitter 状态持久化 key 也要改（现在是全局单个 `window/mainSplitter`）
- 改动面：大（≥ 15 处调用点）

### 方案 B：用 QVBoxLayout 分成上下两层（tabBar 上，内容 splitter 下）

把 central widget 改成 `QWidget + QVBoxLayout`：

```cpp
centralWidget (QWidget, VBox)
├─ 上：m_tabWidget->tabBar()（或一个独立 QTabBar，配合 QStackedWidget 切换 pages）
└─ 下：m_mainSplitter（H）
    ├─ m_contentStack   ← QStackedWidget, 每页是 editor|preview splitter
    └─ m_tocPanel
```

实现有两种变体：

- **B1 拆 QTabWidget**：不再用 `QTabWidget`，改成 `QTabBar + QStackedWidget`，tabBar 塞 VBox 顶层、QStackedWidget 塞进 mainSplitter 左侧
- **B2 保留 QTabWidget，隐藏其内部 tabBar 移到 VBox 顶层**：比较 hack，不推荐

- 优点：
  - 全局单例 TocPanel 不变，`setTheme`/信号桥接全部保留（改动点聚焦到 `MainWindow` 构造函数和少量 splitter 状态 key）
  - Tab 栏横跨整个宽度这一视觉目标完全满足
- 缺点：
  - `QTabWidget` 换成 `QTabBar + QStackedWidget` 要重接 currentChanged / tabCloseRequested / tabMoved / customContextMenuRequested 等信号（但都是 1:1 搬家）
  - `addTab(splitter, title)` 改成 `tabBar->addTab(title)` + `stack->addWidget(splitter)`，调用点（`MainWindow.cpp:406`、`563`、`1121`、`1552` 等）需要对应适配
- 改动面：中等（约 8-10 处调用点，但模式一致）

### 方案 C：保留 QTabWidget，外包一层容器

central widget = `QWidget + VBox`：

```
VBox
├─ 顶：一个"伪 tabBar 容器"（占满宽）——里面塞 m_tabWidget 但通过 CSS / tabPosition
│     …（实质上无法让 QTabWidget 的 tabBar 脱离其 page 区）
```

`QTabWidget` 的 `tabBar()` 几何上被 QTabWidget 自己管理，无法独立搬家而不影响 pages 区域位置。方案 C 本质不可行，**否决**。

## 推荐方案：B1（拆 QTabWidget 为 QTabBar + QStackedWidget）

### 理由

- 用户视觉目标 100% 达成：tabBar 横跨整窗，TOC 与内容区顶端齐平
- 保留**全局单例 TocPanel**的架构——这是当前代码的核心前提（`MainWindow.cpp:611-620` 按 activeTab 过滤向单个 TocPanel 推送 entries），方案 A 要推翻这个前提，方案 B1 无需
- `QTabBar + QStackedWidget` 是 Qt 官方推荐的"自己掌控布局的 QTabWidget"替代，API 与 QTabWidget 基本一一对应，信号搬家成本低
- 方案 A 的架构迁移 bug 风险（漏接某个信号、某个 tab 的 TOC 没跟主题、演示模式出 bug）显著高于 B1 的"调用点批量改名"风险

### 权衡

方案 B1 唯一要小心的是当前 `QTabWidget::setDocumentMode(true)` 带来的样式（`MainWindow.cpp:98-99`）以及样式表对 `QTabWidget::pane` 的规则（主题系统可能有选择器）——迁移到 QTabBar 后需要检查主题 qss 是否还命中。

## 验收（T 条目）

- **T1 Tab 栏横跨整窗**：窗口最大化后，tabBar 右端贴近窗口右边（不被 TOC 截断）
- **T2 TOC 顶端对齐内容**：截图对比 TOC 顶端 y 坐标与 editor/preview 顶端 y 坐标差 ≤ 2px（布局取整容差）
- **T3 Tab 栏拖拽 resize 无感**：拖拽 `m_mainSplitter` 分隔条（editor|preview 分隔不变、contentStack|TOC 分隔会变），tabBar 整体位置/宽度不变
- **T4 切换 Tab 不影响 TOC 布局**：从含大量章节的 tab A 切到 tab B，TOC 内容换了但面板位置/顶端 y 不动
- **T5 关闭 Tab 不影响 TOC**：关掉当前 tab（`onCloseTab`），m_contentStack 跟着移除页，TOC 保持可见
- **T6 窗口 resize**：窗口从 1920 缩到 960，tabBar 随窗口宽度收缩并始终横跨，TOC 仍贴右
- **T7 拖拽重排 Tab**：拖 tab 顺序（`QTabBar::tabMoved`），m_contentStack 内 widget 顺序同步（不能出现"tab 换了但 page 没换"）
- **T8 演示模式**：进入演示模式 tabBar 隐藏（`MainWindow.cpp:1944`）；退出恢复显示
- **T9 主题切换**：深色/浅色主题下 tabBar 外观与旧 QTabWidget 模式视觉一致（可对 before/after 截图）
- **T10 Splitter 状态持久化**：关闭重开，TOC 宽度仍按上次拖拽状态恢复

## 风险

- **TOC 信号桥接**：现有 `MainWindow.cpp:611-620` 按 `currentTab()` 比较 preview 指针过滤信号，新架构下 `currentTab()` 返回值依赖 `m_tabBar->currentIndex() → m_tabs[idx]`——`currentTab()` 的实现必须同步改（从 `m_tabWidget->currentIndex()` 改为 `m_tabBar->currentIndex()`），否则 TOC 显示错位或空白
- **Splitter 状态 QSettings 可能失效**：`window/mainSplitter`（`MainWindow.cpp:1410`）存的是两栏 splitter 的 state，新架构 splitter 仍是两栏（contentStack | tocPanel），理论上可恢复。但列宽含义变了（原左栏是 QTabWidget 实际宽 = 窗口宽 - TOC 宽 - splitHandle，新左栏是 QStackedWidget 只覆盖 content 行）——QSplitter state 只记 sizes 不记 children 身份，应可沿用，但建议首次加载后做一次 sanity clamp
- **拖拽重排 tabMoved 的 stack 同步**：新架构下 `tabMoved(from, to)` 还要 `m_contentStack->insertWidget(to, m_contentStack->widget(from))`（QStackedWidget 的 removeWidget + insertWidget 配合）。一旦漏做，切 tab 后显示错页
- **`m_tocPanel->setTheme`、演示模式 show/hide**（`MainWindow.cpp:832`、`1944`、`1981`、`1984`、`1935`、`1994`）引用点不变（仍是 `m_tocPanel` 单例），迁移风险低
- **主题 qss 选择器**：若 `QTabWidget::pane`/`QTabWidget::tab-bar` 被样式引用，迁移后要改成 `QTabBar::tab`；未检查的主题可能出现丑陋边框
- **与 `plans/2026-04-14-TOC宽度自适应.md` 冲突**：两个 plan 都要动 `m_mainSplitter->setSizes(...)`，建议按顺序合入（先本 plan 改层级，再做宽度自适应），避免互相 merge 冲突。二者都计划新建 `specs/模块-preview/08-TOC面板.md`，第一个合入者建 Spec，第二个追加 INV 条目
- **与 `plans/2026-04-14-菜单栏与Tab栏分割线.md` 协同**：若那个 plan 改了 tabBar 与上下相邻区域的分隔线样式，本 plan 改变了 tabBar 的兄弟结构（下面紧邻的是 mainSplitter 而非 TOC），需要联合回归截图
- **高 DPI**：VBox 间距和 tabBar 高度不得硬编码，保持 Qt 默认；若需要微调 padding 必须用 `fm.height()*ratio`（INV-DPI-NO-HARDCODE）
