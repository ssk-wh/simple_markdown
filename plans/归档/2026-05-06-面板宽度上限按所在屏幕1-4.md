# 资源管理器和右侧目录区域：拖拽宽度不超过所在屏幕 1/4

## 背景
用户反馈：左侧资源管理器、右侧目录这两个面板，**拖拽分隔条调整宽度时**可以超过当前窗口所在屏幕宽度的 1/4——视觉上压得编辑/预览区过窄。期望：拖拽时强制上限 = 所在屏幕宽度 / 4。

仅记录待办，本次不修复。

## 关键概念
- **"所在屏幕"**：必须用窗口当前所在屏幕（多屏环境下即用户主操作屏），不是 `QApplication::primaryScreen()` 或主显示器
- 推荐：`QGuiApplication::screenAt(window->geometry().center())` 或 `window->screen()`（Qt 5.14+）
- DPI 缩放：`screen->geometry().width()` 已经是逻辑像素，无需额外缩放

## 动作（修复时）
- [ ] 在分隔条拖拽事件中（左侧面板 / TOC 面板各一处）截获新宽度，clamp 到 `screen.width() / 4`
- [ ] 拖拽限制点应在 SnapSplitter 或对应 splitter 的 `splitterMoved` 信号 / `setSizes` 路径上拦截
- [ ] **窗口跨屏移动**：窗口被拖到另一块屏幕后再调整宽度，应该按**新屏幕**的宽度计算上限（不要缓存旧值）
- [ ] **窗口未最大化**：上限仍以**屏幕**而非**窗口**为准（用户的诉求是"屏幕的 1/4"，不是"窗口的 1/4"），但要保证 clamp 后面板宽度仍 ≤ 窗口当前宽度（避免面板比窗口还宽）
- [ ] 现有的 `view/leftPanelWidth` / 目录宽度持久化值若超出新上限，加载时应被同样 clamp（避免恢复时违反约束）
- [ ] 在相关 Spec 补不变量：**[INV-PANEL-WIDTH-CAP]** 资源管理器和右侧目录面板的宽度不得超过窗口所在屏幕宽度的 1/4
- [ ] 测试：构造 1920×1080 屏幕环境下断言两个面板的可达最大宽度 = 480px
- [ ] CHANGELOG（用户视角措辞，例：左侧资源管理器和右侧目录面板拖拽时的最大宽度限制为屏幕宽度的四分之一，避免主编辑区被压缩过窄）

## 验收
- 在 1920×1080 屏幕下，把左侧 / 右侧面板分隔条向中间拖到极限 → 面板宽度 ≤ 480px
- 跨屏拖拽窗口（1920 屏 → 2560 屏）后再次拖面板 → 上限按 2560/4 = 640px 生效
- 持久化的旧宽度若超过当前屏幕 1/4，启动时被自动夹紧到 1/4，不破坏会话恢复
- 高 DPI（125%、150%、175%）下面板上限仍按逻辑像素 / 4 计算，视觉一致

## 注意
- 已存在 plan「面板宽度持久化」（2026-04-27 已完成，见 1.0.6 CHANGELOG）—— 本次只加上限约束，**不影响**现有持久化逻辑，但需确保恢复时同样 clamp
- 用户可能在小屏（1366×768）上感觉 1/4 太窄；如果反馈较多，可考虑加最小阈值（如 `max(screen/4, 200px)` 还是 `min(screen/4, 600px)`）—— 本次先按用户诉求按硬上限实现，后续再调
