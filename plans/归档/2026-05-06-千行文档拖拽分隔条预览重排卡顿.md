# 千行文档拖拽目录宽度时预览区重排卡顿

## 背景
用户反馈：约 **1000 行**的 Markdown 文档，拖动右侧目录分隔条调整宽度时，预览区重新渲染过程明显卡顿。期望：拖拽过程顺滑（≥ 30 fps，理想 60 fps）。

仅记录待办，本次不修复。

## 初步分析（待修复时核实）

拖拽分隔条 → splitter `splitterMoved` 信号 → 预览区 `resizeEvent` → `PreviewWidget::rebuildLayout()` → `PreviewLayout::buildFromAst()`：

- `buildFromAst` 是**全量**重排版（O(节点数 × 内联段数)），1000 行的 Markdown 节点数可达数百到上千
- 拖拽时 `splitterMoved` 信号每移动 1 像素就触发一次（高频）
- 每帧重新跑全量布局 → 单帧时间超过 16ms → 视觉卡顿

参考点：splitter 拖拽时鼠标移动每秒可触发 60-120 次，单帧布局预算应当 < 8ms 才不卡。

## 候选优化方向（仅记录，需修复时实测对比）

### A. 拖拽期间节流（最低成本）
- splitter 拖拽中只标记"宽度变化待应用"，不立即 buildFromAst
- 用 QTimer 节流到 33ms（30fps）触发一次重排版；或者 splitter 释放（`splitterMoved` 静默 100ms）后才真正重排
- 拖拽中预览仍画旧布局（视口范围内），用户感知可能是"轻微滞后但不卡"

### B. 拖拽期间画占位（最低质量）
- splitter 拖拽中预览区显示模糊占位（半透明遮罩 / 文本"调整中..."）
- 释放后才真正 buildFromAst
- 视觉妥协但绝不卡

### C. 增量重排版（最高难度，最高收益）
- buildFromAst 改为支持"宽度变化重排"分支：仅重新计算受影响 block（其内容会因换行变化），其他 block 复用旧的 LayoutBlock
- 需要 LayoutBlock 携带"上次构建时的可用宽度"，对比后决定是否 invalidate
- 难点：标题字号 scale、列表 indent、表格列宽分配等都隐式依赖 viewportWidth；增量边界判定要小心

### D. 视口裁剪重排（中难度，中收益）
- 拖拽期间只重排"当前视口可见的 ±N 屏"区块；其余 block 保留旧布局，等用户滚动到时再重排
- 需要 LayoutBlock 支持 lazy 状态

### E. 复合方案（推荐起点）
- 先做 A（节流）+ 现有性能监控（PerfProbe SM_PERF_SCOPE）量化 buildFromAst 单次时间
- 若 A 已经够用 → 收工
- 若 A 不够 → 加 D 或 C

## 验收
- 1000 行文档在拖拽分隔条全程帧时间 ≤ 33ms（30fps）；理想 ≤ 16ms
- 拖拽释放后预览区与编辑器保持滚动同步、TOC 高亮正确
- 不引入新的视觉抖动（节流方案不应让用户在拖拽中看到突兀的"跳变"）

## 实施结果（2026-05-06）

**意外途径解决**：本 plan 的目标在 plan 「2026-05-06-表格单元格内容超出与下方重合」 的修复中**顺带达成**。

修复表格 cell 越界 bug 时把 `estimateParagraphHeight` 的换行算法从「整段总宽 ÷ 行宽 上取整」改为
「逐字符级模拟 paint 端换行」（见 `specs/模块-preview/02-布局引擎.md` INV-15）。
意外发现：旧算法对长 run 调用 `fm.horizontalAdvance(整段长字符串)` 触发 Qt 完整 shaping pipeline，
而新算法逐字符 advance 走快路径，整体 buildFromAst 反而更快。

| 指标 | 修复前 | 修复后 | 改善 |
|------|--------|--------|------|
| 拖拽 P50 | 41.4 ms | 16.7 ms | -60% |
| 拖拽 P95 | 77.8 ms | 19.1 ms | -75% |
| 稳态 P50 | 43.2 ms | 15.3 ms | -65% |
| 稳态 P95 | 67.2 ms | 18.3 ms | -73% |

测试机：Win11，1024 行 / 37912 字节样例 markdown。

**验收对照**：
- ≤ 33ms (30fps 必须线) ✓ 全部场景达标
- ≤ 16ms (60fps 理想线) ✓ P50 接近达标（16.7 ≈ 16）
- 没动 splitterMoved / 滚动同步代码，无新抖动

**剩余空间**（如用户复测仍感卡顿可走）：
- splitterMoved 节流到 33ms（方案 A）—— 把 P95 19ms 的 5% 慢帧降到 16ms 以下
- 增量重排（方案 C）—— 仅当用户实际反馈"P95 仍卡"时再做，避免过度工程

## 注意 / 风险
- 增量重排方案（C）容易引入"宽度未触发但已 invalid"的隐藏 bug；如果走这条路，先把 invalidate 触发点列清楚（字体变化、wordWrap 切换、主题切换、行间距、宽度变化）
- 不要在没有数据的情况下提前优化；Step 0 基线测量是非可选的，否则可能"优化了一个不在关键路径的部分"
- 与 plan「编辑区与预览区字号视觉不一致」 / 「Tab 休眠」 不冲突，但都涉及 PreviewLayout 内部状态，修复时序注意：先做这条性能 plan 再做休眠 plan，避免休眠唤醒路径上的延迟优化和拖拽优化互相干扰
