# 多 Tab 超阈值时休眠最早打开的文档（保留 Tab 头）

## 背景
长时间使用过程中，用户可能打开几十甚至上百个文档 Tab。每个 Tab 的 EditorWidget + PreviewWidget + 解析后 AST + 布局缓存都常驻内存，导致进程内存只增不减。

期望行为：**当打开 Tab 数超过阈值时，自动"休眠"（释放重资源）最早打开的文档，但顶部 Tab 头保留**；用户点击该 Tab 时**重新打开**文档（从磁盘加载、重新解析、重新布局）。

仅记录待办，本次不实现。

## 目标
- 上限可控：用户在长会话中的内存占用不会无限增长
- 体验无感：休眠对用户基本透明（除了点回去时有一次轻微"加载感"）
- 不丢数据：未保存的修改**绝不**因为休眠而丢失

## 待决策的设计点（draft 阶段必须记录）

### A. 触发条件
1. 阈值 N（默认值？）→ 候选：固定（如 10 / 20）/ 按内存压力动态计算 / 用户可配
2. 触发时机：
   - a. 打开第 N+1 个文档时同步休眠最早一个？
   - b. 后台定时扫描？
   - c. 内存达到某阈值时主动收缩？
3. 是否对**未保存修改**的 Tab 豁免？（强烈建议：是）
4. 是否对**当前活动 Tab + 最近一个**豁免？（建议：是，避免 Ctrl+Tab 来回切马上撞墙）

### B. 淘汰策略
- 用户原话："最早**打开**的" → 按打开时间 FIFO
- 替代方案：按**最近访问**时间 LRU（与现代浏览器一致）
- 决策需明确写入 Spec（候选 INV）

### C. "保留 Tab 头" 的状态
休眠后必须保留：
- 文件路径（必需，唤醒入口）
- Tab 标题（即文件名，标题栏显示）
- 是否已修改？休眠时**禁止**有 dirty 修改 → 见 A.3
- 上次的滚动位置 / 光标位置（唤醒时恢复）
- 锁定的 Tab？（如有此能力，应豁免）

可视化：
- 休眠 Tab 是否给一个微弱的视觉标记（如标题灰一点 / 小图标）？还是完全无感？
- 决策需写入 Spec

### D. 唤醒路径
- 用户点击 Tab → 重新读盘 → 解析 → 布局 → 显示
- 需要 loading 指示？（建议：阈值如 200ms 内不弹，超过才弹）
- 唤醒失败怎么办？（文件已被删除 / 无权限）→ 提示并保留 Tab 头还是关闭？
- 性能预算：唤醒一个普通 Markdown 文档应在 < 200ms 内完成

### E. 与会话恢复的相互作用
- 已经存在的"会话恢复"机制本身就是"启动时把 N 个 Tab 还原成头但延迟加载"——本 Feature 实质上是**把会话恢复的延迟加载策略推广到运行期**
- 应统一为同一套机制：`TabState ∈ { Active, Dormant, NeverLoaded }`
- 关闭应用时，休眠 Tab 持久化为 NeverLoaded 即可

### F. 实际能省多少内存？
- 真实大头不是 QString 内容，是：解析后的 AST、PreviewLayout 缓存、QTextDocument（如使用）、字体度量缓存等
- 需要一次基线测量：单 Tab 加载 100KB / 1MB / 10MB Markdown 后进程 RSS 增量
- 如果释放收益 < 1MB/Tab，那么这个 Feature 价值有限

## 风险与回滚
- **风险 1：误丢未保存修改** —— 若实现疏忽把 dirty Tab 也休眠，用户数据会丢
  - 缓解：休眠前置硬性断言 `!isModified()`，并在测试中专门覆盖
- **风险 2：唤醒失败 / 文件已变** —— 期间文件被外部修改/删除/重命名
  - 缓解：唤醒时按"打开文件"完整流程处理失败分支；保留 Tab 头并显示明确错误
- **风险 3：滚动同步 / 跨 Tab 导航被打断** —— 例如 TOC 跳转到一个休眠 Tab 时如何处理
  - 缓解：跳转前同步唤醒；明确所有"程序触发激活"路径
- **风险 4：性能反向劣化** —— 频繁休眠/唤醒在边界条件下抖动
  - 缓解：阈值要有滞回（hysteresis）；当前 Tab + 最近一个豁免；阈值不要太低
- **回滚方案**：把阈值调为 ∞（即关闭休眠），降级到"全部常驻"行为；功能开关在 Spec 中预留
