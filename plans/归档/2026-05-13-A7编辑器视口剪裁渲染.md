# A7 编辑器视口剪裁渲染（与 A1 对称）

## 背景

A1 plan（2026-05-12）完成预览侧 PreviewLayout 视口剪裁后，benchmark 数据 100x 加速。
但用户实测拖动左/右侧 splitter 时仍体感卡顿——定位到根因：

**EditorLayout::setWrapWidth 内部走 m_lines.clear() 抛弃全部 QTextLayout 缓存，每帧都做开销不小**
（源自 `src/editor/EditorWidget.cpp:309-313` 注释）

中央 SnapSplitter 已有 `smSnapDragging` 拖拽态保护路径（拖动期间跳过 setWrapWidth，
松手时由 dragFinished 触发 recomputeWrapForCurrentWidth）；但左/右侧普通 QSplitter
没有这层保护，每次 resize 实时触发完整 wrap 重建。

10k 行编辑器 wrap 重排估计 100ms+ 量级。用户拖左/右 splitter 时：
- 预览侧：4ms（A1 已优化）✅
- 编辑器侧：100ms+（无视口剪裁）❌

体感卡 ≈ 编辑器卡。

## 目标

- 10k 行编辑器文档 setWrapWidth 单次 ≤ 16ms（P95）
- 拖左/右侧 splitter 不再有可感知延迟
- 与 A1 PreviewLayout 视口剪裁路径对称（INV-16/17/18/19 类比）
- 不破坏现有编辑器 INV：撤销/重做粒度、IME 中文输入、语法高亮性能

## 影响范围

- 修改：`src/editor/EditorLayout.cpp/h`（核心）、`src/editor/EditorWidget.cpp`（视口监听 + 触发）
- 新增：EditorLayout 加 placeholder 概念 / setViewportYRange API
- Spec 变更：`specs/模块-editor/` 新增 INV「编辑器视口剪裁」、
  `specs/横切关注点/70-性能预算.md` 加 INV-10「编辑器视口剪裁后 setWrapWidth 预算」

## 风险与回滚

- 风险 1：QTextLayout 是 Qt 的"行级"缓存，比预览的"块级"细——视口外占位精度要求更高
  - 缓解：用平均行高估算 placeholder height，进入视口前精算
- 风险 2：撤销/重做 / 搜索 / 当前光标行需要跨视口仍能访问行数据
  - 缓解：lazy 升级——非视口区域的"操作"先触发该区域的 layout 升级
- 风险 3：编辑器和预览的差异——编辑器有「光标可视性」要求，placeholder 区域必须保证
  Ctrl+End 跳到末尾时光标位置可见
  - 缓解：跳转操作先调 applyViewportCrop(新光标行附近) + rebuild
- 回滚：feature flag `editor/virtualized_layout`，问题时关回全量

## 进展

- 2026-05-13 启动，对应用户反馈"拖动目录左边分隔条仍卡"
- 2026-05-13 阅读 EditorLayout/EditorPainter 源码定位真因——不是 setWrapWidth 慢
  （rebuild 只是 clear + resize），而是：
  - rebuild → invalidateYCache → 下次 updateScrollBars 调 totalHeight → ensureYCache
    遍历所有行调 lineHeight(i)
  - wrap 模式 / 非默认行距下 lineHeight 触发 ensureLayout → 每行构建 QTextLayout +
    SyntaxHighlighter 高亮 token 化
  - 10k 行 × 几十微秒/行 = 几百 ms 卡顿
- 2026-05-13 实施两点最小修复（绕开"视口剪裁"等大改造）：
  1. **EditorLayout::lineHeight** 在 layout 未构建时返回 `m_defaultLineHeight × m_lineSpacingFactor`
     估算值，**不**触发 ensureLayout。代价：未 layout 行高度不准（wrap 行被低估）
  2. **EditorPainter::paint** 改为本地 cursor Y 推进：起始用 `lineY(firstLine)`，
     之后每行 `layoutForLine` 触发 ensureLayout 后用精算 `lineHeight(line)` 推进。
     视口内行 paint 时 ensureLayout 完成后，调 lineHeight 返回精算 height——不再
     依赖 ensureYCache 的估算 yCache 排版
- 2026-05-13 用户实测：「不卡了，但是编辑区出现了内容重叠的问题」——发现 paint
  路径仍用 lineY 给每行单独算 Y，wrap 多行实际高度被估算低估 → 下一行 Y 与上一行
  wrap 内容重叠。修复见上面第 2 点
- 2026-05-13 用户二次实测：「自动换行后没有重叠了，滚动明显并流畅了」——✅ 通过
- 13/13 单元测试 0 回归

## 实施约束（写入 spec INV-10/横切 70）

- 编辑器 ensureYCache / lineHeight 路径**禁止**对未 layout 行批量触发 ensureLayout，
  否则 N 行同步构建 QTextLayout + 语法高亮会触发可感知卡顿
- 视口外行的高度允许估算（`defaultLineHeight × spacingFactor`），导致滚动条 maximum
  在 wrap 模式下略偏小——属于可接受 trade-off
- 视口内行的 paint 位置**必须**基于 ensureLayout 后的精算高度推进（本地 cursor Y），
  不得依赖 ensureYCache 的估算 yCache，否则 wrap 行视觉重叠
