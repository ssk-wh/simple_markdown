# A2 编辑增量解析（基于 sourceLine 局部 reparse）

## 背景

当前编辑器每次输入触发 30 ms 防抖后 **全量 parse** 整文档（INV-4「1k 行 < 10ms，10k 行 < 50ms」）。
对大文档持续编辑时：

- 改了第 5000 行 → 重 parse 第 1-4999 行的工作纯属浪费
- 即便在预算内，每次按键的 P99 长尾会显著高于 INV-2「打字延迟 < 3ms」
- 主线程被持续占用，影响 INV-3 预览滚动平滑度

通用做法：把 AST 按 block 切片，仅重 parse 受影响的 block 集合，配合 sourceLine 锚点定位。

## 目标

- 10k 行文档单点编辑触发的 parse 耗时 ≤ 5 ms（P95）
- 改一个段落不引发整篇 AST 失效（其他段落的 inline 高亮 / TOC / 标记保持原样）
- 不破坏 INV-4 全量 parse 兜底（首次加载、粘贴大段、撤销等场景仍允许全量）

## 影响范围

- 修改：`src/parser/`（核心）、`src/editor/EditorWidget.cpp`（触发增量解析的事件路径）
- 修改：`src/preview/PreviewWidget.cpp`（接收 patch 而非整 AST 的接口）
- Spec 变更：`specs/模块-parser/` 新增 INV「增量解析的边界判定（block 起止）」、
  `specs/横切关注点/70-性能预算.md` 新增 INV-10「增量 parse 预算」

## 步骤拆分

- [ ] Step 1: profile 当前 10k 行文档单字符输入触发 parse 的实测耗时，建立基线
- [ ] Step 2: 设计「最小影响块集合」算法（修改的 line 范围 → 受影响的 AST block）
- [ ] Step 3: AST 数据结构支持 patch 应用（替换 / 删除 / 插入 block）
- [ ] Step 4: 预览侧 LayoutBlock 复用未变化 block 的 layout 结果
- [ ] Step 5: 边界场景：跨 block 编辑（如表格 → 段落转换）、围栏代码块开闭、frontmatter
- [ ] Step 6: 兜底：检测到「block 边界改变」时降级到全量 parse

## 风险与回滚

- 风险 1：边界判定漏掉某个跨 block 影响 → AST 与源文不一致 → 渲染错乱
  - 缓解：定期（每 N 次增量后或 idle 1s）全量 reparse 自检，差异时纠正
- 风险 2：内容标记（荧光笔）的字符 offset 在 patch 后失效
  - 缓解：标记锚点改用 (sourceLine, columnOffset) 替代纯 charOffset
- 回滚：feature flag `editor/incremental_parse`，有问题关回全量路径

## 验证清单

- [ ] 新增 T 条目：10k 行文档单字符输入 parse < 5ms（P95）
- [ ] AST 自检：随机编辑 100 次后全量 reparse 结果与增量累计结果 byte-for-byte 一致
- [ ] 跨 block 编辑（表格→段落 / 围栏开闭）测试通过
- [ ] 内容标记持久化测试通过（编辑后标记位置不漂移）
- [ ] CHANGELOG.md 更新（用户可感知：大文档持续输入更流畅）

## 进展

- 2026-05-13 评估后 **aborted**。原因：
  1. **A1 视口剪裁 + A6 防抖自适应已覆盖主要痛点**——视口外块不精算（80-100x 加速），
     大文档（5k+ 行）防抖窗口拉到 50-80ms 让 parse 自然容纳
  2. **完整实施成本与 A1 相当**——需要 cmark 增量解析改造 + AST patch + LayoutBlock
     复用 + 边界处理（跨 block 编辑 / 围栏开闭 / frontmatter）
  3. **内容标记字符 offset 漂移**是高风险点（plan 风险段已列）——标记锚点改用
     (sourceLine, columnOffset) 替代纯 charOffset 需要 marking 持久化协议变更
  4. **INV-4「10k 行 < 50ms」当前满足**——是预防性优化，非紧迫 bug
  5. 用户实际**未报告** 10k 行持续输入卡顿——A2 是工程师驱动而非用户驱动
- 若未来用户在 10k+ 行场景明确报告"持续输入有延迟"，再启动该 plan
