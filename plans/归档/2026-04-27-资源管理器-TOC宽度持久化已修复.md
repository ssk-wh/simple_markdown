# Bug: 用户调整资源管理器/TOC 宽度后重启未恢复

## 复现
1. 打开文件夹，拖拽左侧面板/TOC 的 splitter 调整宽度
2. 关闭应用
3. 重新打开，宽度未恢复到上次调整的值

## 真实根因
`showEvent` 触发时 `m_mainSplitter->width()` 仍是 ctor 中 `resize(1280,800)` 设的初始值（最大化几何过渡未完成），导致：

1. 旧代码用 `totalW/8` 作为默认值，`totalW/5` 作为 maxW；用户保存的 350 px 会被错误的 maxW=256 截断到 256。验证证据：`[WIDTH-DBG]` 输出显示 `screenW=1920` 而 `totalW=1280`
2. 上一会话 commit `0a01fe8` 引入的"显式 view/leftPanelWidth restore"段同样基于 `totalW` 算 maxW，仍然踩同一个坑
3. TOC 有第二层覆盖：`clampTocWidthToScreen()` 在 resizeEvent 中把 `sizes[2] > screen/8` 砸回去，**未尊重 `m_userDraggedToc`**；而 `m_userDraggedToc` 是会话级未持久化，重启总是 false → 用户 saved 300 被自动砸回 240

## 修复（2026-04-27）
1. `MainWindow::showEvent` 默认值改用"应用所在屏幕宽度 / 8"（不依赖 splitter 瞬时 width），合并段 A/C 为统一逻辑：saved 优先，否则 screen/8
2. `MainWindow::saveSettings` 同时持久化 `view/tocPanelWidth`
3. showEvent 中读到 saved tocPanelWidth 时强制 `m_userDraggedToc = true`，避免 content-aware 自适应覆盖
4. `clampTocWidthToScreen()` 加 `m_userDraggedToc` 检查
5. `updateLeftPaneVisibility` 段 D 改为仅在 `sizes[0] <= 0` 时补宽度，不再 clamp maxW
6. Spec 新增 INV-LP-WIDTH-DEFAULT / INV-LP-WIDTH-PERSIST / INV-TOC-WIDTH-PERSIST

## 验收（已通过）
- T1（清空 saved）：启动 → `leftW=240, tocW=240`（screenW=1920, totalW=1280）✓
- T2（saved 350/300）：启动 → `leftW=350, tocW=300`；优雅关闭 → reg 中仍是 350/300（未被覆盖）✓
