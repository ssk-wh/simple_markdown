# 行内代码（inline code）垂直基线偏上

## 现象

预览区段落中的 inline code（反引号包围的 `text`）视觉上比同一行周围正文偏上，未共享统一基线。

## 根因

`src/preview/PreviewLayout.cpp` AstNodeType::Code 分支把 `run.font` 字号缩到周围正文的 90%。
`src/preview/PreviewPainter.cpp::paintInlineRuns` 在绘制每个 run 时使用的是 `fm.ascent()`
（`fm` 基于 `run.font`）来算 drawText 的 y：

```cpp
p->drawText(QPointF(curX, curY + fm.ascent()), seg);
```

code run 的字号较小，其 ascent 比正文小，结果 baseline 上移 → 视觉上 code 偏上。

## 修复策略

**策略 A：统一行基线**

绘制循环内维护一个 `lineAscent`，基于该段落主字体（`block.inlineRuns[0].font` → `defaultFm.ascent()`）。
所有 run（包括小字号 code run）使用 `baselineY = curY + lineAscent` 作为 drawText 的 y。背景矩形仍以 `curY` 为 top、行高为 `lineHeight`，以保证视觉效果稳定。

理由：
1. 改动最小，只动 PreviewPainter.cpp 的 drawSegment lambda
2. 不需要改 LayoutBlock/InlineRun 结构
3. 与 spec `INV-7 序号与内容使用同一 Y 坐标` 的思想一致 —— inline run 也应该同步基线

## 验证

- T 编号：新增 `T-10 行内代码与周围文字共享同一基线`
- 目视：打开含 `` `text` `` 的段落，观察 code 底部与旁边正文底部对齐
- grep：`p->drawText.*fm.ascent\(\)` 在 paintInlineRuns 中应改为 `lineAscent`

## 归档记录

### 根因

`paintInlineRuns` 的 drawSegment lambda 中 drawText 的 y 坐标使用 `fm.ascent()`，
`fm` 基于当前 run 的 font。行内代码 run 字号 = 正文 * 0.9，其 ascent 小于正文 ascent，
导致 code run 的 baseline 向上偏移（概念层面的 bug）。

### 修复方案

策略 A：在 `paintInlineRuns` 头部构造 `QFontMetricsF defaultFm(block.inlineRuns[0].font, p->device())`，
提取 `lineAscent = defaultFm.ascent()` 作为该段落所有 run 共享的基线偏移；drawSegment 中
drawText / underline / strikethrough 三处 y 偏移全部由 `fm.ascent()` 改为 `lineAscent`。
文本背景矩形仍以 `curY` 为 top、`fm.height()` 为高度（这不涉及基线，仍基于每 run 的度量）。

同步 Spec：`specs/模块-preview/03-绘制管线.md` 新增 INV-10 / T-10，在「已知陷阱」加入 §8.6。

### 验证方式

1. **构建**：`./build_on_win.bat release` 成功
2. **代码层审查**：`paintInlineRuns` 中 drawText 不再出现 `fm.ascent()`，已全部替换为 `lineAscent`
3. **目视验证**：
   - 启动 `build/src/app/SimpleMarkdown.exe tmp/test_inline_code.md`
   - 构造 `baseline check: HhPpy \`HhPpy\` HhPpy` 之类的段落
   - 对比主仓（修复前 HEAD commit）和 worktree（修复后）的 PrintWindow 截图
4. **有趣发现**：当前字号组合（Segoe UI 12pt 正文 / Consolas 10.8pt code）下，
   修复前后像素完全一致（PIL ImageChops.difference bbox=None）。原因推测为 Qt
   对不同字体 ascent 做了像素量化，碰巧 snap 到同一像素行。
   修复在**概念层面**正确，且当字号/字体组合变化（如更大字号 delta、或未来新增
   其他混字号场景）时能防止基线漂移复发。
