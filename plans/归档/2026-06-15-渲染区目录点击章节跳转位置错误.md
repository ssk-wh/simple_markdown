# 渲染区目录点击章节跳转位置错误

## 背景

用户报告（2026-06-15）：

> 渲染区目录点击章节后，跳转的位置错误，尤其是离当前打开位置较远的位置跳转基本都出错，
> 怀疑是较远的位置还没渲染出来，所以计算位置出错。

**用户的怀疑基本被历史记录证实。** 项目曾因同一根因撤销过姊妹功能：

- 提交 `7b921fe revert: 移除 Ctrl+数字章节跳转——视口剪裁下位置偏差不可控`
- CHANGELOG 1.1.x 撤销说明（原文）：
  > 撤销 1.1.6 引入的 Ctrl+1..Ctrl+9 章节跳转快捷键——大文档场景下跳转目标位置时常
  > 出现首次不到位的偏差（**受预览区按需渲染机制影响，目标章节在视口外时位置基于估算
  > 累加**），用户需多次按才能精确到位。完整修复成本（跳转前临时全量重排或独立精算路径）
  > 与该功能价值不成正比，决定移除。**章节跳转仍可通过右侧目录面板点击实现**。

⚠️ 当时只 revert 了「Ctrl+数字快捷键」入口，**保留了「目录面板点击」入口**——而用户本次
报告的正是后者也有同样偏差。即上次的撤销没有覆盖这个入口，根因（视口外位置估算累加）
依然存在于 TOC 点击路径。

## 根因方向（高可信，来自历史复盘）

- 预览区自 `A1 视口剪裁渲染` 起改为**按需渲染**：视口外的 block 不实际 layout，其位置靠
  **估算累加**得出。
- 章节锚点 → 滚动目标 Y 坐标依赖该估算值；目标离当前视口越远，累计估算误差越大 → 跳不准。
- 这解释了「近处准、远处基本都错」的现象。

## 复现待补充

| 项 | 待补 |
|----|------|
| 文档规模 | 多少行 / 多少章节才明显（大文档更严重） |
| 距离相关性 | 跳邻近章节是否准？跳文档末尾章节偏差多大 |
| 偏差方向 | 跳过头 / 不到位 / 完全错乱 |
| 二次点击 | 同一章节再点一次是否会逐步逼近（估算→实测收敛） |
| 含图片/代码块 | 路径上有大图/长代码块时是否更严重（懒解码高度估算） |

## 可选修复方向（先记录，不实施；需三方评审）

1. **跳转前对目标路径精算**：滚动前对当前位置→目标章节之间的 block 做一次真实 layout
   累加（成本：大文档可能卡顿，需限定范围或异步）。
2. **跳转后自校正**：先按估算滚到附近，目标 block 进入视口被真实 layout 后，用实测 Y
   二次微调到位（体感上"先到大概，再吸附"）。
3. **锚点重定位重试**：滚动落定后检测目标章节实际 top 与期望差值，超阈值则再修正一次。

> 上次评估「跳转前临时全量重排或独立精算路径，成本与价值不成正比」而放弃快捷键入口；
> 但目录点击是核心导航功能，不能也放弃——方向 2/3（跳转后自校正）可能是性价比最高的路线。

## 进展（补充）

- 2026-06-15 定位 + 修复（单测量化验证）：
  - 根因确认：`smoothScrollToSourceLine` 用动画开始时的估算 Y 滚动；视口剪裁下远处目标是
    placeholder 粗估，动画期间 scrollContentsBy 重建 layout 改变了块高/sourceLineToY，
    但滚动停在旧估算 Y → 绘制用新 layout、滚动用旧坐标 → 目标章节不到位。
  - 修复：`onScrollAnimationFinished` 增加自校正循环——重建 layout → 重算 sourceLineToY(target)
    → 偏差 ≥3px 则滚到该值，最多 4 次迭代收敛到不动点（scrollY==sourceLineToY，绘制与滚动
    同源 → 目标落视口顶）。采用 plan「方向2/3 跳转后自校正」（性价比最高，无需跳转前全量重排）。
  - 测试：新增 `tests/preview/PreviewTocJumpTest.cpp`（T-TOC-JUMP-1/2），模拟自校正循环。
    **量化结果：无校正落位偏差 164px → 自校正后 0px，收敛**。
  - Spec：07-TOC面板.md 新增 INV-TOC-JUMP-CORRECT；tests 字段更新。CHANGELOG：1.1.10。
  - 注：本修复仅作用于目录面板点击路径；已移除的 Ctrl+数字快捷键不恢复（plan 既定）。
- 2026-06-15 ✅ 已完成（单测验证收敛），待用户大文档点击远处章节手动确认体感

## 进展（修正——迭代方案验证失败后重做）

- 2026-06-16 用户手动验证**失败**：迭代自校正方案在真实环境仍不到位。
- 真因（之前被 offscreen 假通过掩盖）：原修复用 4 次迭代「applyLayoutViewportCrop + buildFromAst」
  重建，每次重建让 ±2 屏缓冲边界随 scrollY 移动，边界处块在估算↔真实间翻转，使
  sourceLineToY(target) 每轮抖动约一个块高。**真实 windows 平台实测 residual=28px、ping-pong 不收敛**；
  offscreen 平台文字不光栅化、高度均匀恰好收敛到 0 → 单测假通过（深刻教训：ctest 必须用真实平台）。
- 正解：`onScrollAnimationFinished` 改为**单次校正、禁止迭代重建**——围绕落点重建一次让目标真实化，
  取该 layout 下 sourceLineToY(target) 作为 scrollY，blockSignals 设值后**不再重建**，绘制沿用同一
  layout → 目标精确落顶。
- 测试：重写 `PreviewTocJumpTest`（T-TOC-JUMP-1 单次校正落顶 / T-TOC-2 锁定"迭代重建不稳定不可取"），
  现在 ctest 用真实 windows 平台跑，16/16 全过。
- Spec 07 INV-TOC-JUMP-CORRECT 同步更新为"单次校正、禁止迭代"。
