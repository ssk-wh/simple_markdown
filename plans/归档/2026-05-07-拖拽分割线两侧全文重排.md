# 拖拽编辑/预览分割线两侧全文重排导致卡顿

## 背景

用户反馈：拖动编辑区与预览区中间的 SnapSplitter 分割线时仍有卡顿感，**即使现有 `PreviewRenderBenchmark` 显示 1024 行文档拖拽 P50 = 11.4ms（达成 60fps 预算）**。

根因调查（2026-05-07）：

1. `SnapSplitter` 用 QSplitter 默认 `OpaqueResize=true`，拖拽过程中每个 mouseMove 都同步触发两侧子 widget 的 `resizeEvent`。
2. **预览侧**：`PreviewWidget::resizeEvent`（`PreviewWidget.cpp:180-192`）→ `rebuildLayout()` → `PreviewLayout::buildFromAst(m_currentAst)`——对整个 AST 树整轨重排，无屏内/屏外裁剪、无节流、无 debounce。
3. **编辑器侧**：`EditorWidget::resizeEvent`（`EditorWidget.cpp:299-315`）→ `EditorLayout::setWrapWidth` → `rebuild()`（`EditorLayout.cpp:73-86`）→ `m_lines.clear()`，把全部行 `QTextLayout` 抛弃；后续 paintEvent 时按可见行（≈30 行）懒构建。
4. **现有 benchmark 的盲区**：`PreviewRenderBenchmark.cpp` 用 `QImage` 离屏只测 `buildFromAst` 算法本身，**不计 paintEvent 实际绘制 / ScrollBar 几何更新 / 编辑器侧 rebuild + 可见行 QTextLayout / DPR 同步路径**。所以 11.4ms 是"算法地板"不是"端到端单帧"。

用户洞察："两个区域都在渲染"——两侧均做全文级 layout 失效，不只是重画。

## 目标

- [ ] 1024 行文档拖拽分割线**端到端单帧** P50 ≤ 16 ms / P95 ≤ 22 ms（包含 paintEvent）
- [ ] 拖拽**结束**时最终布局与"非节流路径"一致（不能因为节流让停手位置布局错位）
- [ ] 不影响小文件拖动观感（< 200 行文档原本就不卡，新机制不能引入新延迟）
- [ ] 现有 `PreviewRenderBenchmark` 基线数据不劣化（算法 hot path 不改）

## 设计

### 方向 A：节流 rebuild（推荐 / 待用户拍板）

保留实时跟手观感，把全文重排合并到 16ms 节奏。

**预览侧**：
- `PreviewWidget` 新增 `QTimer m_rebuildCoalesce`（singleShot, 0ms） + 节流 token
- `resizeEvent` 不直接 `rebuildLayout()`，而是 `m_rebuildCoalesce.start()`；多次 resize 合并为一次
- 进一步：拖拽态识别——`SnapSplitter::beginDrag/endDrag` 暴露信号，预览侧拖拽期改为 16ms 节流，停手时强制 flush

**编辑器侧**：
- `EditorLayout::setWrapWidth` 同样合并；停手时统一 flush
- 由于 paintEvent 是懒构建，编辑器侧主要节省 `m_lines.clear()` 的开销

**结束时强 flush**：拖拽 release → `endDrag()` signal → 两侧 widget 立即跑一次同步 rebuild，确保最终态准确

### 方向 B：lazy resize（备选 / 改一行见效）

`SnapSplitter::SnapSplitter` 构造里加 `setOpaqueResize(false)`。拖动时只显示 rubber band（VS Code / Visual Studio Windows 风格），松手才 resize。
- 改动量：最小
- 视觉代价：拖动时内容不实时跟随
- 兼容点：与 `SnapOverlay` 三条吸附线共存——SnapOverlay 已经是独立 child widget，OpaqueResize 切换不影响它

### 方向 C / D（暂不采纳）

- C：增量 layout + `PreviewBlockCache` 复用——风险高，缓存失效复杂，留作后期
- D：屏内优先 layout 分阶段——中等复杂度，可作为方向 A 之后的进一步优化

## 任务

最终选定**方向 E（拖拽态识别 + dragFinished 信号）**——介于方向 A（节流）与 B（lazy resize）之间的折衷：
- 拖拽期间子 pane 跳过全文 rebuild（接近 B 的"内容延迟更新"效果）
- 但 handle 跟手 + SnapOverlay 吸附线实时响应（保留 A 不破坏现有交互的优势）
- 解决 B 的吸附 UX 退化问题（QSplitter non-opaque 模式下 splitterMoved 仅 release 时 emit 一次）
