# 编辑区与预览区正文字号视觉不一致（继续追踪）

## 背景

用户反馈：编辑区域普通文本大小和预览区域普通文本大小**仍然**不一致。

历史修复链（在 spec 80 §9 已记录）：
- **0.2.1** (2026-04-07)：第一次修复，强制 pointSize 严格相等
- **0.2.3** (2026-04-13)：第二次修复，引入 INV-2 的 xHeight 对齐 + `balanceEditorFontSize` 框架
- **2026-05-06**：第三次修复（plan #4），`balanceEditorFontSize` 改为综合 xH/capH/height 三项 score
- **2026-05-07 用户反馈**：第四次报告——仍然感觉不一致

同日发现的 CI 证据（详见 `plans/2026-05-07-CI-FontConsistencyTest-delta-4-Win失败.md`）：
- 同样的 "Segoe UI 8pt" 在本地和 CI Windows runner 上 `QFontMetricsF::height()` 实测差 4px
- `balanceEditorFontSize` 在 CI 上某些 delta 下选不出 INV-2 三项全满足的 pt
- 现实是 **Windows GDI/DirectWrite 字体度量在不同环境物理上不同**——算法假设字体度量是稳定常量，错了

## 推测的真实根因

CI 失败证明：当前"族不同 + 综合 score"路线在跨环境/小字号下**数学上不可能**让 xH/capH/height 三项同时差 ≤ 1-2 px。等宽（Courier New）vs 比例（Segoe UI）的行高系数（height/pt）天生不同（约 1.5 vs 1.7），所以小字号下绝对差几乎注定 ≥ 3px。

## 候选方向（待评估）

A. **改用同一字体族**：编辑区也用 Segoe UI（牺牲等宽，列对齐 / Tab 缩进会乱）——**保守，不推荐**

B. **预览正文也用等宽**：把 preview 的 baseFont 改成 Consolas/Courier New（视觉太"代码感"）——**不推荐**

C. **采用 pointSizeF 亚整数字号**：让 `balanceEditorFontSize` 在 [pt-3, pt+3] 步长 0.5 搜索而非整数——可能让 height 更精细对齐，但 Qt 字号取整规则可能让最终 hint 变化不显著

D. **降低 zoom delta 范围**：当前 ±4 档位太极端，限制到 ±2 档位（±2pt 内字号差）——保守，影响用户体验

E. **维持族不同，接受家族差异**：算法**只严守 xH ≤ 1**（视觉感知的主导），其他两项作为软约束。视觉上"x 高度对齐"足够人眼判定"字号一致"——这是教科书排版理论的依据

F. **加 INV-2 硬约束 filter（CI 修复同步落地）**：先 filter 出满足约束的候选再 score。在能选出的 runner/字体下严格保证 INV-2；选不出的退化到原算法（CI 失败处理）

G. **重新评估字体族**：Segoe UI Mono（Win11 自带）替代 Courier New——同族变体，xH/capH/height 系数应该接近预览的 Segoe UI
