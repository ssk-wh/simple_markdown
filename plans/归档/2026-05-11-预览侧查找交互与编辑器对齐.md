# 预览侧查找交互与编辑器侧对齐（修复三个连环 bug）

## 背景

用户在 2026-05-11 一次会话内连续报告 3 个预览区查找问题：

1. **点 ">" 箭头后搜索框消失**——预期：跳到下一个命中且搜索框保留
2. **有时有内容却搜不到**——输入文本后立即按回车，结果显示 0/0
3. **回车跳转位置错误**——回车永远跳到第一项，忽略用户当前滚动位置

三者同根：`PreviewWidget::findNextHit / findPrevHit` 与编辑器侧
`EditorWidget::findNext / findPrev` 的实现走样（反模式 B：同一语义两份独立代码）。

用户明确要求"保证编辑区和搜索区交互逻辑一致"。

## 根因（确认）

对比 `EditorWidget::findNext`（src/editor/EditorWidget.cpp:653）与
`PreviewWidget::findNextHit`（src/preview/PreviewWidget.cpp:1104）：

| 项 | 编辑器侧 | 预览侧（当前） | 后果 |
|----|----------|---------------|------|
| 早返回 | `text.isEmpty()` | `text.isEmpty() \|\| m_searchHits.isEmpty()` | Bug 2：debounce 未触发时直接退出 |
| 同步重搜触发 | `text!=m_currentSearchText \|\| m_searchMatches.isEmpty()` | 仅 `text!=m_currentSearchText` | Bug 2：首次回车不重搜 |
| 跳转锚点 | 当前 cursor offset → 下一个 > cursor 的命中 | `(m_currentSearchIndex+1) % size`，初始 -1 | Bug 3：永远从第 1 项绕回 |
| keepFocus | 每条 return 路径都调 `m_searchBar->keepFocus()` | 无 | Bug 1：FocusOut 100ms 后 `hideBar()` 触发 |

Bug 1 的隐藏机制：`SearchBar::eventFilter`（src/editor/SearchBar.cpp:143-148）
对 QLineEdit 的 FocusOut 装了 100ms 延迟自动 `hideBar()`。编辑器侧每次跳转后
`keepFocus()` 把焦点送回 `m_findEdit`，100ms 检查通过；预览侧没调用 → 触发隐藏。

## 目标

让预览侧 `findNextHit / findPrevHit` 在以下四个维度与编辑器侧逻辑一致：

- [ ] 输入后立即回车能触发同步重搜（不依赖 100ms debounce）
- [ ] 回车跳转位置基于"用户当前可视区域"，不再永远从第一项开始
- [ ] 所有返回路径调用 `m_searchBar->keepFocus()`，避免按钮点击后搜索框被 FocusOut 关闭
- [ ] 同步搜索使用与 worker 一致的 `caseSensitive / wholeWord / regex` 选项（已有，确认不退化）

## 设计

### 1. 重写 `PreviewWidget::findNextHit(const QString& text)`

```cpp
if (text.isEmpty()) return;  // ← 去掉 m_searchHits.isEmpty()

// 同步重搜：text 改变 OR m_searchHits 还没回填（首次回车 / debounce 未触发）
if (text != m_currentSearchText || m_searchHits.isEmpty()) {
    m_currentSearchText = text;
    m_searchDebounce.stop();
    m_searchHits.clear();
    const Qt::CaseSensitivity cs = ...;
    int pos = 0;
    while ((pos = m_plainText.indexOf(text, pos, cs)) != -1) {
        m_searchHits.append({pos, text.length()});
        pos += text.length();
    }
}

if (m_searchHits.isEmpty()) {
    m_currentSearchIndex = -1;
    m_searchBar->updateMatchInfo(-1, 0);
    viewport()->update();
    m_searchBar->keepFocus();   // ← 补
    return;
}

// 锚点跳转：基于当前可视区域顶部 Y 找下一个命中（位置语义对齐编辑器 cursor 语义）
int targetIdx = pickNextHitIndexByScroll(/*forward=*/true);
m_currentSearchIndex = targetIdx;
scrollToCharOffset(m_searchHits[targetIdx].first);
m_searchBar->updateMatchInfo(m_currentSearchIndex, m_searchHits.size());
viewport()->update();
m_searchBar->keepFocus();   // ← 补
```

### 2. 新增 `pickNextHitIndexByScroll(bool forward)`

类似 `scrollToCharOffset` 的块树遍历，但**预先**为每个 hit.charOffset
求出对应 LayoutBlock.bounds.y，然后：

- forward：从已选 currentIndex 之后开始找下一个；首次（currentIndex==-1）
  找第一个 y > 当前 scrollY 的；都不满足回绕到首项
- backward：对称镜像

复杂度 O(N · K) 其中 N=块数 K=命中数。命中数通常小，可接受。

### 4. 不动 `onSearchTextChanged` / `onSearchResultsReady`

异步 debounce 路径继续用于"输入文本时即时高亮全部命中（INV-1 描述的行为）"，
不与"回车跳转"路径冲突。回车时如果 worker 已经回填好结果，
`m_searchHits.isEmpty()` 为假 → 跳过同步重搜直接跳；否则同步重搜兜底。

## 风险

- 锚点跳转重新计算每个 hit 的 y → 块树大且命中多时可能轻微卡顿。
  缓解：限制为单次调用 walk 一遍即返回，整体仍 < 一次 paintEvent。
- 同步重搜路径与异步 worker 同时被触发可能产生竞态：worker 后续回填 m_searchHits
  覆盖刚同步算出的 hits。缓解：同步重搜后立即调 `m_searchDebounce.stop()` +
  `++m_searchRequestId` 让 worker 回填的旧 reqId 过期被丢弃（line 1094-1095 已有保护）。
