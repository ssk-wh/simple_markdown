# 大文件渲染性能基线脚本（用于优化前后对比验证）

## 背景
用户提出：需要一个**性能基线脚本**——
- 打开一个 1000 行以上的大 Markdown 文件
- **仅显示预览区**（隐藏编辑器）
- **反复触发渲染**（如反复改窗口宽度 / 强制 rebuildLayout）
- **确认每次渲染耗时**

最终用途：**作为优化效果的标尺**，对比 plan「千行文档拖拽分隔条预览重排卡顿」（#10）实施前后的性能差异。

仅记录待办，本次不实现。

## 与其他 Plan 的关系
- **直接配套** `plans/2026-05-06-千行文档拖拽分隔条预览重排卡顿.md` 的 **Step 0（基线测量）**——本脚本就是 Step 0 要用的工具
- 复用 `core/PerfProbe.h` 的 `SM_PERF_SCOPE` 埋点（已用于 `preview.paintEvent`）
- 该脚本是**长期资产**——Tab 休眠（#6）、字号一致性优化（#4）等也可能依赖它做回归测试

## 设计要点
- **脚本类型**：CMake 测试可执行文件（GoogleTest 或独立的 `tests/perf/PreviewRenderBenchmark.cpp`）
  - 优点：纳入 ctest 工作流，CI 可跑；不需要 Python / PowerShell 外部依赖
  - 但需要纯 widget 级，不弹窗
- **测试数据**：项目内 `tmp/` 下放一个生成器脚本，从 docs/ 现有 markdown 拼接出 ≥1000 行的样例（避免测试时随机生成）
- **测量指标**（必须）：
  - `PreviewLayout::buildFromAst` 单次耗时 P50 / P95 / P99
  - `PreviewWidget::paintEvent` 单次耗时
  - 多次连续 `setViewportWidth` + `buildFromAst` 的总耗时
  - 内存峰值（可选）
- **场景覆盖**：
  - 场景 A：宽度从 800 px 拖到 1600 px（每 +20px 触发一次 rebuildLayout，模拟拖拽分隔条）
  - 场景 B：固定宽度，反复 `buildFromAst` 50 次（衡量稳态成本）
  - 场景 C：连续切换 `wordWrap on/off` 各 10 次

## 验收
- 脚本可重复运行，结果稳定（同机同输入下 P95 偏差 < 10%）
- 输出包含：场景标识、运行次数、单次耗时分布、内存峰值
- 可由 ctest 调用，但默认不参与常规测试集（避免影响 CI 时长）

## 实施后基线数字（2026-05-06，未优化版本）

测试机：Win11，1024 行 / 37912 字节样例 markdown，QImage 1600×1200 离屏 device

| 场景 | N | P50 | P95 | P99 | mean | max |
|------|---|-----|-----|-----|------|-----|
| A 拖拽模拟（宽度 800→1600 步进 20） | 41 | **41.4 ms** | **77.8 ms** | 80.9 ms | 43.9 ms | 86.9 ms |
| B 稳态（固定 1200，50 次） | 50 | **43.2 ms** | **67.2 ms** | 73.5 ms | 43.2 ms | 73.7 ms |

**关键结论**：
- 单次 `buildFromAst` P50 ≈ 41–43 ms
- 远超 60fps 单帧预算（16 ms，**超 2.6 倍**）
- 也超 30fps 单帧预算（33 ms，**超 1.3 倍**）
- 拖拽场景 max=87 ms 比稳态 max=73 ms 还差（每帧不同宽度无缓存命中）
- 实锤用户报告的"千行文档调整目录宽度时卡顿"

**plan #10 的优化目标**（量化）：
- **必须**：P50 ≤ 25 ms（不再低于 30fps）
- **理想**：P50 ≤ 10 ms（接近 60fps，留出绘制余量）
- 优化后用同一脚本复跑，对比新 P50 / P95 数字写入 `specs/横切关注点/70-性能预算.md`

## 注意 / 风险
- **测试稳定性**：性能测试容易受机器/系统/温度影响；建议同一机器多次 warmup（前几次抛弃）+ 取 P50/P95 而非平均
- **隔离 GUI 干扰**：脚本中应避免 `show()` 真实显示；用 offscreen QPaintDevice / `QImage` 作为 paint device，让度量稳定
- **DPI 影响**：QFontMetricsF 在不同 DPI 下耗时差异可能很大；脚本应固定一个 DPR（QPaintDevice 上设置）便于跨机器对比
- 与 plan「Tab 休眠」（#6）的 Step 0 类似精神，都是"先测量再优化"——这条 plan 完成后这两个性能 plan 都受益
