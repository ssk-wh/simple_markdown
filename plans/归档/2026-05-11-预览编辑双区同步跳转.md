# 预览编辑双区同步搜索 + 同步跳转（用户愿景）

## 背景

2026-05-11 第三轮反馈用户描述完整愿景：

> 应用打开后旧接收 Ctrl+F，然后只有存在打开得文档，编辑区和预览区又一个在显示，
> 就显示。显示搜索结果得时候，编辑区和预览区域谁显示出来了谁就把搜索结果坐上标记，
> 只有只有编辑区，那么编辑区列出标记搜索结果即可，预览区同理。用户点击上一个下一个
> 的时候，两边的区域同时跳动到搜索结果处

当前 INV-9 是"互斥单栏"——一次只一侧搜索栏可见。用户愿景是"统一搜索 + 双区同步
标记/跳转"，是更大的范式转变。

## 目标（待细化）

- 应用启动后 Ctrl+F 即响应（不依赖文档/焦点）——可考虑显示提示"未打开任何文档"
- 一个搜索框（顶部居中浮层？编辑+预览共享）
- 输入关键词后：编辑器侧搜 raw markdown，预览侧搜 plain text，**各自**算自己的命中并高亮
- 点"上一个/下一个"：两边**同时**滚动到对应命中的位置——需要做 charOffset
  跨域映射（编辑 source line ↔ preview LayoutBlock）

## 设计难点（先列，方便未来动手）

### 1. 跨域字符映射

编辑器字符流：raw markdown source（含 `**bold**` 等标记符号）
预览字符流：plain text（去标记，由 PreviewWidget::extractPlainText 提取）

两侧的"第 k 个匹配项"通常**不对齐**——同一个匹配词在 raw 是 offset A，在 plain
是 offset B，两者无法 1:1 推导（特别是穿过 frontmatter、代码块、内联格式
等场景）。

可选方案：
- A：按 **AST 节点 / source line** 做粗映射。编辑器搜结果 → 找到所在
  `sourceLine` → 找预览中对应 LayoutBlock（编辑/预览之间已有 source-line 双向
  映射，见 `specs/模块-sync/`）→ 滚动到该 block。反向同理。
- B：在 parser 阶段保留每个 AST inline span 的 raw range ↔ plain range，搜索时
  按字符级映射。改动大但精确。

建议先 A 方案（块级精度，与 INV-4 一致）。

### 2. 搜索结果合并 vs 各自独立

- 方案 a：单搜索结果集，按文档顺序排序，每项标注"在编辑器 offset X / 在预览
  block Y"，点上一个/下一个推进统一索引，双侧同时 highlightCurrent + scroll
- 方案 b：双独立结果集，UI 显示 "5 / 12" 是合并后的，但两侧各自维护

方案 a 更接近用户描述"两边同时跳"。

### 3. 一个搜索框 vs 两个

当前两侧各有一个 SearchBar。用户愿景"一个搜索框"——需要：
- 拆 SearchBar 为可独立挂载的浮层 widget
- 由 MainWindow 拥有，不再由 editor/preview 自管
- 编辑器/预览各订阅"搜索文本 / 当前命中"以更新自己的高亮

工作量较大；折衷方案是保留两个 SearchBar，但通过信号联动让它们的输入互相镜像
（用户在任一侧输入都同步到另一侧），视觉上接近"统一搜索"。

### 4. INV-9 互斥要废除

新设计下双侧同时显示搜索栏 + 同时显示搜索结果——INV-9 必须删除或改为"互斥
仅指搜索栏 UI，搜索结果状态同步"。
