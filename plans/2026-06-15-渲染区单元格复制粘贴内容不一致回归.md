## 上次根因（2026-05-13 定位，供本轮排查起点）

- `PreviewWidget::copySelection` 在 walk 中命中**单一 block** 时，按 `sourceStartLine..sourceEndLine`
  切**整行**源文 → 失去 selection 字符精度 → 单元格选一个词吐出整张表。
- 上次修复逻辑：判定 `hitCount == 1 && !anyWholeBlockSelected`（选区是单一 block 的真子集）
  → 用 `m_plainText` 切片（字符精度）；否则保留 sourceLine 切片（跨块/整块保留 markdown 标记）。

**本轮怀疑方向**（待验证，先不动手）：

1. 1.1.x 之后预览复制默认写 `text/plain` + `text/markdown` 双 MIME——可能仅某一种 MIME
   的切片路径回归（如 markdown 路径仍按 block 粒度，无视字符选区）。
2. 后续视口剪裁 / selection 算法改造，可能把上次的 `hitCount==1` 真子集判定条件冲掉了。
3. `m_plainText` 与渲染文本流是否仍一致（反模式 B：两份字符流走样）。

## 复现细节补充（2026-06-16 用户）

> 单元格内容有时候无论是 Ctrl+C 还是右键选择复制，最终粘贴的内容都是**空白**的。

→ 症状是「粘贴为空」而非「内容错乱」。强嫌疑：`copySelection` 的早返回路径——
  `if (start == end) return;`（选区折叠时直接 return，**不写剪贴板** → 粘贴拿到的是
  剪贴板旧内容，若之前为空则粘出空白）。即用户以为选中了单元格内容，但 m_selStart==m_selEnd
  （选区实际为空/折叠），可能因：
  1. 单元格内 hit-test 在某些位置返回相同 index（textIndexAtPoint snap 到同一边界）；
  2. 选区被某操作重置为 collapsed；
  3. m_plainText.mid(start,end) 落在空白/越界区间返回空串。
- 待查：复现"有时候"的条件（哪种单元格/选法触发 start==end 或空切片）；在 copySelection
  早返回处加诊断确认是否走了 start==end 分支。
