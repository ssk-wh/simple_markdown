# 预览元素缩放一致性修复

## 背景

用户反馈：按住 Ctrl + 鼠标滚轮调整字号时，预览区的代码块（```…```）、内联代码（` `）看起来没变化。经排查，问题范围远不止代码块——围栏代码、表格单元格、列表序号、图片占位符及其装饰文字全部不随缩放变化。

根本原因分析：

1. **`PreviewPainter.cpp` 大量硬编码字号**：11 处 `QFont("Segoe UI"/"Consolas", N)` 字面量，完全绕过 `PreviewLayout` 的字体状态。
2. **`PreviewLayout::setFont` 不同步 `m_monoFont`**：只更新 `m_baseFont`，导致代码块字号锁死在构造函数的初始值。
3. **`font_defaults::defaultMonoFont()` 缺 `sizeDelta` 参数**：即便有调用方想传 delta 也没法传。
4. **内联代码的 `pointSizeF * 0.9` 硬编码比例**：与 `m_monoFont` 的派生规则不一致，造成缩放时内联代码与代码块视觉不一致。

## 目标

达成 `specs/横切关注点/80-字体系统.md` 的 INV-8 / INV-9 / T-7 / T-8 / T-9。用户 Ctrl+滚轮缩放后，预览区**所有可视元素**的字号都按 delta 同步变化，不存在"死字号"。

## 风险与回滚

- **风险 1**：装饰元素（图标、文件名）的派生比例选得不合理，视觉上比原版小或大。
  - 缓解：在 delta=0 时的派生值必须等于原硬编码值（12*1.33=15.96≈16），保证零基线一致；然后再看缩放后的视觉。
- **风险 2**：`PreviewPainter` 依赖 `PreviewLayout` 指针生命周期问题（painter 存活时 layout 已析构）。
  - 缓解：Painter 在每次 `paint()` 前从 widget 取新鲜指针；或用弱引用/raw pointer + 文档化生命周期。两者在同一 `PreviewWidget` 中共存，风险很低。
- **风险 3**：表格单元格的高度估计依赖行高，行高依赖 `m_baseFont`，改动后若 layout 的 `estimateParagraphHeight` 仍按冷启动值计算会出现块重叠。
  - 缓解：`setFont` 中同步更新 `m_lineHeight`，并在下一次 paint 的 `updateMetrics` 中再次校准。
- **风险 4**：内联代码从 `* 0.9` 改为 `m_monoFont` 会改变视觉大小（9pt vs 10.8pt）。
  - 缓解：如果用户视觉偏差过大，可以保留 `* 0.9`，改为基于 `m_baseFont.pointSizeF() * 0.9`，但仍然从 `m_baseFont` 派生——决策权在主线程。

## 回滚方案

- `git revert` 本次所有 commit
- 或者分阶段开关：Step 1-2 是向前兼容的（只加参数和 getter），可单独合入；Step 4 风险集中，可独立 PR 便于回滚。
