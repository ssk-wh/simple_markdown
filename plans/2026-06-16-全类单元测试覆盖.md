# 全类单元测试覆盖（输入输出契约 + bug 回归）

## 背景

用户准则（2026-06-16）：

> 理论上每一个类都应该有对应的单元测试，确保函数的输入输出符合预期；然后还要额外
> 增加平时出现的 bug 对应的单元测试。

两层要求：
1. **契约测试**：每个类的公开函数都应有单测，锁定「给定输入 → 预期输出」的契约（不只是
   等 bug 出现才补）。
2. **bug 回归测试**：每个平时出现的 bug 修复时配一条复现该问题的回归测试（已是 CLAUDE.md
   黄金法则 8，且 [[2026-06-15-历史bug单测覆盖审计补齐]] 已审计过历史 bug 这一层）。

本 plan 负责第 1 层的存量补齐与常态化：把「类 → 是否有契约测试」盘清并逐步补。

## 目标（验收条件）

- [ ] 盘点 `src/` 下所有类，建「类 → 对应测试文件 → 有/无契约测试」矩阵
- [ ] 区分可低成本纯逻辑单测的类（parser / core / layout 估算 / 序列化 / 版本比较等）
      与难单测的类（自绘 widget / 异步 / 窗口级交互），前者优先补齐
- [ ] 对纯逻辑类补输入输出契约测试（嵌入 Spec 的 T 编号；无 T 则先补 Spec）
- [ ] 难单测类在对应 Spec 标注替代验证方式（像素截图 / 手动复测）
- [ ] 建立常态：新增类必带契约测试、新修 bug 必带回归测试（写入工作流 / CLAUDE.md 已有第 8 条）

## 设计（待细化）

- 优先级排序：被复用多、逻辑复杂、易回归的类先补（如 MarkdownParser、PreviewLayout 估算、
  Document、RecentFiles、UpdateChecker 版本比较、序列化等）。
- 复用现有测试基建（gtest + QAppFixture，sm_* 库直接链或把 app .cpp 编进测试 exe）。
- 避免反模式 B：测试调用产品真实函数，不复刻算法。
