# 标记应绑定段落而非全文字符偏移（编辑后段落标记不应消失）

## 背景
用户反馈：在预览区添加标记后，**编辑文档**会导致**预览区的标记高亮消失**，但**右侧目录上的标记指示仍然存在**——形成不一致状态。期望：标记应**绑定段落**，编辑某段落时**仅取消当前段落**的标记，其他段落和标题不受影响。

**已确认根因**（在 2026-05-06 内容标记持久化任务中发现，刻意未当场修复）：

- `PreviewWidget::updateAst()` 中 `m_highlights.clear()` 在**每次编辑**都触发（`ParseScheduler::astReady` 是编辑过程也会发的信号，不仅切换文档时）
- 注释「切换文档时清空标记」是错的，作者预期的语义和实际触发时机不一致
- 但 TOC 面板的 `m_tocHighlighted` 是通过 `tocHighlightChanged` 信号同步的，updateAst 没有 emit 该信号——所以 TOC 那边的标记指示**残留**，与已被清空的 `m_highlights` 不一致

仅记录待办，本次不修复（这次需要的是数据模型变更，影响面比单纯延后清空更大）。

## 目标 / 设计方向

用户给出的方向是合理的："标记应绑定段落，编辑某段落时仅取消当前段落的标记内容"。这意味着标记的**锚点**从「全文字符偏移」改为「(段落锚点, 段内偏移)」复合键。

### 候选锚点方案
| 方案 | 锚点 | 跨编辑稳定性 | 跨重启稳定性 | 实现复杂度 |
|------|------|--------------|--------------|------------|
| A. 当前：全文字符偏移 | `(start, end)` | 差（每次编辑全乱） | 差（一处插入全偏） | 简单 |
| B. AST 节点 ID + 节点内字符偏移 | `(nodeId, nodeStart, nodeEnd)` | 较好（仅当节点本身被改时失效） | 差（重建 AST 后 ID 不稳定） | 中 |
| C. 源行号 + 行内偏移 | `(sourceLine, colStart, colEnd)` | 中（行号也会随编辑漂移） | 较好（行号是文本内置坐标） | 中 |
| D. 段落起始源行号 + 段内字符偏移 | `(blockSourceLine, blockOffset, len)` | **较好**（段内编辑可比对内容；段间编辑只影响插入/删除段） | **较好**（持久化只需文件内容稳定） | 中 |
| E. 段落内容 hash + 偏移 | `(blockHash, offset, len)` | 段内编辑 hash 失效，跨段稳定 | 较好 | 中-高 |

**初步建议方向 D**：用 AST 节点的 `sourceLine`（节点在源 markdown 中的起始行）作为段落锚点，段内字符偏移作为细粒度位置——AST 节点本身已有 sourceLine 字段（用于编辑器/预览滚动同步），复用即可。

### 配套行为
- **段内编辑**：仅当前段落 `sourceLine` 命中的标记被清除（用户的"取消当前段落标记"诉求）
- **段间编辑**（添加/删除整段）：sourceLine 漂移；用 AST diff 或近似策略迁移标记，匹配不到的静默丢弃（§8.3 漂移容忍延伸）
- **TOC 联动**：TOC 的标记指示同步刷新（修复目前"目录标记残留"的不一致）
- **持久化**：序列化格式从 `(start, end)` 改为 `(blockSourceLine, blockOffset, len, [optional content_snippet])`；旧格式（version=1）保持向后兼容（仅按字符偏移恢复，段间编辑后必然丢失，记录为已知限制）

## 验收
- 添加 5 处不同段落的标记 → 编辑其中第 2 段 → 仅第 2 段的标记消失，其余 4 段保留
- 上述操作后右侧目录的标记指示与预览区保持一致（不再出现"预览没了，目录还在"的不同步）
- 添加标记后保存关闭重启 → 标记仍在原位置（与 INV-5 一致）
- 旧版本（v1 字符偏移）的会话数据仍可读取，越界条目静默丢弃

## 注意 / 风险
- 这是**数据模型改动**，比单纯修"清空时机"风险大；务必先做 Step 1 的小修复（TOC 同步），让 bug 先变得"可见但不诡异"，再做大改造
- AST 节点的 sourceLine 字段在某些 inline node（如 emph 内部）上是否稳定？需要确认
- 跨段编辑时的"标记迁移"必然有歧义，必须明确"匹配不到就丢弃"的策略，避免标记跑到错误位置
- 与 plan「内容标记持久化」（已完成）共享 serializeMarkings/deserializeMarkings 接口；本 plan 涉及格式 version 升级，要保证向后兼容
