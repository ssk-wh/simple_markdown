# 标记内容与目录标记不对应

## 背景
用户反馈：预览区的标记（荧光笔）和右侧目录 TOC 的标记指示**不对应**——出现"两边状态不一致"的视觉异常。

期望（来自 Spec `08-内容标记.md` INV-4）：标记保存后必须更新 `m_headingCharOffsets`，TOC 面板据此判断每个标题下是否有标记；标记的增删/恢复都应让 TOC 同步刷新。

仅记录待办，本次不修复。

## 复现要点（待用户/复测时补充精确步骤）
现象的"不对应"可能涵盖多种子场景，需要逐一区分：

1. **TOC 残留**：预览区已无标记（被编辑或清除），但 TOC 上对应章节仍显示有标记
2. **TOC 缺失**：预览区有标记，但 TOC 上对应章节没显示
3. **章节归属错位**：标记在章节 A 内，TOC 却把章节 B 标为有标记
4. **持久化恢复后不同步**：重启恢复标记后预览有，但 TOC 一直空着，直到滚动 / 切换 Tab 才刷新
5. **跨章节标记**：选区跨越两个标题，TOC 上只标了其中一个章节（或两个都没标）

## 已知相关问题（已分析）

### 路径 A：updateAst 清空但不 emit
`PreviewWidget::updateAst` 在每次编辑都触发 `m_highlights.clear()`（line 94），但**没 emit `tocHighlightChanged({})`**——`m_tocHighlighted` 不变，TOC 面板继续显示旧标记。这是子场景 1（TOC 残留）的已知根因，已记录在 `plans/2026-05-06-标记应绑定段落而非全文字符偏移.md` 的 Step 1（先做的小修）。

### 路径 B：持久化恢复后 TOC 没刷新
2026-05-06 实施的内容标记持久化（plan「内容标记持久化未实现」）中，恢复路径调 `deserializeMarkings`。当 `m_currentAst` 已就绪时立即调 `updateTocHighlights` + `viewport()->update()`；但 lazy tab 下 AST 还未就绪，标记被存到 `m_pendingMarkings`，由 `updateAst` 末尾的 `applyPendingMarkings` 兑现——**该路径中调用了 `updateTocHighlights` 但 TOC 信号是否到达 TocPanel 取决于 connect 时序**。需要核实：
- lazy tab 切换激活后第一次绘制时，TocPanel 是否已 connect 到该 PreviewWidget 的 `tocHighlightChanged`
- `addPage` / 切 Tab 路径中信号 connect 顺序是否保证 TOC 能接收到首次 emit

### 路径 C：章节归属算法
`updateTocHighlights()`（PreviewWidget.cpp line 751-774）按 `m_headingCharOffsets` 区间判定章节是否含标记：
- 章节 i 范围 = `[headingCharOffsets[i], headingCharOffsets[i+1])`
- 末章节 = `[headingCharOffsets.back(), m_plainText.length())`
- 如果标记 `[hl.first, hl.second)` 与章节范围有交集，则章节 i 被标亮

可能的盲点：
- 跨章节标记：标记从章节 A 末尾延伸到章节 B 开头，应当**两个章节都被标亮**——当前代码确实是 `if (hlStart < headEnd && hlEnd > headStart)` 取交集判断，理论正确，但 m_headingCharOffsets 是否准确收录了**所有**层级的标题（h1/h2/h3...）？看 `buildHeadingCharOffsets` 实现确认是否只对顶层 Heading 计数。
- 持久化恢复时 `buildHeadingCharOffsets` 是否一定先于 `applyPendingMarkings` 执行？看 `updateAst` 末尾顺序：
  ```
  m_highlights.clear();
  buildHeadingCharOffsets();  ← 先重建章节索引
  applyPendingMarkings();     ← 再恢复标记并 updateTocHighlights
  ```
  顺序正确，但若 buildHeadingCharOffsets 在某些 AST 形态下产出错误索引，TOC 联动会跑偏。
