# 拖拽无后缀文本文件 + 不支持文件提示

## 主线程裁定（2026-04-13）

5 个待决策点的方向已锁定：

1. **白名单宽窄：宽松**——任何能通过 sniff 判定为文本的都允许，黑名单只拦明确二进制（`.exe/.dll/.zip/.png/.jpg/.mp4/.pdf/.docx` 等）
2. **dragEnter 拒绝时机：晚拒绝**——dragEnter 全接受，drop 时再校验，拒绝时弹窗
3. **EditorWidget 非图片拖拽：冒泡到 MainWindow** 自动开 Tab（不在编辑区当场插入文件链接 markdown）
4. **菜单"打开"是否 sniff：只在明显二进制时拦截**（用户主动选的文件保留宽容）
5. **二进制阈值：控制字符占比 > 10%**（不强求支持 UTF-16 文本，简化判定）

实施时按上述决策直接执行。

## 背景

用户报告：

1. 当前拖拽到主窗口只支持 `.md` / `.markdown` / `.txt` 三种扩展名（`MainWindow.cpp` 中的 `isMarkdownFilePath`），无法拖拽项目里常见的**无后缀纯文本文件**（如 `TODO`、`LICENSE`、`README`、`.gitignore`、`CMakeLists.txt` 等）。
2. 拖入不支持的文件（图片之外的二进制：`.exe`、`.zip`、`.pdf` 等）时，当前在 `dragEnterEvent` 中不调用 `acceptProposedAction()`，结果是**静默丢弃**——光标变成禁止符号，用户没有任何反馈，体验差。
3. 编辑区（`EditorWidget.cpp` 的 dropEvent）目前只识别图片插入，不支持文件类型时没有提示。

### 现状摘要（调研结论）

| 位置 | 当前行为 |
|------|---------|
| `src/app/MainWindow.cpp:983` `isMarkdownFilePath` | 硬编码白名单：`.md` / `.markdown` / `.txt` |
| `src/app/MainWindow.cpp:990` `dragEnterEvent` | 只对 markdown/图片文件调用 `acceptProposedAction()`，其他静默忽略 |
| `src/app/MainWindow.cpp:1003` `dropEvent` | 图片调 `insertImageMarkdown`，markdown/文本调 `openFile`，其他类型在 dropEvent 之前已被 dragEnter 拒绝，**不会**走到这里 |
| `src/editor/EditorWidget.cpp:791` 拖拽事件 | 仅识别图片，其他类型走父类默认行为 |

## 动作

- [ ] **扩展 `isMarkdownFilePath` 为更宽的判定函数**（如重命名为 `isOpenableTextFile`），纳入：
  - 现有：`.md` / `.markdown` / `.txt`
  - 新增常见纯文本扩展：`.log` / `.conf` / `.cfg` / `.ini` / `.json` / `.yaml` / `.yml` / `.toml` / `.xml` / `.csv` / `.tsv` / `.gitignore` / `.gitattributes` / `.editorconfig` / `.env`（按需精简）
  - 新增 CMake/Make 类：`CMakeLists.txt` 已被 `.txt` 覆盖；`Makefile`、`Dockerfile`、`Jenkinsfile` 等无后缀但有特定 basename 的文件，加 basename 白名单
  - 新增**无后缀文件**：通过下方 sniff 判定
- [ ] **二进制嗅探（sniff）**：对无后缀或非白名单扩展的文件，读前 4 KB 判定是否文本：
  - 含 `\x00`（NUL 字节）→ 视为二进制，拒绝
  - 控制字符（`\x01-\x08`、`\x0B`、`\x0C`、`\x0E-\x1F`，排除 `\t \n \r`）占比 > 10% → 二进制，拒绝
  - 否则尝试 UTF-8 解码（`QString::fromUtf8` + 通过 `QTextCodec` 或检测 BOM），失败再尝试 GBK/系统本地编码；都失败则拒绝
  - 参考 git 的 `buffer_is_binary` 逻辑（前 8000 字节内出现 NUL 即为二进制）
- [ ] **明确二进制扩展名黑名单**（即使文件改名也快速拒绝）：`.exe` `.dll` `.so` `.dylib` `.zip` `.7z` `.rar` `.tar` `.gz` `.bz2` `.pdf` `.docx` `.xlsx` `.pptx` `.mp3` `.mp4` `.avi` `.mkv` `.iso` `.bin` `.obj` `.o` `.a` `.lib` `.class` `.jar`
- [ ] **图片**保持现有行为（拖到编辑区插入 markdown，拖到主窗口空白处也走插入）
- [ ] **拒绝时弹 QMessageBox**（不再静默）：
  - 标题：`tr("Cannot Open File")`
  - 单文件正文：`tr("\"%1\" appears to be binary or an unsupported format. SimpleMarkdown only opens Markdown and plain text files.").arg(fileName)`
  - 多文件正文：合并成一条消息，列出所有被拒绝的文件名（避免连弹多个弹窗）
  - 按钮：OK
- [ ] **多文件混合拖拽**：
  - 逐个校验
  - 支持的文件正常打开（仍按现有逻辑：图片插入 / 文本打开 Tab）
  - 不支持的文件**累积**到列表，结束时一次性弹一个 QMessageBox 列出所有被拒文件名
- [ ] **dragEnterEvent 放宽**：当拖入包含**至少一个**可能可打开的文件（包括无后缀文件）时，就 `acceptProposedAction()`，让 drop 阶段才做精确判定。否则光标永远是禁止符号，用户根本无法触发提示。
- [ ] **EditorWidget 同步处理**：编辑区的 dragEnter/drop 现在只认图片。拖入文本类文件时应**冒泡到 MainWindow**（或直接调 MainWindow 的 openFile），保持行为一致。
- [ ] **浏览器拖拽 URL** (`text/uri-list` 中的 `http://` 或 `https://` 而非 `file://`)：当前 `url.toLocalFile()` 会返回空字符串，需要识别这种情况并给出"暂不支持网络 URL"的提示，而非崩溃或静默。
- [ ] **Outlook 等邮件附件拖拽**：mime 可能不是 `text/uri-list` 而是 `application/x-qt-windows-mime;value="FileGroupDescriptor"` 等，**最低要求是不崩溃**。短期方案：检测到无 `hasUrls()` 时直接 `event->ignore()`。
- [ ] **统一打开路径**：将 `MainWindow::openFile` 内部也加上 sniff 校验（防止用户通过菜单 "Open..." 强行选了一个二进制文件）。最近文件菜单点击同理。
- [ ] **更新 Spec**（实施阶段，不在本 Plan）：在 `specs/模块-app/03-文件操作与拖拽.md` 中追加：
  - INV：无后缀文件必须通过前 4KB 二进制嗅探判定才允许打开
  - INV：拒绝文件时必须弹 QMessageBox 给出原因，禁止静默丢弃
  - INV：拖拽多个文件时，被拒绝的文件名必须合并到一个弹窗中显示
  - INV：编辑区拖拽除图片外的文本文件，应被冒泡处理（不能丢弃）
  - 对应 T 用例
- [ ] **CHANGELOG**：在 `[0.2.3]` 段追加：
  - `### Added`：支持拖拽无后缀纯文本文件（TODO / LICENSE / .gitignore 等）
  - `### Fixed`：拖入不支持的二进制文件时弹窗提示，不再静默丢弃
