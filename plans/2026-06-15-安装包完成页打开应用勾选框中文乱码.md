# 安装包完成页"打开应用"勾选框中文乱码

## 背景

用户报告（2026-06-15）：

> GitHub Action 生成的安装包，安装的最后一步，勾选框（安装结束后打开应用的"立即启动"
> 选项）这里，中文内容是乱码。

对应 NSIS 完成页的 `MUI_FINISHPAGE_RUN_TEXT`（「立即启动 SimpleMarkdown」）。

## ⚠️ 关键定位：本地配置看起来是对的 → 怀疑 CI 环境

`installer/SimpleMarkdown.nsi` 现状（已检查）：

| 项 | 现状 | 说明 |
|----|------|------|
| `Unicode True` | ✅ 第 22 行已设 | Unicode 安装器，支持中文 |
| 文件编码 | UTF-8 **with BOM** | NSIS 3 Unicode 模式可识别 |
| 语言包 | `MUI_LANGUAGE "SimpChinese"` | 简体中文 |
| 文案 | `MUI_FINISHPAGE_RUN_TEXT "立即启动 ..."` | 中文字面量 |

本地若打包正常 → 乱码大概率出在 **GitHub Actions 这一侧**，怀疑方向：

1. **checkout 改了文件编码/BOM**：CI 的 git `core.autocrlf` / `.gitattributes` 处理可能
   剥离 UTF-8 BOM 或转 CRLF↔LF，导致 makensis 误判编码 → 中文字面量乱码。
2. **CI 的 makensis 版本/编码默认**：Actions runner 上的 makensis 版本对无 BOM / 特定
   代码页源文件处理与本地不同。
3. **控制台代码页**：CI 里调用 makensis 的 step 未设 UTF-8 代码页（chcp 65001），
   日志/传参环节把中文转坏（若文案来自变量而非字面量）。

## 进展（补充）

- 2026-06-15 定位 + 修复（CI 侧）：
  - 根因确认：`.github/workflows/build.yml` 注入版本号步骤用 pwsh
    `(Get-Content $nsi) -replace ... | Set-Content $nsi`。pwsh7 的 Set-Content 默认 utf8NoBOM，
    **写回时剥离了 UTF-8 BOM**。NSIS `Unicode True` 靠 BOM 识别 UTF-8 源；无 BOM 时按 runner
    系统代码页（西文）解析中文 → 完成页"立即启动"乱码。本地直接 makensis 不经此 round-trip 故正常。
  - 修复：读写均强制 `-Encoding utf8BOM`，保留 BOM。
  - 双保险：新增根目录 `.gitattributes` 把 `installer/*.nsi` 标为 `-text`，禁止 checkout 时
    行尾/编码规范化剥离 BOM。
  - 验证：根因与修复逻辑确定；CI 产物需下次 GitHub Action 发版后下载安装包确认完成页中文正常
    （本地无法复现 CI 环境，属外部系统验证）。
