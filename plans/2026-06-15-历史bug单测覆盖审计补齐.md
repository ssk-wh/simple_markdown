# 历史 bug 单测覆盖审计与补齐

## 背景

用户准则（2026-06-15）：

> 开发过程中出现的 bug 都应该有单元测试可以保障不会再次对应问题。之前的 plans 记录的问题也检查一下是否有相关的单元测试，没有的需要补充。

已沉淀为 CLAUDE.md 黄金法则第 8 条：**每个 bug 修复必须配回归单元测试**。

本 plan 负责存量审计：把 `plans/` 与 `plans/归档/` 里所有 `type: bug|fix` 的条目逐一核对，
确认是否存在对应的回归单元测试（按 Spec 的 T 编号或 plan 关联测试定位），缺失的补齐。

## 目标（验收条件）

- [ ] 列出所有 `plans/`（含归档）中 `type: bug|fix` 的条目清单
- [ ] 对每条逐一判定：是否有对应单元测试（grep Spec `tests:` 字段 + tests/ 实际文件）
- [ ] 产出覆盖矩阵：bug → 对应测试文件/T 编号 → 有/无
- [ ] 对「可写复现测试但缺失」的，补齐单元测试（嵌入 T 编号）
- [ ] 对「纯视觉/外部环境无法单测」的，在对应 plan 验证段标注替代验证方式
- [ ] 命中红旗反模式 D（Spec tests 字段写了但 tests/ 零匹配）的，要么补实现要么修正 Spec status

## 设计

分两类处理：
1. **逻辑类 bug**（分词、切片、布局估算、序列化、宽度持久化等）→ 必须有 gtest 回归测试
2. **纯视觉/像素类 bug**（选区高度、基线抖动、空白等）→ 优先做像素级渲染测试（渲染到 QImage
   扫像素），实在无法则截图对比 + plan 标注
