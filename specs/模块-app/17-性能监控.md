# 性能监控（PerfProbe）

## 1. 目的

为开发者提供**可开关的运行时埋点**，向 stderr 打印关键路径（解析 / 布局 / 绘制）的耗时，帮助定位实际性能瓶颈是否突破 Spec `70-性能预算.md` 的预算。

V1 定位为**诊断工具**：不做面板、不做 UI 图表，只输出文本日志（grep / awk 足够分析）。
通过命令行参数 `--debug` / `-d` 启用，从终端启动时可直接查看 stderr 输出。

## 2. 输入 / 输出

- **输入**：
  - 命令行参数 `--debug` / `-d`（启动时读取一次，优先级最高）
  - 环境变量 `SM_PERF=1` / `true` / `on`（启动时读取一次，命令行未指定时回退）
- **输出**：
  - stderr 文本行：`[perf] <name>: <elapsed> us\n`，例如：
    ```
    [perf] parser.doParse: 1823 us
    [perf] preview.buildFromAst: 4271 us
    [perf] preview.paintEvent: 312 us
    ```
- **不输出**：
  - 不写文件
  - 不发送网络请求
  - 不在 UI 上显示（菜单 checkable 状态反映开关本身）

## 3. 行为契约（不变量）

### [INV-1] Release 构建零开销

`core::PerfScope` 构造函数先读 `perfEnabled()`（`std::atomic<bool>` relaxed load），
若为 `false` 则**不启动** `QElapsedTimer`；析构时同样先判 `m_enabled`，不开启时只有一次 bool 判断即 return。

`QElapsedTimer::start()` 本身是 syscall 级开销，关闭时必须跳过。实测关闭开关后开销 < 5ns / scope。

### [INV-2] 全局单一开关

`core::g_perfEnabled` 是 `inline std::atomic<bool>`（C++17 inline variable），
多 translation unit 共享同一实例。使用 `std::memory_order_relaxed` — 精度与诊断需求匹配（少量 race 下的报告缺失可接受）。

### [INV-3] 埋点位置（V1 固定 3 个核心路径）

| 名字 | 位置 | 含义 |
|------|------|------|
| `parser.doParse` | `ParseWorker::doParse` | Markdown → AST 的工作线程解析时间 |
| `preview.buildFromAst` | `PreviewLayout::buildFromAst` | AST → LayoutBlock 树构建（主线程） |
| `preview.paintEvent` | `PreviewWidget::paintEvent` | 预览区一次 paintEvent 总耗时（主线程） |

后续可逐步增加（编辑器 syntax highlight、滚动事件等），但必须**保持 name 稳定**——外部 grep/awk 脚本依赖这些字符串。

### [INV-4] 使用宏封装，行号唯一化

用户代码用宏 `SM_PERF_SCOPE("name")`，不直接写 `PerfScope`。宏用 `SM_PERF_CONCAT(_sm_perf_, __LINE__)` 生成变量名，允许同一个 scope 里多次埋点而不冲突。

### [INV-5] 线程安全

- 埋点本身无线程竞争：每个 `PerfScope` 实例是栈变量，本地存储
- 全局开关用 atomic，多线程读写安全
- stderr 写入理论上有竞争（Windows 的 `fprintf` 内部有锁但不强保证有序），**允许**不同线程的行交错——诊断工具不需要严格顺序

### [INV-6] 默认关闭

应用冷启动后 `g_perfEnabled` 为 `false`。即便 debug build 也默认关闭——避免开发者偶尔看 stderr 被大量日志干扰。

### [INV-7] 开关状态不持久化

`g_perfEnabled` **不**写入 `QSettings`：这是"本次会话诊断用"的短期开关，下次启动默认关闭。
若开发者要长期开启，用 `SM_PERF=1` 环境变量配合启动脚本。

## 4. 接口

### 4.1 `core/PerfProbe.h`（header-only）

```cpp
namespace core {
    inline std::atomic<bool> g_perfEnabled{false};
    inline void setPerfEnabled(bool on);
    inline bool perfEnabled();

    class PerfScope {
    public:
        explicit PerfScope(const char* name);
        ~PerfScope();
        // non-copyable, non-movable
    };
}

#define SM_PERF_SCOPE(name_literal)  ::core::PerfScope <unique>(name_literal)
```

### 4.2 埋点使用示例

```cpp
void MyFunc() {
    SM_PERF_SCOPE("module.operation");
    // ... 工作 ...
}
```

### 4.3 启动参数读取（`main.cpp`）

```cpp
// 命令行参数优先
bool debugMode = false;
for (int i = 1; i < argc; ++i) {
    if (strcmp(argv[i], "--debug") == 0 || strcmp(argv[i], "-d") == 0) {
        debugMode = true;
        break;
    }
}
// 回退到环境变量
if (!debugMode) {
    const QByteArray envPerf = qgetenv("SM_PERF");
    if (envPerf == "1" || envPerf.compare("true", Qt::CaseInsensitive) == 0
                       || envPerf.compare("on",   Qt::CaseInsensitive) == 0) {
        debugMode = true;
    }
}
if (debugMode) core::setPerfEnabled(true);
```

## 5. 性能预算

| 操作 | 开启时预算 | 关闭时预算 |
|------|-----------|-----------|
| 单次 PerfScope 构造+析构 | < 1 us（含 QElapsedTimer start/nsecsElapsed） | < 10 ns（仅 atomic load + bool 判断） |
| 开/关切换 | < 100 ns（atomic store） | — |
| stderr 单行写入 | 受 OS / 终端影响，典型 < 50 us | — |

## 6. 验收条件

- **[T-PERF-1]** 开关关闭（默认）时启动应用、正常编辑，**stderr 无任何 `[perf]` 行**输出
- **[T-PERF-2]** `SimpleMarkdown.exe --debug` 启动：stderr 出现 `[perf] enabled via command line`；随后编辑文档，每次触发解析/绘制都看到对应 `[perf] parser.doParse / preview.buildFromAst / preview.paintEvent` 行
- **[T-PERF-3]** `-d` 短参数与 `--debug` 效果一致
- **[T-PERF-4]** 设环境变量 `SM_PERF=1` 启动（不加命令行参数）：stderr 出现 `[perf] enabled via SM_PERF env var`，行为同 `--debug`
- **[T-PERF-5]** Release 构建，开关关闭：连续编辑 1MB 文档 30s，不感到卡顿（埋点开销几乎不可测）
- **[T-PERF-6]** 输出格式严格为 `[perf] <name>: <integer> us`（便于 grep/awk），时间单位固定微秒

## 7. 已知陷阱

### 7.1 stderr 输出被吞

Windows GUI 应用（`WIN32` subsystem）默认无控制台窗口，`fprintf(stderr, ...)` 可能被丢弃。使用 `--debug` / `-d` 参数时，用户自然是从终端启动，因此 stderr 可达。

### 7.3 埋点覆盖完整度

V1 只埋 3 个核心路径。如果性能问题出在其他路径（如 TOC 更新、highlight 缓存失效），现有埋点看不到。由 INV-3 说明，后续按需扩展——追加埋点不影响已有 name。

### 7.4 开关切换时的原子性

从"关"切到"开"的瞬间，正在执行中的 `PerfScope` 实例的 `m_enabled` 字段是构造时读的 false，所以即便中途开了开关也不会在析构时输出（反之亦然）。这是预期行为——一个 scope 的开关状态由入口决定，避免 race。

### 7.5 Header-only 的 inline variable

C++17 `inline std::atomic<bool> g_perfEnabled{false};` 在多 .cpp 中 include 后仍然是同一对象（inline variable 规则）。Qt 5.12 + MSVC 2019 + Clang 10+ + GCC 7+ 都支持。
