# 编辑/预览分隔条拖拽吸附刻度线

对应源文件：`src/app/SnapSplitter.h/cpp`、`src/app/MainWindow.cpp`（createTab）

## 1. 背景

每个 Tab 的 editor | preview 用 `QSplitter(Qt::Horizontal)` 分栏，用户希望拖拽分隔条时有常用比例的视觉参考线，并在靠近刻度时自动吸附，方便还原到常见等分/1:3/3:1 布局。

## 2. 行为定义

### 2.1 视觉刻度

拖拽开始（mouse press on handle）到拖拽结束（mouse release）之间，在 splitter 区域覆盖一层透明 overlay，绘制 3 条垂直虚线：

- 位置：splitter 宽度的 25% / 50% / 75%
- 样式：1 像素虚线（`Qt::DashLine`，pattern 建议 `{4, 4}` 逻辑像素）
- 颜色：`theme.accentColor`，alpha = 96（非激活）/ 200（激活/命中吸附）
- 非拖拽状态：overlay 隐藏，不占空间、不拦截鼠标

### 2.2 吸附逻辑

- 阈值：5 逻辑像素（固定，V1 不暴露配置）
- 判断：对拖拽中的 `pos`（handle 左边 pane 的宽度），若 `|pos - target_i| <= 5`，则 `setSizes({target_i, total - target_i})` 强制对齐
- 命中时对应虚线切换到"激活"颜色
- 防递归：`setSizes` 会再次触发 `splitterMoved`，用 `m_snapping` guard 屏蔽一次

### 2.3 只适用 editor|preview splitter

- 主窗口的 content|toc 主 splitter 不加吸附（TOC 有自适应宽度逻辑）
- 演示模式下 editor 被收起（`setSizes({0, 1})`），handle 宽度 0 仍存在但用户无法拖动——overlay 不显示（无 press 事件）

## 3. 不变量（INV）

- **[INV-SNAP-THRESHOLD]** 吸附阈值固定为 5 逻辑像素
- **[INV-SNAP-TARGETS]** 吸附目标固定为 25% / 50% / 75%，基于 `splitter->width()` 实时计算
- **[INV-SNAP-VISIBLE-ON-DRAG]** overlay 仅在 handle press→release 期间可见
- **[INV-SNAP-TRANSPARENT]** overlay 必须 `Qt::WA_TransparentForMouseEvents`，不得拦截 handle/editor/preview 的鼠标事件
- **[INV-SNAP-THEME]** overlay 颜色随主题 `accentColor` 实时更新
- **[INV-SNAP-NO-RECURSION]** 吸附产生的 `setSizes` 必须带 guard，禁止递归触发 `splitterMoved`
- **[INV-SNAP-NO-COLLAPSE]** editor / preview 任一侧都不得被拖到宽度 0：
  - `SnapSplitter` 构造函数必须 `setChildrenCollapsible(false)`
  - `MainWindow::createTab` 必须给 `tab.editor` / `tab.preview` 都设 `setMinimumWidth(80)`
  - 合起来保证 handle 始终留有至少 80px 两侧可视空间 + handle 本身可抓
  - 吸附算法命中最极端目标（25% / 75%）时不与本不变量冲突——25% × 窗口宽 > 80 在常见窗口尺寸下恒成立
- **[INV-SNAP-DOUBLE-CLICK-RESET]** 双击 handle 强制重置为 50/50：
  - `SnapSplitterHandle::mouseDoubleClickEvent` 覆写，调 `splitter->setSizes({w/2, w/2})`
  - 作为 INV-SNAP-NO-COLLAPSE 的救急通道：即使未来某路径绕过 min width，用户双击仍能恢复对称
- **[INV-SNAP-LAZY-RESIZE-VISUAL]** SnapSplitter 必须设 `setOpaqueResize(false)`：
  拖拽期间子 pane 几何**不实时变化**，Qt 内置 rubber band 在原位置画一条参考虚线，
  松手时 base class 一次性 setSizes 到目标尺寸。
  - 视觉动机：之前 OpaqueResize=true + INV-SNAP-LAZY-PANE-REBUILD 组合的实测体验是
    "容器宽度跟随手柄但内容布局不变"——用户看到旧文字被新容器裁切/留白，比 lazy resize
    更混乱。OpaqueResize=false 让两侧容器维持原状直到松手，参考虚线指示目标位置。
  - 代价：吸附辅助线（SnapOverlay 三条虚线）拖拽过程中**不高亮**——`onSplitterMoved`
    在 OpaqueResize=false 下只在 release 时 emit 一次。吸附行为退化为"松手时跳到刻度"
    （刻度位置仍正确吸附，只是缺少拖拽中的实时高亮反馈）。
  - 与 INV-SNAP-LAZY-PANE-REBUILD 协同：mouseRelease 时 base class setSizes 触发子
    widget resizeEvent，守护检测 smSnapDragging=true 跳过 rebuild；随后 endDrag 清 property
    并 emit dragFinished；MainWindow slot 触发完整 rebuild 完成最终态对齐。
- **[INV-SNAP-LAZY-PANE-REBUILD]** 拖拽期间子 pane（editor / preview）跳过全文 layout 重建，
  松手时由 `dragFinished` 信号触发一次完整 rebuild：
  - 性能动机：`PreviewLayout::buildFromAst` + `EditorLayout::rebuild` 在每个 mouseMove
    帧都跑全文重排，叠加 `paintEvent` 后单帧成本远超 `specs/横切关注点/70-性能预算.md`
    INV-8 的 16ms 预算（1024 行文档实测拖拽体感卡顿，即使 `buildFromAst` 算法本身
    `PreviewRenderBenchmark` 显示 P50 = 11.4ms 也不够——benchmark 是离屏算法基线，
    不含 paintEvent / 跨 widget 联动）
  - 实现机制：
    - `SnapSplitter::beginDrag` 在 splitter 自身设 dynamic property `smSnapDragging = true`
    - `SnapSplitter::endDrag` 清 property = false，并 emit `dragFinished()` 信号
    - `EditorWidget::resizeEvent` / `PreviewWidget::resizeEvent` 检查
      `parentWidget()->property("smSnapDragging").toBool()`，true 时仅 `updateScrollBars()`
      不重排
    - `MainWindow::createTab` 把 `dragFinished` 连到 `editor->recomputeWrapForCurrentWidth()`
      + `preview->rebuildLayout()` 完成最终态对齐
  - 解耦保证：`sm_preview` / `sm_editor` 模块**不**反向 include `src/app/SnapSplitter.h`，
    通过 dynamic property 传递拖拽态——避免架构反向依赖
  - 视觉权衡：拖拽期间内容不实时跟随宽度（与 macOS Pages / Visual Studio Windows / Typora
    类似的"内容延迟到松手"风格），换取 handle 流畅跟手 + 吸附辅助线实时响应

## 4. 算法

```
on handle mousePress:
    overlay.setActive(-1)
    overlay.setGeometry(splitter.rect())
    overlay.show(); overlay.raise()
    splitter.setProperty("smSnapDragging", true)  // INV-SNAP-LAZY-PANE-REBUILD

on splitterMoved(pos, 1):
    if snapping: return
    W = splitter.width()
    targets = {0.25*W, 0.5*W, 0.75*W}
    hit = index i such that |pos - targets[i]| <= 5 (最近的)
    overlay.setActive(hit)  // -1 表示没命中
    if hit != -1:
        snapping = true
        setSizes({targets[hit], W - targets[hit] - handleWidth})
        snapping = false

on handle mouseRelease:
    overlay.hide()
    overlay.setActive(-1)
    splitter.setProperty("smSnapDragging", false)  // INV-SNAP-LAZY-PANE-REBUILD
    emit dragFinished()                            // 触发各 pane 一次性最终重排

on splitter resize while dragging:
    overlay.setGeometry(splitter.rect())  // 跟随宽度
```

## 5. 主题集成

- `MainWindow::applyTheme` 调用 `snapSplitter->setAccentColor(theme.accentColor)` 同步给 overlay
- 主题切换时若 overlay 可见（拖拽中切主题），下一次 `paintEvent` 自动使用新色

## 6. 高 DPI

- 虚线 dash pattern 用逻辑像素，Qt 会按 devicePixelRatio 自动缩放
- 吸附阈值 5 逻辑像素在 200% DPI 下物理像素约 10 像素，手感符合预期

## 7. 验收条件

- **T1** 静止状态未拖拽：splitter 区域无虚线，editor/preview 渲染不受影响
- **T2** 鼠标按住 handle：overlay 显示 3 条等距虚线，位置分别对应 splitter 宽度 25% / 50% / 75%
- **T3** 拖到 50% 附近（|Δ| ≤ 5px）：尺寸自动吸附到精确 50%，中间虚线高亮
- **T4** 拖到 25% / 75% 附近同样吸附 + 高亮
- **T5** 拖到非吸附区（所有目标距离 > 5px）：正常跟随，无吸附，3 条虚线均非激活态
- **T6** 松开鼠标：overlay 立即消失
- **T7** 高 DPI（175%/200%）下虚线位置按逻辑像素准确对齐到比例点
- **T8** 拖拽中切主题：虚线颜色同步（下次 mouseMove 或 repaint 生效）
- **T9** 演示模式下 editor 被 setSizes({0,1}) 收起：不触发 overlay（handle 宽度不变但用户不拖）
- **T10** Tab 切换：上一个 Tab 的 overlay 随其 splitter 隐藏，不会"串"到新 Tab
- **T11** 拖 handle 到最右：预览区宽度保持 ≥ 80px，handle 始终可见可再次抓取，松开后可以拖回
- **T12** 拖 handle 到最左：编辑区宽度保持 ≥ 80px，同样可恢复
- **T13** 双击 handle：editor / preview 立即变为 50/50 布局（±1px 误差，handleWidth 占用）
- **T14** 吸附刻度（25 / 50 / 75）在 INV-SNAP-NO-COLLAPSE 约束下仍然全部可达——窗口宽度正常（≥ 320px）时 25% × width ≥ 80，吸附不被 minimumWidth 顶死
- **T16** Lazy resize 视觉（INV-SNAP-LAZY-RESIZE-VISUAL）：
  - `splitter.opaqueResize() == false`
  - 拖拽期间子 pane 的 `width()` 不变（直到 mouseRelease）
  - 视觉上只看到一条参考虚线（Qt 内置 rubber band）随手柄移动
- **T15** 拖拽态 lazy rebuild（INV-SNAP-LAZY-PANE-REBUILD）：
  - mouse press 后 `splitter->property("smSnapDragging") == true`
  - 拖拽过程中 `PreviewLayout::buildFromAst` 不被调用（用 PerfProbe 或测试 spy 验证）
  - mouse release 后 `splitter->property("smSnapDragging") == false`
  - mouse release 后 `dragFinished()` 信号被 emit 一次（不是 splitterMoved 那种连续多次）
  - 松手后预览/编辑器内容与新宽度匹配（不残留旧布局）

## 8. 非目标

- 不支持用户自定义吸附点位置 / 阈值（V1 硬编码）
- 不支持关闭吸附（可作为 V2 菜单项）
- 不在主 splitter（content|toc）应用吸附
