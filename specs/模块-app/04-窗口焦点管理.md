# 窗口焦点管理

## 1. 目的

保证用户在浏览器、文件管理器或命令行触发打开 `.md` 文件时，SimpleMarkdown 主窗口能被可靠地**提升到前台**并获得焦点。解决"应用加载成功但窗口留在后台用户看不到"的问题。

## 2. 输入 / 输出

- **输入**：
  - 命令行参数指定的文件路径
  - 单实例模式下，第二个进程转发过来的文件路径
  - 用户通过菜单/快捷键的"打开文件"操作
- **输出**：
  - 主窗口显示在最前
  - 键盘焦点在编辑器中

## 3. 行为契约

### [INV-1] 打开文件后必须三步提升

文件加载完毕后必须依次调用以下三个方法，任何一个缺失都可能在某些平台/窗口管理器下失败：

```cpp
setWindowState((windowState() & ~Qt::WindowMinimized) | Qt::WindowActive);
raise();
activateWindow();
```

### [INV-2] 两种场景都必须提升

- **打开新文件**：加载完成后提升
- **切换到已打开 Tab**：用户双击同一个文件时，激活对应 Tab 后也要提升窗口

### [INV-3] 单实例转发场景

第二个进程把路径转发给已有进程后，已有进程必须：
1. 打开或切换到对应 Tab
2. 执行 INV-1 的三步提升

### [INV-4] 不干扰用户正在使用的其他应用

- 只在用户主动触发打开文件时提升
- 后台定时任务（如会话自动保存）不得调用 `raise()` / `activateWindow()`

### [INV-5] Tab 右键菜单禁用状态反映可操作性

Tab 右键菜单的批量关闭项必须按当前上下文禁用掉无实际可操作对象的项，
避免出现「菜单可点但点了什么也不发生」的 UI 与行为不一致。具体规则：

| 菜单项 | 禁用条件（setEnabled(false)） |
|--------|-------------------------------|
| Close Others | `m_tabs.size() <= 1`（没有其他 tab 可关） |
| Close to the Left | `idx == 0`（右键位置已是最左 tab，左侧无 tab） |
| Close to the Right | `idx == m_tabs.size() - 1`（右键位置已是最右 tab，右侧无 tab） |
| Close | 始终可点（只要是合法的 tab 索引） |

实现用 `QMenu::addAction` 的返回值 `QAction*` 上接 `setEnabled(bool)`，
与既有 "Open Containing Folder" 对空路径禁用的模式一致。

### [INV-6] 「新建」按钮独立于 Tab 滚动，始终可见可点

当打开的 Tab 数量过多、Tab 总宽超过 Tab 栏可视宽度时，Qt 原生 `QTabBar` 会启用横向滚动
（左右箭头），末尾的 Tab 会被滚出可视区。「新建」(+) 按钮**不得**绑定在「最后一个 Tab 的
右侧」这一会随滚动移动的锚点上，否则按钮会被一起推出可视区，导致用户无法新建文档（只能先
关闭部分 Tab），这是 2026-06-16 用户报告的实际故障。

约束：

- 「+」按钮是 Tab 栏右侧的**固定兄弟控件**（独立于可滚动的 Tab 区），无论 Tab 多少都常驻可见。
- Tab 区自身保留原生横向滚动，用户仍可通过滚动 / 侧边 Tab 模式访问全部 Tab。
- 「+」按钮的几何不得与 Tab 区重叠：按钮左边界 ≥ Tab 区右边界。

实现：`TabBarWithAdd` 由「`QTabBar` 子类 + 自绘按钮（按钮几何 = 最后一个 Tab 右侧）」改为
「`QWidget` 容器 = `QTabBar`（stretch=1）+ `QToolButton`（固定宽）」，水平布局天然保证按钮恒在
右端、Tab 区滚动不影响按钮位置。容器对外暴露 `bar()` 访问内部 `QTabBar`，可见性切换（隐藏顶部
Tab 栏、演示模式、侧边模式）一律作用于容器而非内部 `QTabBar`，避免按钮孤立残留。

## 4. 接口

```cpp
class MainWindow : public QMainWindow {
public:
    // 打开文件（主入口）
    void openFile(const QString& path);

private:
    // 提升窗口到前台并获得焦点
    void raiseToFront();
};

void MainWindow::raiseToFront() {
    setWindowState((windowState() & ~Qt::WindowMinimized) | Qt::WindowActive);
    raise();
    activateWindow();
}

void MainWindow::openFile(const QString& path) {
    // ... 加载或切换 Tab ...
    raiseToFront();
}
```

## 5. 核心算法

### 三步提升的作用

| 方法 | 作用 | 不做会出现 |
|------|------|-----------|
| `setWindowState(... & ~Minimized | Active)` | 取消最小化，标记活跃 | 窗口如果处于最小化状态，后续 raise/activate 无效 |
| `raise()` | 在 Z 序中提升到最前 | 窗口留在其他窗口后面 |
| `activateWindow()` | 给予键盘焦点 | 窗口在前但按键输入到别处 |

三步顺序不能换：先恢复显示状态，再改 Z 序，最后拿焦点。

### Windows 特殊情况

在 Windows 上，前台窗口切换有限制（防止应用偷焦点）。如果 `activateWindow()` 仍然失败，可以配合：
- `SetForegroundWindow()`（需要 AllowSetForegroundWindow 授权）
- 或者先 `showMinimized()` 再 `showNormal()` 触发系统默认的焦点切换

当前实现依赖 Qt 的默认行为，如果未来发现有问题再加 Windows 特殊处理。

## 6. 性能预算

- 三步提升 < 10 ms

## 7. 验收条件

- **[T-1]** 从浏览器点击本地 `.md` 链接，应用启动后窗口置于前台
- **[T-2]** 应用最小化时双击 `.md` 文件，窗口从最小化恢复到前台
- **[T-3]** 应用被其他窗口遮挡时打开文件，窗口升到最前
- **[T-4]** 已有 Tab 打开同一文件时，窗口激活到对应 Tab 并前台
- **[T-5]** 单实例转发：第二个进程退出前，第一个进程已显示在前
- **[T-6]** 非用户触发的事件（自动保存、定时器）不提升窗口
- **[T-7]** 只有 1 个 tab 时右键 → "Close Others" 为禁用状态（灰色不可点）
- **[T-8]** 在最左 tab (idx==0) 上右键 → "Close to the Left" 为禁用状态
- **[T-9]** 在最右 tab (idx==m_tabs.size()-1) 上右键 → "Close to the Right" 为禁用状态
- **[T-10]** 任何合法 tab 索引下 "Close" 菜单项始终可点
- **[T-11]** 打开足够多 Tab 使 Tab 总宽溢出 Tab 栏后，「新建」(+) 按钮仍然可见（INV-6）
- **[T-12]** Tab 溢出时「新建」按钮的左边界不小于 Tab 区的右边界（按钮与 Tab 区不重叠，INV-6）

## 8. 已知陷阱

### 8.1 只调用 raise()

只 `raise()` 不 `activateWindow()`：窗口在最前但键盘焦点不在，用户打字进入别处。

### 8.2 只调用 activateWindow()

只 `activateWindow()` 不 `raise()`：某些窗口管理器下窗口仍留在 Z 序后面。

### 8.3 忘记处理最小化

最小化窗口直接 `raise()` + `activateWindow()` 无效，必须先 `setWindowState` 清 `Minimized`。

### 8.4 新文件和已打开文件分开处理

早期代码只在"新文件"路径调用三步提升，"已打开文件切换 Tab"路径没调用，导致用户双击同一文件不前置。必须两个路径都调用。
