# SimpleMarkdown 产品愿景

## 1. 目的

定义项目的**产品定位、目标用户、核心价值、非目标**，作为所有功能取舍的最高判断依据。

当某个功能提案出现争议时，回到本文档校准：该功能是否服务于定位？如果不是，拒绝。

## 2. 输入 / 输出

- **输入**：用户的 Markdown 文件（`.md` / `.markdown` / `.txt`）
- **输出**：可视化的 Markdown 预览 + 可编辑的源码视图

## 3. 行为契约（产品级不变量）

- **[INV-1] 轻量**：冷启动 < 0.5 秒，空载内存 < 30 MB
- **[INV-2] 响应**：打字延迟 < 3 ms，预览更新 < 80 ms
- **[INV-3] 跨平台**：Windows 10/11 与主流 Linux 发行版功能等价
- **[INV-4] 自包含**：不依赖外部浏览器/WebEngine，不走 HTML 渲染
- **[INV-5] 离线优先**：所有核心功能在无网络环境下可用
- **[INV-6] 用户数据安全**：禁止上传、遥测、后台联网
- **[INV-7] 保守功能**：只实现 Markdown 阅读/编辑的核心场景，不做云同步、协作、插件系统

## 4. 接口（面向用户的能力清单）

### 编辑
- 分屏模式（左源码 + 右预览）
- Markdown 语法高亮
- 搜索替换（Ctrl+F / Ctrl+H）
- 撤销/重做
- IME 中文输入
- Markdown 格式化快捷键

### 预览
- 支持 GFM 元素：标题、段落、粗体、斜体、行内代码、围栏代码块、引用、列表、表格、图片（含 SVG 矢量图）、链接、分割线、删除线
- 30ms 防抖自动更新
- 编辑-预览双向滚动同步

### 多 Tab
- 同时打开多个文件
- Tab 标题显示修改标记（`*`）
- 关闭未保存 Tab 时提示

### 文件操作
- 新建 / 打开 / 保存 / 另存为
- 拖拽 `.md` / `.markdown` / `.txt` 到窗口
- 最近文件列表（最多 10 个）
- 外部修改检测，弹窗提示重载

### 视觉
- 亮色 / 暗色主题
- 主题自动跟随系统
- TOC 面板（右侧）
- 内容标记（荧光笔效果）

### 会话
- 单实例模式
- 退出前确认未保存
- 会话状态实时持久化（进程被杀后可恢复）

## 6. 性能预算

| 指标 | 目标值 |
|------|--------|
| 冷启动 | < 500 ms |
| 空载内存 | < 30 MB |
| 打字延迟 | < 3 ms |
| 预览更新 | < 80 ms（含 30 ms 防抖） |
| 切换 Tab | < 50 ms |
| 打开 10k 行文件 | < 50 ms |

## 7. 验收条件

- [T-1] Windows 和 Linux 平台功能等价，差异项在文档中明确标注
- [T-2] 所有性能指标在中等配置机器上可复现（i5 + 16GB）
- [T-3] 监控启动/运行阶段无外部网络请求（可用 Wireshark/抓包验证）
- [T-4] 拔网线后所有核心功能正常

## 8. 已知陷阱 / 非目标

### 明确的非目标（不做）

- ❌ **云同步**：不引入账号体系
- ❌ **实时协作**：不做多用户编辑
- ❌ **插件系统**：保持单体架构，避免复杂性
- ❌ **富文本编辑**：只做 Markdown 源码编辑，不做 WYSIWYG
- ❌ **WebEngine 预览**：坚持 QPainter 自绘，避免 Chromium 开销
- ❌ **远程图片**：MVP 阶段只支持本地图片（远程图片可能在后续评估）
- ❌ **LaTeX / Mermaid**：优先保证核心 Markdown 性能

### 设计取舍

- **为什么自绘而不用 QTextEdit？** QTextEdit 的打字延迟和大文件表现不满足 INV-2
- **为什么不做 Electron？** 违反 INV-1 和 INV-4
- **为什么只支持 GFM？** CommonMark 子集难以覆盖用户期望，GFM 是事实标准
