# 字体系统

## 1. 目的

统一编辑区和预览区的字体规则，消除"两边默认字号不一致"这一类反复出现的回归 bug。确保无论用户是否调整过字号、在哪个操作路径下启动应用，编辑区正文和预览区正文的基础字号都相等。

## 2. 输入 / 输出

- **输入**：用户字号设置（delta，相对于默认值）、平台默认字体
- **输出**：所有 widget 的 `QFont`，保证编辑区与预览区基础字号一致

## 3. 行为契约

### [INV-1] 默认基础字号必须是唯一常量

全项目只允许**一个**地方定义默认基础字号的数值（例如 `kDefaultBaseFontSizePt = 12`），所有其他地方必须引用这个常量。

```cpp
// src/core/FontDefaults.h（建议位置）
namespace font_defaults {
    constexpr int kDefaultBaseFontSizePt = 12;
    constexpr const char* kEditorFontFamily = "Courier New";
    constexpr const char* kPreviewFontFamily = "Segoe UI";
    constexpr const char* kMonoFontFamily = "Consolas";
    constexpr int kMonoFontSizePt = 12;  // 代码块/内联代码的等宽字号（2026-04-14 起 = base）
}
```

禁止在源码里出现字面量 `12`、`11`、`10` 等字号数字（小于 10 的小字号如按钮次要文字除外，必须加注释说明原因）。

### [INV-2] 编辑器和预览的视觉度量必须对齐

> **修订一（2026-04-13）**：原 INV 要求 `pointSize` 严格相等——废弃。
> **修订二（2026-05-06）**：单一 xHeight 不足；纳入 capH/height 综合 score。
> **修订三（2026-05-07）**：height 容忍 ≤ 2 → ≤ 3（CI 字体度量分散）。
> **修订四（2026-05-11）**：**新增 avgCharWidth 主导项**——前三轮算法选 editor=15pt
> 让 capH/height 接近，但 avgCharWidth 差 4px（Courier New 字符比 Segoe UI 宽 36%），
> 用户连续 4 轮反馈"字号还是不一致"。**真正的视觉一致 = 水平字符密度 + 垂直度量综合**。

冷启动 + 任意 `m_fontSizeDelta` ∈ {-4, -2, 0, +2, +4} 下，编辑器与预览的字体度量**全部**满足：

```
| editor.xHeight        − preview.xHeight        | ≤ 0.5 px    （same family 下应严格 = 0）
| editor.capHeight      − preview.capHeight      | ≤ 0.5 px    （same family 下应严格 = 0）
| editor.height         − preview.height         | ≤ 0.5 px    （same family 下应严格 = 0）
| editor.avgCharWidth   − preview.avgCharWidth   | ≤ 0.5 px    （same family 下应严格 = 0）
```

**修订五（2026-05-11）阈值收紧**：方向 A1 把 family 改成相同后，same family + same pt
天然让四项度量完全相等。阈值从修订四的 1/3/8/2 收紧到统一 0.5——保护未来不再被
"分等宽/比例"的设计假设倒退回去。若某天某测试 runner 上同 family 同 pt 出现微小
浮点差（理论上不应发生），0.5 留出余量。

**修订四阈值放宽理由**：等宽 Courier New vs 比例 Segoe UI 的 capH/height 系数天生
差异——强行追求 capH ≤ 1 会让 editor.pt 推得远大于 preview.pt（如 delta=0 旧算法
选 15pt），导致 avgChar 差 4px → 用户感觉"编辑器字号明显大"。新阈值容纳字体族
先天垂直差异，把 avgCharWidth 作为水平视觉密度的硬约束顶上。

`pointSize` 之间允许 ±3pt 的差异（由 `font_defaults::balanceEditorFontSize()` 综合度量计算）。

预览侧 `pointSize` = `kDefaultBaseFontSizePt + delta`（基准固定）；编辑器侧由
`balanceEditorFontSize` 在 [previewPt-3, previewPt+3] 范围内按综合 score 选择最优。

这是**硬性不变量**，在 `tests/app/FontConsistencyTest.cpp` 中用三项度量分别断言保护。

### [INV-3] 构造函数默认值必须与中心化设置一致

所有控件（`EditorWidget`、`EditorLayout`、`PreviewLayout`）的构造函数设置的初始字号**必须**等于 `kDefaultBaseFontSizePt`。构造函数里不得出现与中心化常量不一致的硬编码数字。

**理由**：`MainWindow` 不一定在每种启动路径下都能在第一帧绘制前触发 `applyFontSize()`。构造函数的默认值可能成为首帧实际使用的值，必须和中心化设置等价。

### [INV-4] applyFontSize 必须无条件触发初始化

`MainWindow` 创建 Tab 后必须**无条件**调用 `applyFontSize()`，不得加 `if (fontSizeDelta != 0)` 这类跳过条件。原因：即使 `delta == 0`，该函数仍然是所有字号的最终同步点，跳过会让两边走各自构造函数里的默认值。

```cpp
// ✗ 错误
if (m_fontSizeDelta != 0)
    applyFontSize();

// ✓ 正确
applyFontSize();
```

### [INV-5] 字号调整（zoom in/out）必须同时作用于两边

`applyFontSize()` 必须在同一次调用内同时更新编辑器和预览的字号，不允许只更新一边。用户调整字号后 `pointSize(editor) == pointSize(preview)` 继续成立。

### [INV-6] 编辑器与预览使用同一字体族（修订五，2026-05-11，方向 A1）

> **历史修订一（2026-04-13）**：见 INV-2 修订说明。
> **修订五（2026-05-11，方向 A1）**：彻底改设计——编辑器**不再**使用等宽字体。
> 编辑器和预览的正文**必须**使用同一字体族 `kEditorFontFamily == kPreviewFontFamily ==
> "Segoe UI"`，且 `pointSize` 相等（由 `balanceEditorFontSize` 选出，same family 下
> 自然选 same pt）。

**为何不再分等宽/比例**：用户连续 5 轮反馈"编辑器和预览字号不一致"——根因是字体族
（Courier New 等宽 vs Segoe UI 比例）的**先天度量差异**（capH 矮 18%、height 矮 21%、
avgChar 宽 36%）。pt 维度的所有补偿（同 pt / +3pt / score 加权 4 项）都只能在
"水平差异"和"垂直差异"之间二选一，无法消除。唯一根治是 same family。

**为何 Segoe UI 不是等宽**：
- SimpleMarkdown 是 Markdown 编辑器不是 IDE——主要写中英文混排正文，不是代码
- 中文 fallback 到 Microsoft YaHei（比例），编辑器内部本就不等宽
- 代码块和内联代码继续用 `kMonoFontFamily = "Consolas"` 等宽渲染（INV-7 / INV-9）
- 与现代 Markdown 编辑器（Typora / Obsidian / Notion）惯例对齐

`pointSize` 相等（不再有 ±3pt 容差，由 `balanceEditorFontSize` 在 [pt-3, pt+3]
范围选最优 score → same family 下 same pt 必然胜出）。

### [INV-7] 代码块内联字号的独立性

代码块和内联代码使用等宽字体（`kMonoFontSizePt`），可以与基础字号不同。但该差值必须是项目决定的稳定比例（当前 `kMonoDelta = 0`，即与基础字号相等；历史上曾为 `-3`），不得按启动路径随机变化。

**2026-04-14 修订**：`kMonoDelta` 由 `-3` 调整为 `0`，理由——预览代码块应与编辑器中代码视觉等价（用户在编辑器里写 cpp 代码，预览渲染时不应突然缩小一号），避免"预览代码字小得读不动"的体感。

### [INV-8] 预览区所有元素必须随 applyFontSize 缩放

预览区的任意元素（代码块、内联代码、表格单元格、标题、列表序号/项目符号、图片占位符及其装饰、错误提示标签、文件名标签、图标字符等），其 `QFont` 的 `pointSize` 必须由 `m_baseFont` 或 `m_monoFont` 直接/间接派生，**禁止**出现形如 `QFont("<family>", N)` 的硬编码字面量。

```cpp
// ✗ 错误：硬编码字号，Ctrl+滚轮缩放无效
QFont monoFont("Consolas", 9);
QFont cellFont("Segoe UI", 12);
QFont iconFont("Segoe UI", 16);

// ✓ 正确：从 baseFont / monoFont 派生
QFont cellFont = m_baseFont;          // 表格单元格 == 正文字号
QFont monoFont = m_monoFont;          // 代码块 == 等宽字号
QFont iconFont = m_baseFont;
iconFont.setPointSizeF(m_baseFont.pointSizeF() * 1.33); // 按比例缩放
```

等价实现是把 `PreviewLayout::m_baseFont` / `m_monoFont` 通过 getter 暴露给 `PreviewPainter`，或者把绘制阶段每个元素需要的 `QFont` 预先写入 `LayoutBlock` 的字段（例如 `cell.font`、`block.monoFont`），让 `PreviewPainter` 只读取不构造。

**强制规则：**`src/preview/PreviewPainter.cpp` 中不得出现任何以字面量数字初始化的 `QFont` 构造。所有字号必须来自布局阶段传入的字体，或来自 `font_defaults::defaultXxxFont(m_fontSizeDelta)` 派生。

### [INV-9] 等宽字号与基础字号的联动

`PreviewLayout::m_monoFont` 和 `font_defaults::defaultMonoFont(...)` 返回的字体，其 `pointSize` 必须相对 `m_baseFont`（或 `kDefaultBaseFontSizePt + sizeDelta`）动态派生，而不是一个固定的独立常量。

形式化：
```
monoSize(delta) = kDefaultBaseFontSizePt + delta + kMonoDelta
其中 kMonoDelta = kMonoFontSizePt - kDefaultBaseFontSizePt  // 2026-04-14 起为 0
```

实现要求：

1. `font_defaults::defaultMonoFont` 必须接受 `int sizeDelta = 0` 参数：
   ```cpp
   inline QFont defaultMonoFont(int sizeDelta = 0) {
       QFont font(kMonoFontFamily, kDefaultBaseFontSizePt + sizeDelta + kMonoDelta);
       font.setStyleHint(QFont::Monospace);
       return font;
   }
   ```
2. `PreviewLayout::setFont(const QFont& base)` 必须同时更新 `m_monoFont`：
   ```cpp
   void PreviewLayout::setFont(const QFont& baseFont) {
       m_baseFont = baseFont;
       m_monoFont = font_defaults::defaultMonoFont(
           baseFont.pointSize() - kDefaultBaseFontSizePt);
       // ... 其他度量刷新
   }
   ```
3. `MainWindow::applyFontSize()` 传递给每个 tab 的 previewLayout 的字体必须能让 `m_monoFont` 派生出正确字号（即 `setFont` 必须正确处理）。

缩放后 `(m_monoFont.pointSize() - m_baseFont.pointSize()) == kMonoDelta` 必须恒成立。

### [INV-10] 编辑器字号必须由 balanceEditorFontSize() 动态补偿

> **新增（2026-04-13）**：与 INV-2 / INV-6 配套。

`MainWindow::applyFontSize()` 必须使用 `font_defaults::balanceEditorFontSize(editorFont, previewFont, device)` 计算编辑器最终字号，**禁止**直接用 `defaultEditorFont(delta)` 的返回值赋给编辑器（那样视觉会不齐）。

```cpp
// ✓ 正确
QFont previewFont = font_defaults::defaultPreviewFont(m_fontSizeDelta);
QFont editorFont = font_defaults::defaultEditorFont(m_fontSizeDelta);
editorFont = font_defaults::balanceEditorFontSize(editorFont, previewFont, viewport);
tab.editor->setFont(editorFont);
tab.preview->setFont(previewFont);

// ✗ 错误
tab.editor->setFont(font_defaults::defaultEditorFont(m_fontSizeDelta));  // 不做补偿，视觉偏大
```

`balanceEditorFontSize` 的实现要求（2026-05-11 修订四）：
1. 在 `[previewFont.pointSize() - 3, previewFont.pointSize() + 3]` 窗口内搜索 editor 候选（共 7 个）
2. 对每个候选用 `QFontMetricsF(font, device)` 取 xHeight / capHeight / height / avgCharWidth 四项度量
3. **第一阶段：INV-2 硬约束 filter**——保留满足以下四项的候选：
   ```
   |editor.xHeight      − preview.xHeight     | ≤ 1
   |editor.capHeight    − preview.capHeight   | ≤ 3
   |editor.height       − preview.height      | ≤ 8
   |editor.avgCharWidth − preview.avgCharWidth| ≤ 2     ← 修订四新增
   ```
4. 计算综合 score（仅在 filter 子集；为空时退化到全集）：
   ```
   score = 2.0 * |editor.avgCharWidth − preview.avgCharWidth|   # 主导项
         + 1.0 * |editor.xHeight       − preview.xHeight|
         + 1.0 * |editor.capHeight     − preview.capHeight|
         + 0.3 * |editor.height        − preview.height|
   ```
5. 选 score 最小的 pointSize 返回
6. 若 device 为 null 退化到原 editor 字号（无补偿）——保证测试可用

> **修订四（2026-05-11）**：把 avgCharWidth 加入硬约束 filter + score 主导项。
> 用户连续 4 轮报告"字号不一致"——前三轮算法（仅 xH/capH/height）在 delta=0 选
> 15pt 让 capH/height 完美但 avgChar 差 4px。实测：12pt（same pt）xH=0 capH=2.58
> h=6 avg=1——综合最接近用户感知。新算法选 12pt（same pt 类）作为最优。

**权重选择理由（修订四）**：
- `avgCharWidth` ×2.0：水平字符密度——人眼对文字排列的"字号"判断主要看行宽 / 字符
  间距，这是中文用户在 Markdown 编辑场景的主导视觉因素
- `capHeight` ×1.0：大写字母高度——降权（不再主导，因为字体族先天差异）
- `xHeight` ×1.0：经典指标
- `height` ×0.3：含 leading 影响行间距，最低权

**禁止**硬编码任何经验补偿值（如历史上的 `setPointSize(11)`）。补偿必须基于度量计算。

## 4. 接口

```cpp
// src/core/FontDefaults.h
namespace font_defaults {
    constexpr int kDefaultBaseFontSizePt = 12;
    constexpr int kMonoFontSizePt = 9;                    // 等宽默认字号（等价于 base + kMonoDelta）
    constexpr int kMonoDelta = kMonoFontSizePt - kDefaultBaseFontSizePt;  // = -3
    constexpr const char* kEditorFontFamily = "Courier New";
    constexpr const char* kPreviewFontFamily = "Segoe UI";
    constexpr const char* kMonoFontFamily = "Consolas";

    // 构造编辑器默认字体
    QFont defaultEditorFont(int sizeDelta = 0);
    // 构造预览默认字体
    QFont defaultPreviewFont(int sizeDelta = 0);
    // 构造等宽字体（代码块用）— 必须随 sizeDelta 缩放
    QFont defaultMonoFont(int sizeDelta = 0);
}

// MainWindow
class MainWindow {
public:
    // 根据当前 m_fontSizeDelta 应用字体到所有 Tab
    // 无条件调用，不受 delta 值影响
    void applyFontSize();

private:
    int m_fontSizeDelta = 0;  // 相对默认字号的增量
};
```

## 5. 核心算法

### 字号传播链路

```
冷启动
  │
  ▼
main() → MainWindow ctor
  │
  ├─ 创建菜单/工具栏
  ├─ loadSession() → 读取 m_fontSizeDelta
  ├─ 恢复/新建 Tab（EditorWidget + PreviewWidget）
  │    │
  │    ├─ EditorWidget ctor
  │    │    └─ m_layout->setFont(defaultEditorFont())  ← 使用常量
  │    └─ PreviewWidget ctor
  │         └─ m_layout->setFont(defaultPreviewFont()) ← 使用常量
  │
  └─ applyFontSize()  ← 无条件调用，统一校准
       │
       ├─ QFont editorFont = defaultEditorFont(m_fontSizeDelta)
       ├─ QFont previewFont = defaultPreviewFont(m_fontSizeDelta)
       └─ 遍历所有 Tab 调用 setFont
```

### 为什么需要双重保证（构造函数 + applyFontSize）？

既然 `applyFontSize()` 会覆盖构造函数的值，为什么还要求构造函数也用常量？

- `EditorWidget` 可能在独立场景下使用（如单元测试、未来的嵌入场景），不一定走 `MainWindow` 路径
- `applyFontSize()` 到达前可能有 paintEvent 先发生
- 防御式编程：两层保障比单层可靠

## 6. 性能预算

- `applyFontSize()` 全量刷新 < 50 ms（含所有 Tab 重排版）

## 7. 验收条件

- **[T-1]** 冷启动后，`editor.font().pointSize() == preview.baseFont().pointSize() == kDefaultBaseFontSizePt`
- **[T-2]** 未设置 `m_fontSizeDelta`（初次启动）时，两边字号仍然一致
- **[T-3]** 用户缩放字号后两边同步变化，始终相等
- **[T-4]** `grep -n "setPointSize\|QFont.*,.*[0-9]" src/` 除 `FontDefaults.h` 外不得出现字号字面量
- **[T-5]** `grep -n "if.*fontSizeDelta.*!= *0.*applyFontSize" src/` 无匹配（确认条件应用被移除）
- **[T-6]** `tests/app/FontConsistencyTest.cpp` 断言 INV-1/INV-2/INV-3 成立
- **[T-7]** 反复 Ctrl+滚轮向上/向下后，预览区以下元素的字号必须全部按 delta 同步变化：
  1. 段落正文
  2. 标题 H1–H6（按既有 scale 派生）
  3. 内联代码（反引号）背景色区域宽高同步
  4. 围栏代码块（```…```）文本和行高
  5. 表格单元格文本
  6. 列表的序号/圆点
  7. 图片占位符的主图标（`\u25A3`）、错误图标（`\u2717`）、文件名标签、尺寸标签、"Image"/"Failed to load image" 等描述文字

  验证方式：分别在 delta=0/+4/+8/-4 下截屏，用像素测量确认同一元素字号比例与正文一致。

- **[T-8]** `grep -rn 'QFont\s*(\s*"' src/preview/` 只允许在被禁止列表之外的场合出现；在 `PreviewPainter.cpp` 中必须为 0 次匹配；在 `PreviewLayout.cpp` 中只允许出现在从 `m_baseFont` / `m_monoFont` 派生的位置（不得带字面量数字作为第二参数）。
- **[T-9]** 验证 `(previewLayout->monoFont().pointSize() - previewLayout->baseFont().pointSize()) == kMonoDelta` 在所有 `m_fontSizeDelta` 下恒成立。

## 8. 已知陷阱

### 8.1 条件初始化遗漏（本次 bug 的根因）

`MainWindow::loadSession()` 中曾有：

```cpp
if (m_fontSizeDelta != 0)
    applyFontSize();
```

默认启动时 `delta == 0`，此行不执行，两边走各自构造函数的硬编码值——`EditorWidget(11pt)` vs `PreviewLayout(12pt)` 差 1pt。修复：去掉条件，无条件调用。

### 8.2 构造函数硬编码数字

`EditorWidget::EditorWidget()` 中：

```cpp
QFont font = QFontDatabase::systemFont(QFontDatabase::FixedFont);
font.setPointSize(11);  // ← 硬编码与中心化常量不一致
```

修复：改用 `font_defaults::defaultEditorFont()` 或直接使用常量。

### 8.3 一处修复不彻底

历史修复曾把"预览 10pt → 12pt"，假设编辑器是 12pt，但没检查 `EditorWidget` 构造函数里的 11pt。单分支修复是反复回归的根源，本 Spec 的 INV-1 / INV-2 / T-1 就是为了拦住这种"单点修复"。

### 8.4 不要依赖 setStyleHint 决定字号

`QFont::setStyleHint(QFont::Monospace)` 只影响字体族回退，不影响字号。字号必须显式设置或继承。

### 8.5 代码块字号独立变化

代码块字号独立于基础字号时，要确保用户缩放后比例保持：`monoSize = baseSize + fixedDelta`，不是独立的 delta。否则缩放几次后代码块和正文字号关系错乱。

### 8.6 PreviewPainter 硬编码字号导致 Ctrl+滚轮缩放对部分元素无效（2026-04-13 发现）

本次 bug 复现步骤：打开含有代码块、内联代码、表格、图片的文档，按住 Ctrl 滚动滚轮。预期：预览区所有元素字号同步变化。实际：正文/标题变化，但代码块、表格、图片占位符等元素字号纹丝不动。

**根因分析：**

1. `PreviewPainter.cpp` 中大量元素使用 `QFont("<family>", <literal>)` 硬编码字号，完全不读取布局阶段传入的字体。
2. `PreviewLayout::setFont()` 只更新 `m_baseFont`，未重新派生 `m_monoFont`，导致即便绘制侧改为读取 `m_monoFont`，它仍是冷启动时的值。
3. `font_defaults::defaultMonoFont()` 没有 `sizeDelta` 参数，不接受缩放。

**为什么内联代码看起来也没变化：**

严格讲，`PreviewLayout::collectInlineRuns` 的 `Code` 分支（第 304-313 行）确实写了 `run.font.setPointSizeF(currentFont.pointSizeF() * 0.9)`，所以内联代码 `run.font.pointSize` 会随 `m_baseFont` 缩放。但缩放因子是 0.9 且 `m_monoFont` 的家族没变化，用户视觉上变化幅度比正文小了 10%，加上代码块完全不动，所以产生"代码全部没动"的印象。**修复方案应当把内联代码也改为基于 `m_monoFont` 的 delta 派生，而不是直接 `* 0.9`**，以保证与代码块内字号规则一致。

**禁止方案（不可接受的"修复"）：**

- 仅把 `9` 改成 `9 + m_fontSizeDelta`：`PreviewPainter` 不应该感知 `MainWindow` 的 delta。字号应自上而下通过 `PreviewLayout::setFont` → `PreviewPainter`（经由字段或 getter）传播。
- 用宏 `#define PREVIEW_BASE_FONT_SIZE 12` 替换字面量：仍然是硬编码，违反 INV-1。

**可接受的修复方向（二选一或并用）：**

- **方案 A**：让 `PreviewLayout` 公开 `baseFont()` / `monoFont()` getter，`PreviewPainter` 通过 `ctx` 或 layout 指针读取并派生出装饰字体（按比例 1.33x/1.66x 等）。
- **方案 B**：把绘制时需要的 `QFont` 预先写入 `LayoutBlock` 或 `RenderContext` 字段，绘制阶段只使用，不构造。表格单元格和代码块最适合走此方案——布局阶段已持有正确字体，把它写入 block 即可。

### 8.7 defaultMonoFont 缺 sizeDelta 参数

这是一个"半修半漏"的历史遗留：`defaultEditorFont` / `defaultPreviewFont` 都有 `sizeDelta`，但 `defaultMonoFont` 忘记加。只要有任何一个调用方调 `defaultMonoFont()` 就会得到冷启动值——本次 `PreviewLayout` 构造函数就是这么中招的。
