# 预览布局引擎（PreviewLayout）

## 1. 目的

把不可变的 `MarkdownAst` 转换成可绘制的 `LayoutBlock` 树，计算每个块的绝对位置、大小、字体度量。**PreviewLayout 是高 DPI bug 的高发区**，所有布局阶段的字体度量必须严格遵守 `横切关注点/40-高DPI适配.md`。

## 2. 输入 / 输出

- **输入**：
  - `std::shared_ptr<const MarkdownAst>` — 解析产生的不可变 AST
  - `int width` — 可视区宽度（像素）
  - `QPaintDevice* device` — 用于计算字体度量
  - `const Theme& theme` — 主题（影响颜色，不影响布局）
- **输出**：
  - `LayoutBlock root` — 布局树根节点，含所有子块的相对 bounds 和总高度

## 3. 行为契约

### [INV-1] 必须遵守高 DPI 规则
遵守 `横切关注点/40-高DPI适配.md` 的所有 INV（INV-1 到 INV-6）。

### [INV-2] 必须遵守坐标系统规则
遵守 `横切关注点/20-坐标系统.md` 的所有 INV（INV-1 到 INV-4）。

### [INV-3] 构造函数不得计算 DPI 度量
构造函数只设临时常量：
```cpp
PreviewLayout::PreviewLayout() : m_lineHeight(20.0), m_codeLineHeight(20.0) {}
```

### [INV-4] updateMetrics 必须先于 layout
调用 `layout()` 前必须先 `updateMetrics(device)`，否则使用的是临时值。

### [INV-5] 布局结果以相对坐标存储
`LayoutBlock::bounds` 是相对于**父块**的坐标，不是绝对坐标。绝对坐标在绘制阶段由 `PreviewPainter` 递归计算。

### [INV-6] estimateParagraphHeight 不得二次调整 m_lineHeight
估算函数只能读 `m_lineHeight`，不得根据 run 的字体再算一次。

### [INV-7] 容器块子块 X 偏移
布局阶段对容器块（List / BlockQuote / Table）的子块应用 X 偏移（缩进、序号宽度）时，必须写入 `child.bounds.x()`，供绘制阶段正确还原绝对坐标。

### [INV-8] m_monoFont 必须随 m_baseFont 同步派生

遵守 `横切关注点/80-字体系统.md` 的 INV-9。具体要求：

1. 构造函数中 `m_monoFont = font_defaults::defaultMonoFont(0)`。
2. `setFont(const QFont& base)` **必须**在更新 `m_baseFont` 的同时，同步更新 `m_monoFont`：
   ```cpp
   void PreviewLayout::setFont(const QFont& baseFont) {
       m_baseFont = baseFont;
       int delta = baseFont.pointSize() - font_defaults::kDefaultBaseFontSizePt;
       m_monoFont = font_defaults::defaultMonoFont(delta);
       // m_lineHeight / m_codeLineHeight 会在下一次 updateMetrics 中重算
   }
   ```
3. 任何读取 `m_monoFont` 的地方（`layoutBlock` 的 CodeBlock/HtmlBlock 分支、`collectInlineRuns` 的 Code 分支、以及未来的 `monoFont()` getter）都必须拿到与当前 `m_baseFont` 相差 `kMonoDelta` 的字体。

断言：`m_monoFont.pointSize() - m_baseFont.pointSize() == font_defaults::kMonoDelta`。

### [INV-9] 代码/单元格字体必须写入 LayoutBlock

为了让 `PreviewPainter` 完全摆脱硬编码字号（见 `03-绘制管线.md` INV-8），布局阶段需要把以下字段写入 `LayoutBlock` 或其子块：

- `CodeBlock` / `HtmlBlock`：在 block 上保存"代码用字体"（family + size），或让绘制侧通过 layout getter 取 `m_monoFont`。禁止绘制侧重新构造。
- `Table` / `TableCell`：单元格内的 `inlineRuns` 已经持有 font（正文用 `m_baseFont`，表头加粗）。表头加粗字段也必须基于 `m_baseFont` 派生，而非新建 `QFont("Segoe UI", 12)`。
- `List` 的序号 / 项目符号：可由绘制侧读取 layout 的 `baseFont()` getter。

### [INV-10] estimateParagraphHeight 行高必须与渲染侧 paintInlineRuns 一致

`paintInlineRuns` 使用 `block.inlineRuns[0].font` 作为整段的行高基准（行高 = `QFontMetricsF(firstRun.font, device).height() * m_lineSpacingFactor`）。布局阶段 `estimateParagraphHeight` 必须使用同一基准来估算高度，否则当 `inlineRuns[0].font` 与 `m_baseFont` 不同（典型场景：Heading 用放大字体，H1 = 1.8x、H2 = 1.5x）时，layout 的 `block.bounds.height()` 会严重低估渲染实际占用，导致下一个块绘制时压在标题上重叠。

要求：

```cpp
qreal PreviewLayout::estimateParagraphHeight(const std::vector<InlineRun>& runs, qreal maxWidth) const {
    if (runs.empty()) return m_lineHeight;
    const QFontMetricsF& firstRunFm = cachedFontMetrics(runs[0].font);
    qreal lineHeight = firstRunFm.height() * m_lineSpacingFactor;  // ← 与 paintInlineRuns 同构（INV-13）
    // ...
}
```

不可使用 `m_lineHeight`（基于 `m_baseFont`），因为 Heading 的 inlineRuns 持有的是放大后的 headingFont。乘数从 `m_lineSpacingFactor` 而非字面量来读，确保用户切换"视图 → 行间距"时 layout 与渲染同步变化。

### [INV-11] estimateParagraphHeight 末行不得多附 0.5 倍墨迹空白

`paintInlineRuns` 的实际占用：每换一行 `curY += lineHeight`，最后一行**画完即停**，不再额外推进 `curY`。因此 N 行段落的实际墨迹底部 = `(N-1) * lineHeight + glyphHeight`，**不是** `N * lineHeight`。

布局阶段必须使用同一公式，否则段落底部会多留 `0.5 * glyphHeight` 的空白。**最直观的表现**：连续短列表项（如 `- **DrawBoard**：...`）项间会出现接近一整行的多余空白（INV-10 修标题时未触及此处）。

要求：

```cpp
return (totalLines - 1) * lineHeight + firstRunFm.height();
// 不允许 return totalLines * lineHeight;
```

**ListItem 的 height 下界保底**同步使用 `baseFm.height()`（墨迹高度），不得用 `m_lineHeight`（= `baseFm.height() * 1.5`），否则单行 list item 仍会被抬高至 1.5x。

### [INV-16] 视口剪裁模式（A1 plan 2026-05-12）

`PreviewLayout` 提供 `setViewportYRange(topY, bottomY)` 开关：

- **未设置**（`hasViewportYRange() == false`）：`buildFromAst` 走全量精算路径，所有块都
  调用 `layoutBlock` 完整 layout。**默认行为**，向后兼容。所有现有单元测试不传
  viewport range，仍按原方式运行。
- **已设置**：`buildFromAst` 在迭代根 children 时：
  1. 调用 `quickEstimateHeight(child, maxWidth)` 拿粗估高度
  2. 判定块的 `[yCursor, yCursor + estimatedH]` 是否与 `[topY, bottomY]` 相交
  3. 相交 → `layoutBlock` 完整精算（含 inlineRuns / children / 字体度量）
  4. 不相交 → `makePlaceholder` 创建占位块（`placeholderOnly = true`，
     `inlineRuns/children` 空，`bounds` 用粗估高度）

PreviewWidget 在每次 `buildFromAst` 调用前通过 `applyLayoutViewportCrop()` 注入
当前滚动位置 ± 2 屏 buffer，让真实场景默认走剪裁路径。

### [INV-17] placeholder 高度允许粗估（INV-15 例外）

`quickEstimateHeight` 用 `(endLine - startLine + 1) × m_lineHeight` 基本公式 +
type 特殊修正（Heading 倍数 / CodeBlock × m_codeLineHeight + padding / Image 200px
/ ThematicBreak 20px / Frontmatter entries 数 × lineHeight）。

不模拟 INV-15「字符级软换行」算法——在 placeholder 路径允许 ±30% 高度偏差。
本 INV 是 INV-15 的**显式例外**：精算路径（layoutBlock）仍必须按字符级一致。

副作用：滚动条 `maximum` 会随 placeholder 块进入视口升级为精算而**轻微漂移**——
这是经典 virtualized scrolling trade-off，用户可能感觉到内容"跳一下"，但比
卡顿 100ms+ 可接受得多。

### [INV-18] placeholder 进入视口前必须升级为精算

`PreviewWidget::scrollContentsBy` 监听滚动事件，当滚动距离超过半屏（`vpH * 0.5`）
时：

1. 调 `applyLayoutViewportCrop()` 把新视口 + ±2 屏 buffer 推给 layout
2. 调 `m_layout->buildFromAst(m_currentAst)` 触发 rebuild
3. 滚动到的区域原本是 placeholder 的块此时升级为完整 layoutBlock 精算

节流逻辑保证连续滚动不会每帧都 rebuild——视口外块只有跨入半屏内才升级。
若用户停止滚动，placeholder 块保持原状（直到下次滚动或 widget 重建）。

### [INV-19] sourceLine 映射在 placeholder 块上精度降级到块级

`sourceLineToY` / `yToSourceLine` 依赖 `collectSourceMappings` 递归收集每个块的
`sourceStartLine + bounds.y()`。placeholder 块**保留** sourceStartLine 信息，
所以映射仍工作；但**不递归 children**（children 为空），意味着 List 内 Item
级、Table 内 TableCell 级的细粒度映射在 placeholder 状态下丢失。

精度影响：
- 用户在编辑器移动光标到第 N 行 → 预览滚到包含 N 的整个 List/Table 块开头，
  而不是块内第 K 个 item 处
- placeholder y 是粗估，可能再叠加 ±30% 偏差

实际体感：用户跳转到目标块后会触发 buildFromAst rebuild（半屏阈值），该块升级为
精算 → 后续滚动精度恢复。属于"短暂偏差，最终一致"。

### [INV-PREVIEW-TOP-PAD] 文档首块上方留出字号派生的视觉间距

`PreviewLayout::buildFromAst` 中根容器的首个子块起点 `y` **不得为 0**，必须按 baseFont 度量派生
一个上方 padding（如 `fm.height() * 0.1` ≈ 12pt 下 2-3px），避免 frontmatter 卡片、Heading、
Paragraph 等首块紧贴预览区可视区域上边界的"贴线"视觉拥挤。

派生而非硬编码的目的：
- 高 DPI（125%/150%/175%）下间距随字号自然缩放，不需要单独条件
- 与 `specs/横切关注点/40-高DPI适配.md` INV-DPI-NO-HARDCODE 一致

副作用：`m_root.bounds.height()` 自动包含 padding，滚动条 `maximum` 范围正确；
`sourceLineToY` 的 mappings 也自动包含 padding 偏移（每个 child block 的 `bounds.y()` 起点 ≥ padding）。

### [INV-15] estimateParagraphHeight 换行算法必须与 paintInlineRuns 字节级一致

历史 bug（plans/归档/2026-05-06-表格单元格内容超出与下方重合.md）：
旧 estimate 用 `qCeil(segmentWidth / maxWidth)` 这种「整段总宽 ÷ 行宽 上取整」公式，
**假设**字符可以紧凑填满每一行尾部。但 `paintInlineRuns` 实际是逐字符换行——
每行尾部只要剩余空间小于下一字符宽度就立即换行，**每行末尾几乎必有取整浪费**。
窄列下（如 1/3 屏幕宽的表格 cell）单行只能放 2-3 个汉字，多 run 切换后浪费叠加，
导致 estimate 比实际行数少 30-50%；表格上体现为 `cell.bounds.height` < 真实占用，
下一行 row 起点压在上一行末尾 → 单元格内容视觉重叠。

要求：`estimateParagraphHeight` 必须模拟 `PreviewPainter::paintInlineRuns` 的逐字符换行算法
（`findFitCount` + `curX/curY` 推进逻辑）。布局端的换行决策与渲染端**必须按字符位级别完全一致**，
不允许任何"近似"或"上界估计"。

意外副作用（已实测，2026-05-06）：改用字符级模拟后，1024 行文档 `buildFromAst` P50 由 41ms
降至 16.7ms（-60%），原因是旧算法对长 run 调用 `fm.horizontalAdvance(整段长字符串)` 触发
Qt 完整 shaping pipeline，而新算法逐字符 advance 走快路径，性能反而更优。

### [INV-14] CodeBlock 高度必须按软换行后的视觉行数计算

代码块必须按可视宽度对每物理行做字符级软换行预测，块高度 = `visualLineCount * m_codeLineHeight + 16`（上下 padding 各 8）。**禁止**只用 `\n` 数量乘行高——超长行会越过代码块右边界，且后续 block 的 y 坐标会压在被裁切的代码内容上。

要求：

```cpp
// 内容可用宽度（左右各 8 px padding，与 PreviewPainter::paintCodeBlock 的 textX = drawX + 8 同构）
const qreal hPad = 8.0;
qreal contentWidth = qMax<qreal>(1.0, maxWidth - 2 * hPad);
const QFontMetricsF& fmCode = cachedFontMetrics(m_monoFont);

int visualLineCount = 0;
const QStringList rawLines = block.codeText.split('\n');
const int rawCount = rawLines.size();
for (int li = 0; li < rawCount; ++li) {
    // 末尾 \n 产生的空 trailing 行不计（与 painter 跳过逻辑同构）
    if (li == rawCount - 1 && rawLines[li].isEmpty() && block.codeText.endsWith('\n'))
        break;
    const QString& line = rawLines[li];
    if (line.isEmpty()) {
        visualLineCount += 1;
        continue;
    }
    qreal fullW = fmCode.horizontalAdvance(line);
    if (fullW <= contentWidth) {
        visualLineCount += 1;
        continue;
    }
    // 字符级 wrap 估算：与 painter 的 fit 算法同构
    int physOffset = 0;
    while (physOffset < line.length()) {
        int fit = 0;
        qreal acc = 0;
        for (int i = physOffset; i < line.length(); ++i) {
            qreal cw = fmCode.horizontalAdvance(QString(line[i]));
            if (acc + cw > contentWidth && fit > 0) break;
            acc += cw;
            fit = i + 1 - physOffset;
        }
        if (fit <= 0) fit = 1;  // 至少装一个，避免死循环
        physOffset += fit;
        visualLineCount += 1;
    }
}
qreal h = visualLineCount * m_codeLineHeight + 2 * hPad;
```

**为什么 fit > 0 才允许换行**：当 contentWidth 小于单字符宽度时（极窄视口），强制至少装一个字符防止死循环；painter 端必须用同一保底逻辑，否则两端视觉行数不一致。

历史 bug：2026-04-27 之前 layout 直接用 `lineCount = codeText.count('\n') + 1`，painter 用单次 `drawText` 整行画——超长行从右边界飞出，且后续 block 没有按"扩展到 wrap 后的高度"留位置，造成视觉裁切 + 块重叠。

### [INV-13] m_lineSpacingFactor 是正文行高乘数，由用户设置驱动

正文段落行高 = `firstRun.font.height() * m_lineSpacingFactor`。`m_lineSpacingFactor` 默认 1.5（与历史行为兼容），由 `MainWindow::onLineSpacingTriggered` 经由 `PreviewWidget::setLineSpacingFactor` 推送到 `PreviewLayout`。

要求：

1. `setLineSpacingFactor(qreal factor)` 调用后必须在 `m_device != nullptr` 时立即重算 `m_lineHeight = QFontMetricsF(m_baseFont, m_device).height() * m_lineSpacingFactor`，并 `clearFontMetricsCache()`（虽然乘数不影响 metrics 本身，但调用方通常紧接着会 rebuildLayout，缓存语义最干净）。
2. 当 `m_device == nullptr`（widget 还没首次绘制）时只更新字段，下一次 `updateMetrics(device)` 会自然把新乘数反映到 `m_lineHeight`。
3. **作用域只限正文段落**：代码块 `m_codeLineHeight = fm.height() * 1.4` 和 frontmatter 表格 `lineH = fm.height() * 1.4` 与代码语义绑定，**不**消费此乘数。
4. 渲染端 `paintInlineRuns` 必须读取 `m_layout->lineSpacingFactor()`，禁止硬编码（详见 `03-绘制管线.md` INV-12）。

**为什么默认 1.5 而不是 1.0**：这是历史行为——构造函数与 `MainWindow::loadSettings` 的 `view/lineSpacing` 默认值都是 1.5，新增字段必须保持向后兼容，避免老用户重启后看到行间距突然变小。

### [INV-12] 字体度量缓存 key 必须包含所有影响度量的字体属性

`PreviewLayout::cachedFontMetrics` 内部用 `unordered_map` 缓存 `QFontMetricsF`，避免每次 `paintInlineRuns / estimateParagraphHeight` 重复构造。**禁止使用 `qHash(QFont)` 作为缓存 key**：Qt 5.12 的 `qHash(QFont)` 实现对"同 family、同 weight、同 italic、不同 pointSize"会返回相同 hash 值（实测 SimSun bold size=16.2 与 SimSun bold size=9 都得到 key=2476512596）。一旦 H1 标题先于含粗体的列表项进入缓存，列表项 bold 段落会读到 H1 的度量（高度被错估为 1.8 倍），`estimateParagraphHeight` 把单行段落估为多行，`ListItem.height` 比墨迹高出 0.5~1 整行，相邻列表项之间出现一行多余空白。

**最直观的表现**：新建未命名 tab（默认带 H1 模板）后粘贴 `- **xxx**：...` 列表，**首次渲染**两项之间多出近一整行空白；缩放一次后消失（缩放改变 pointSize、字体 hash 不再撞车）。

要求 cache key 必须基于显式的字体属性元组：

```cpp
// (family, pointSizeF, weight, italic, stretch, styleStrategy)
char buf[256];
QByteArray familyUtf8 = font.family().toUtf8();
qsnprintf(buf, sizeof(buf), "%.*s|%.6g|%d|%d|%d|%d",
          static_cast<int>(familyUtf8.size()), familyUtf8.constData(),
          font.pointSizeF(), font.weight(), font.italic() ? 1 : 0,
          font.stretch(), static_cast<int>(font.styleStrategy()));
std::string key(buf);
```

`m_fontMetricsCache` 类型应为 `std::unordered_map<std::string, QFontMetricsF>`（Qt 5.12 不为 QString 提供 std::hash 特化）。

## 4. 接口

```cpp
class PreviewLayout {
public:
    // 构造函数只设临时值，不依赖 DPI
    PreviewLayout();

    // 设置正文字体（字号从中心化 FontDefaults 派生）
    // 必须同步更新 m_baseFont 和 m_monoFont（INV-8）
    void setFont(const QFont& baseFont);

    // 设置正文行高乘数（默认 1.5）
    // 仅作用于正文段落 / List / Table 行高，不改变代码块/frontmatter 的 1.4 基线（INV-13）
    void setLineSpacingFactor(qreal factor);
    qreal lineSpacingFactor() const;

    // 供 PreviewPainter 读取的字体 getter（INV-9）
    // 让绘制侧不再构造 QFont("Consolas", 9) / QFont("Segoe UI", 12)
    const QFont& baseFont() const;
    const QFont& monoFont() const;

    // 根据 device 的 DPI 重新计算所有度量
    // 必须在任何 layout() 调用前至少调用一次
    void updateMetrics(QPaintDevice* device);

    // 把 AST 布局成 LayoutBlock 树
    // 要求：updateMetrics 已被调用
    void layout(
        std::shared_ptr<const MarkdownAst> ast,
        int width,
        QPaintDevice* device,
        const Theme& theme
    );

    // 返回布局结果（只读）
    const LayoutBlock& root() const;

    // 返回布局总高度（像素）
    qreal totalHeight() const;

private:
    QFont m_baseFont;             // 正文字体（由 setFont 写入）
    QFont m_monoFont;             // 等宽字体（setFont 中同步派生，INV-8）
    qreal m_lineSpacingFactor;    // 正文行高乘数，默认 1.5（INV-13）
    qreal m_lineHeight;           // 普通文本行高 = baseFm.height() * m_lineSpacingFactor
    qreal m_codeLineHeight;       // 代码块行高（与 m_lineSpacingFactor 无关，INV-13）
    LayoutBlock m_root;
};
```

## 5. 核心算法

### 布局流程

```
updateMetrics(device)
  └─ m_lineHeight     = QFontMetricsF(m_baseFont, device).height() * m_lineSpacingFactor   // INV-13
  └─ m_codeLineHeight = QFontMetricsF(m_monoFont, device).height() * 1.4                   // 代码块与 lineSpacingFactor 无关

layout(ast, width, device, theme)
  └─ 清空 m_root
  └─ 递归遍历 AST 根节点：
       for node in ast.children:
           block = layoutNode(node, width, device)
           m_root.children.append(block)
           累加 m_root.height

layoutNode(node, width, device) 根据 node.type 分发：
  - Paragraph    → layoutParagraph
  - Heading      → layoutHeading
  - CodeBlock    → layoutCodeBlock
  - List         → layoutList（递归 layoutNode 处理 item）
  - BlockQuote   → layoutBlockQuote（递归）
  - Table        → layoutTable（递归）
  - ...
```

### layoutList 的子块 bounds

```cpp
qreal itemX = indentWidth;                  // 缩进到内容起点
qreal itemY = 0;
for (auto& item : list.items) {
    auto child = layoutItem(item, width - indentWidth, device);
    child.bounds = QRectF(itemX, itemY, child.width, child.height);
    //             ^^^^^  ← 布局时记录的 X 偏移，绘制时必须读回
    itemY += child.height;
}
```

## 7. 验收条件

- **[T-1]** 1x / 1.25x / 1.5x DPI 下行高比例正确
- **[T-2]** 构造后不调用 updateMetrics 直接 layout 会得到临时度量（测试保护，不允许无声失败）
- **[T-3]** 列表项的 `bounds.x()` 大于 0（证明 X 偏移被记录）
- **[T-4]** 嵌套列表（列表中的列表）的子列表 bounds 相对于父列表项
- **[T-5]** estimateParagraphHeight 返回值 == 实际布局高度（误差 < 1 px）
- **[T-6]** `grep -rn "QFontMetricsF(" src/preview/PreviewLayout.cpp` 所有调用都带 device
- **[T-7]** 在各种 `m_fontSizeDelta` 值下调用 `setFont(defaultPreviewFont(delta))` 后，`monoFont().pointSize() - baseFont().pointSize() == kMonoDelta` 恒成立
- **[T-8]** `grep -n 'QFont(.*[0-9]' src/preview/PreviewLayout.cpp` 无硬编码数字匹配
- **[T-9]** `setLineSpacingFactor(1.0)` 后 `m_lineHeight ≈ baseFm.height()`，`setLineSpacingFactor(2.0)` 后 `m_lineHeight ≈ baseFm.height() * 2`（误差 < 1 px）；变更乘数后 `estimateParagraphHeight` 与 `paintInlineRuns` 行高仍一致（layout 高度 == 渲染占用，误差 < 1 px）
- **[T-10]** CodeBlock 含一行宽度 > maxWidth 的长行时，`block.bounds.height()` ≥ `2 * m_codeLineHeight + 16`（说明 layout 预测了至少 2 个视觉行）；layout 计算的 visualLineCount 与 painter 实际绘制的视觉行数严格相等（误差 0）

## 8. 已知陷阱

### 8.1 构造函数提前计算度量（踩过多次）

见 `横切关注点/40-高DPI适配.md` 第 8.1 节。

### 8.2 estimateParagraphHeight 双重度量

见 `横切关注点/40-高DPI适配.md` 第 8.2 节（INV-4）。

### 8.3 子块 X 偏移丢失

列表 / 引用 / 表格的容器布局中，`moveLeft(offset)` 已经写入子块的 bounds，但绘制时若没读回会出现序号错位。本模块只负责"正确写入"，绘制侧由 `PreviewPainter` 负责"正确读回"。

### 8.4 setFont 忘记同步 m_monoFont（2026-04-13 发现）

历史实现：

```cpp
void PreviewLayout::setFont(const QFont& baseFont) {
    m_baseFont = baseFont;
    QFontMetricsF fm(m_baseFont);
    m_lineHeight = fm.height() * 1.5;
    // ← 没动 m_monoFont，保持构造函数写入的冷启动值
}
```

后果：

- `layoutBlock` 的 CodeBlock 分支计算 `m_codeLineHeight` 时用的是冷启动 `m_monoFont`
- `collectInlineRuns` 的 Code 分支虽然显式 `setPointSizeF(currentFont * 0.9)` 规避了此 bug，但每次都以 `m_monoFont` 的家族开始，字号被覆盖，等效于丢失与 `m_baseFont` 的派生关系
- Ctrl+滚轮缩放后代码块完全不缩

修复：`setFont` 中必须同步派生 `m_monoFont`（见 INV-8）。见横切关注点/80-字体系统.md §8.6、§8.7。
