# 内容标记（Marking）

## 1. 目的

让用户在预览区对段落/词语添加"荧光笔"式高亮，帮助阅读时定位重点内容。标记与 TOC 面板联动，显示每个标题下的已标记内容。

## 2. 输入 / 输出

- **输入**：
  - 用户选区（拖选或双击）
  - 用户操作（右键菜单"添加标记"/"清除标记"）
- **输出**：
  - 预览区的黄色荧光覆盖层
  - TOC 面板中已标记内容所在标题的指示

## 3. 行为契约

### [INV-1] 标记高亮先于选区高亮绘制

在 `PreviewPainter` 的绘制顺序中，`drawMarkings()` 必须在 `drawSelection()` 之前，否则选区会覆盖标记色。

### [INV-2] 标记数据格式

标记以「字符偏移范围」存储，不是以屏幕坐标：

```cpp
struct Marking {
    int startOffset;   // 源字符偏移（UTF-16 码元）
    int endOffset;
    QString text;      // 冗余存储，用于 TOC 显示
};
std::vector<Marking> m_highlights;
```

绘制时按字符偏移映射到当前布局的屏幕坐标。重排版后标记不丢失。

### [INV-3] 深浅主题需要两套颜色

深色模式下的荧光色要与亮色模式不同，避免在深色背景上看不清。由 `core::Theme` 提供：
- `theme.markingHighlight`
- `theme.markingHighlightDark`

### [INV-4] TOC 联动

标记保存后必须更新 `m_headingCharOffsets`，TOC 面板据此判断每个标题下是否有标记。

**[INV-4.1（2026-05-06 补充）]** `m_highlights` 与 `m_tocHighlighted` 必须**始终保持同步**：
任何清空 `m_highlights` 的路径（典型如 `updateAst` 中的 `m_highlights.clear()`，
每次编辑触发 `ParseScheduler::astReady` 都会进入），必须同步清空 `m_tocHighlighted`
并 `emit tocHighlightChanged({})`。否则 TocPanel 会残留上一次的高亮状态——预览区已无标记
但目录条目仍亮，与"无标记"现实不一致。`updateTocHighlights()` 在 `m_highlights` 重新填充
后会再次 emit 覆盖，是 idempotent 的，不必担心冗余。

### [INV-CHAR-OFFSET-CONSISTENT]（2026-05-06 新增）章节偏移与 plainText 字符流必须对齐

`PreviewWidget::buildHeadingCharOffsets` 累加 charIdx 的规则**必须与 `extractBlockText`
写入 m_plainText 的规则字节级一致**——否则 TOC 高亮按章节范围判定标记归属时，会因 char offset
错位指向**错误章节**。

历史 bug（#14 子场景 3）：旧 `buildHeadingCharOffsets` 没处理 Frontmatter 块——但
`extractBlockText` 把 frontmatter rawText + \n 写入 m_plainText。含 frontmatter 的文档下：
- m_plainText 长度 = frontmatter.rawText.length() + 1 + 章节文本长度
- m_headingCharOffsets 偏移 = 章节文本起点（少算了 frontmatter 长度）

用户标记某段内容（char offset 来自 paint 端 m_charCounter，**含** rawText），
`updateTocHighlights` 用错位的 m_headingCharOffsets 比对 → 标记落入错误章节。

**强约束**：任何新增 LayoutBlock 类型如果在 `extractBlockText` 中累加字符流，必须同步在
`buildHeadingCharOffsets` 中累加同样长度。两份代码理想上应抽公共 helper（未来工作）。

### [INV-MARK-EDIT-PRESERVE]（2026-05-06 新增）编辑时保留标记

`PreviewWidget::updateAst` **不得清空** `m_highlights`。该函数由 `ParseScheduler::astReady`
触发——用户每输入一字、防抖后都会进入此函数。如果在此处 clear，用户的标记会被「编辑」操作
反复抹除，与「持久化（INV-5）」矛盾——保存的标记重启后能恢复，但下一次编辑就丢，体感比
完全不持久化还坏。

清空时机由调用方显式处理：
- 创建新 `PreviewWidget`：m_highlights 默认空，无需 clear
- `promptReloadTab` / 重新 `loadFromFile` 同文件：用户期望保留标记（同文档内容更新）
- 用户主动右键「清除标记」：调 `clearHighlights()` 显式清

字符偏移可能因编辑漂移到错误位置——按 §8.3 容忍策略，paint 端按字符偏移自然忽略越界部分；
若标记跑偏到错误内容，由用户感知后手动清除。**不做主动迁移**，避免迁移算法引入不可预期行为。

未来工作（plan「标记应绑定段落」）：把锚点从全局字符偏移改为 (sourceLine, 段内偏移)，
让段间编辑不影响其他段的标记——是本 INV 的演进，不是替代。

### [INV-5] 持久化

标记必须在会话恢复时一并恢复（和 Tab 状态一起保存到 QSettings）。

## 4. 接口

```cpp
class PreviewWidget : public QAbstractScrollArea {
public:
    // 对选区添加标记
    void addMarking(const QString& text, int startOffset, int endOffset);

    // 清除所有标记
    void clearMarkings();

    // 只清除某个范围的标记
    void removeMarkingAt(int offset);

    // 序列化 / 反序列化（会话恢复）
    QByteArray serializeMarkings() const;
    void deserializeMarkings(const QByteArray& data);

private:
    std::vector<Marking> m_highlights;
    std::vector<int> m_headingCharOffsets;  // 标题的字符偏移（用于 TOC 联动）
};
```

## 5. 核心算法

### 标记的渲染

```
drawMarkings(p, block, absX, absY, highlights):
    for each marking in highlights:
        if marking 与 block 范围无交集: continue
        计算 marking 在本 block 中的屏幕矩形（可能跨多行）
        p.fillRect(rect, theme.markingHighlight 或 markingHighlightDark)
```

### 标记后的 TOC 更新

```
addMarking(text, start, end):
    m_highlights.append({start, end, text})
    rebuildHeadingMarkMap()
    emit markingsChanged()  → TOC 面板刷新
    saveSession()           → 写入会话状态
```

## 6. 性能预算

- 添加/删除标记后刷新 < 10 ms
- 绘制标记覆盖层 < 1 ms/块

## 7. 验收条件

- **[T-1]** 标记色与选区色同时出现时，标记可见（未被覆盖）
- **[T-2]** 深色模式下标记在深色背景上清晰可见
- **[T-3]** 添加标记后 TOC 面板立即显示该标题有标记指示
- **[T-4]** 重启应用后标记恢复到原位置
- **[T-5]** 修改文档后，标记字符偏移仍指向原内容（或至少不漂移到错误位置）
- **[T-6]** 清除标记后 TOC 指示同步消失

## 8. 已知陷阱

### 8.1 绘制顺序错位

标记绘制在选区之后会被选区覆盖。见 INV-1。

### 8.2 同色不可见

深色模式下使用亮色模式的黄色荧光会产生可读性问题。见 INV-3。

### 8.3 字符偏移漂移

文档大幅修改后，字符偏移可能指向错误位置。当前策略是「尽力而为」：文档修改时不主动迁移偏移，由用户在显示错乱时手动清除。

**会话恢复场景（2026-05-06 补充）**：从 QSettings 反序列化时执行三条容忍规则——`start < 0` 或 `end <= start` 的条目静默丢弃；`start` 已超出当前文档长度的条目静默丢弃；`end` 超出文档长度的条目把 `end` 夹紧到文末。损坏字节流（magic / version 不匹配）整体丢弃，不阻塞会话恢复。
