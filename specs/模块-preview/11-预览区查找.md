# 预览区查找功能

## 1. 目的

为预览区提供与编辑器对等的纯文本查找能力。用户在阅读长文档时不必再切回编辑器才能定位关键词。

## 2. 行为契约

### [INV-1] 搜索源 = 预览的纯文本提取

预览查找的字符流必须与 `PreviewWidget::extractPlainText()` 返回的字符串严格对齐。
此字符串已是用户视觉上看到的内容（去除 Markdown 标记、保留段落顺序），与
`m_plainText` 等价。

不允许重新做一遍 plainText 提取或基于 LayoutBlock 树独立遍历——必须复用 `m_plainText`
作为搜索 worker 的 `fullText` 输入，避免与选区/复制路径走样（参考反模式 B：同一语义
两份独立代码必然走样）。

### [INV-2] 不支持替换

预览区不可编辑——SearchBar 在预览区调用时**只显示搜索行，不显示替换行**。
通过 `SearchBar::showSearch()`（已有 API，默认不显示 replace 行）即可。

### [INV-3] 高亮叠加在用户标记之上

搜索高亮（`m_searchHits`）与用户标记（`m_highlights`）是两套独立数据集，由
`PreviewPainter` 分别绘制：
- 标记荧光笔：`theme.markerHighlightColor`（黄色或主题色派生）
- 搜索全部命中：`theme.searchHighlightColor`（次要色，与标记色明显不同）
- 当前命中：`theme.currentSearchHitColor`（主调色，比次要色更醒目）

绘制顺序：先画 m_highlights（标记），再画 m_searchHits（搜索）——重叠区域搜索高亮覆盖
标记，让用户看清当前搜索状态。

### [INV-4] 跳转语义

- `findNext()`：`m_currentSearchIndex = (m_currentSearchIndex + 1) % matches.size()`
- `findPrev()`：`m_currentSearchIndex = (m_currentSearchIndex - 1 + size) % size`
- 跳转时滚动让当前命中项可见——精度到 LayoutBlock 级别（找到包含 charOffset 的最深块，
  scroll 到 block.bounds.y 减屏幕中央偏移）
- 没有命中时 `findNext/Prev` 不动作，UI 显示 0/0

### [INV-5] 关闭语义

- Esc 键关闭 SearchBar：`m_searchHits` 清空、`m_currentSearchIndex = -1`、viewport
  update 让搜索高亮消失
- SearchBar `closed` 信号触发同样路径

### [INV-6] Ctrl+F 路由：焦点优先 + 可见性兜底

主窗口 Ctrl+F 快捷键路由策略（焦点判断 + 可见性兜底）：

1. 焦点在 PreviewWidget 子树**且** preview 可见 → preview
2. 焦点在 EditorWidget 子树**且** editor 可见 → editor
3. 否则按可见性默认：editor 可见 → editor；只有 preview 可见 → preview
4. 两侧都不可见 → 静默 no-op（理论 currentTab 非空时至少一侧可见）

**为何不只用焦点判断**：用户可能在 TOC / 资源管理器 / 其他面板获焦后按 Ctrl+F，
焦点判断会失败但用户预期"Ctrl+F 必然唤起搜索"。可见性兜底保证只要有打开文档
且任一区域可见，Ctrl+F 一定响应（2026-05-11 第三轮修复）。

**为何编辑器优先**：编辑器是默认主交互区，绝大多数用户场景在编辑器查找；
若需搜预览，用户可先单击预览区（设 focus）再 Ctrl+F，或在纯预览模式下自动走
preview 兜底。

预览区获焦的标准方式：用户单击预览区任何位置 / Tab 键焦点切换。

**关键约束**：`PreviewWidget::mousePressEvent` 必须在处理左键事件时显式调用
`setFocus()` —— QAbstractScrollArea 的鼠标事件分发到 viewport（默认 NoFocus），
若不显式 setFocus，单击预览不会让 PreviewWidget 自身获焦，Ctrl+F 路由的
`focusInPreview` 检查永远为 false。`EditorWidget::mousePressEvent` line 401 已有
此处理；预览侧必须对齐（反模式 B：QAbstractScrollArea 焦点处理在两侧本应一致
但走样）。

### [INV-7] i18n

新增字符串（占位符 / 提示 / 按钮 tooltip）必须 `tr()` 包裹，并同步
`translations/simple_markdown_zh_CN.ts` 和 `translations/simple_markdown_en_US.ts`。

如能复用 SearchBar 现有字符串（搜索框 placeholder、按钮 tooltip 等）则不重复登记。

### [INV-8] 与拖拽态守护互不干扰

预览区在 SnapSplitter 拖拽期间收到的 resizeEvent 仍按 INV-SNAP-LAZY-PANE-REBUILD
跳过 rebuildLayout——SearchBar 几何在 dragFinished 后由 PreviewWidget 重新调整
位置即可（与编辑器侧 SearchBar 对齐）。

### [INV-9] 编辑器/预览查找栏互斥

任意时刻**最多一侧** SearchBar 可见——避免双栏并存导致用户困惑（"我现在在哪一侧搜
什么"）。

实现：
- `EditorWidget::hideSearchBar()` 和 `PreviewWidget::hideSearchBar()` 都调用各自
  `m_searchBar->hideBar()`，触发 closed 信号 → 清空对应侧 m_searchHits/m_searchMatches
  + 当前命中索引 + 搜索文本
- `MainWindow` Ctrl+F 路由 lambda：决定目标侧后，**先**调用非目标侧的 hideSearchBar，
  **再**调用目标侧的 showSearchBar——保证打开时另一侧及其搜索高亮一定已清

副作用：用户在 A 侧搜索后切焦点到 B 侧再 Ctrl+F，A 侧搜索状态丢失（需重新输入）。
此处主动选择"互斥 + 状态清"而非"互斥 + 状态保留隐藏"，原因：保留隐藏会让 A 侧搜索
高亮残留在 paintEvent 输出中（painter 仍接收 m_searchHits），视觉混乱比少做一次保留
更糟。

判定路由的焦点检查不变（沿父链找 PreviewWidget），但本 INV 让最终 UI 状态不再
依赖"焦点判断 100% 准确"——即使焦点系统因边界 case 路由错了，也只显示单一栏，用户
看到不对就 Ctrl+F 再切一次。

### [INV-10] 跳转后必须 keepFocus 回 SearchBar 输入框

`findNextHit / findPrevHit` 的**所有返回路径**都必须调用 `m_searchBar->keepFocus()`，
让 `m_findEdit` 重新获焦。

原因：`SearchBar::eventFilter` 对 `m_findEdit` 装了 `FocusOut` → 100ms 延迟
`hideBar()` 的自动隐藏（src/editor/SearchBar.cpp:143-148）。当用户用鼠标点击
SearchBar 的"上一个/下一个"工具按钮（按钮非子 QWidget，是自绘的 ToolButton），
鼠标按下让 `m_findEdit` 失焦 → 触发 FocusOut → 若 100ms 内焦点没回来 → 自动关闭
搜索栏。

编辑器侧 `EditorWidget::findNext / findPrev` 每条返回路径都调 `keepFocus()`，预览侧
必须对齐——否则点按钮即关闭搜索栏，用户感知到的现象是"搜索框消失"。

参考反模式 B：同一语义两份独立代码必然走样。

### [INV-11] 输入后立即回车必须能搜出结果（不依赖 debounce）

`findNextHit / findPrevHit` 入口的早返回条件**只能是** `text.isEmpty()`——禁止再加
`m_searchHits.isEmpty()` 等屏蔽条件。

原因：`onSearchTextChanged` 走 100ms debounce → worker 异步搜索 → onSearchResultsReady
回填 `m_searchHits`。用户输入完文本立即按回车时（< 100ms），m_searchHits 仍为空。
若入口处因此早返回，用户看到 0/0 且无任何跳转——表现为"文档里明明有这个词，但搜不到"。

正确语义（对齐编辑器侧 EditorWidget::findNext）：
- 入口只过滤空文本
- 同步重搜触发条件为 `text != m_currentSearchText || m_searchHits.isEmpty()`——
  hits 空也会兜底同步重搜
- 同步重搜后立即 `m_searchDebounce.stop()` + `++m_searchRequestId` 让异步路径回填
  时被 requestId 比对（onSearchResultsReady 已有的过期保护）自然丢弃，避免覆盖
  同步算出的结果

### [INV-12] 首次跳转按可视区域定位，后续连续跳转按索引循环

跳转语义分两种状态：

1. **首次跳转**（`m_currentSearchIndex == -1`）：基于 `verticalScrollBar()->value()`
   选起点——`findNextHit` 取第一个 `hit.y > scrollY` 的命中（都不满足回绕首项），
   `findPrevHit` 取最后一个 `hit.y < scrollY` 的命中（都不满足回绕末项）。
   `m_currentSearchIndex` 在 `onSearchResultsReady` 回填时被设为 -1（line 1097），
   表示"用户尚未按 Enter / F3 锁定任何一项"——此时必须用位置语义，否则永远从第 1
   项开始（旧 bug，2026-05-11 修复）。

2. **后续连续跳转**（`m_currentSearchIndex >= 0`）：使用纯索引循环——
   `findNextHit` 取 `(currentIndex + 1) % size`，`findPrevHit` 取
   `(currentIndex - 1 + size) % size`。

**为什么后续不能继续用位置语义**：当用户按下"下一个"后，`scrollToCharOffset`
把命中滚动到 `scrollY = hit.y - vpH/3`——即**当前命中本身 hit.y > scrollY**。
若再点"上一个"仍用"找 hit.y < scrollY"，遇到所有命中集中在文档一段的密集场景
（典型如 7 个命中都在中下部）会**全部跳过**回绕到末项，表现为 7/7 → 6/7 → 又 7/7
的诡异循环（2026-05-11 复测复现）。

索引循环与编辑器侧 cursor 语义等价：编辑器每次 findNext 更新 cursor 到新命中，
下次基于新 cursor 找下一个 = 单调推进 = 索引 +1。预览没有 cursor，但有
`m_currentSearchIndex`，作用相同。

实现上首次跳转的位置定位单次块树遍历即返回，复杂度 O(N 块) × O(K 命中)，
小于一次 paintEvent，性能可接受。

### [INV-13] 禁止 FocusOut 自动隐藏机制

`SearchBar` **禁止**实现任何"m_findEdit 失焦后延迟检查 → 自动 hideBar"机制。

历史教训（2026-05-11 一日三连击）：
- v1：`QTimer::singleShot(100, ...)` 每次 FocusOut 都启动新 timer → 连续按按钮
  堆叠多个 timer，窗口期内误触 hideBar
- v2：改单实例成员 timer + keepFocus stop → 仍复现，密集交互下任何延迟检查在
  m_findEdit 短暂失焦的事件链中都会误判
- v3（当前）：彻底删除 FocusOut 自动隐藏——SearchBar 只在以下三种**显式**路径
  关闭：
  1. m_findEdit / m_replaceEdit 收到 Esc 键（eventFilter 中处理）
  2. 用户点击 `m_btnClose` 关闭按钮
  3. 上层（MainWindow 互斥路由 / 调用方业务）主动调 `hideSearchBar()`

**为何 FocusOut 自动隐藏不可靠**：用户的"密集点击工具按钮"在 Qt 事件队列中产生
`mousePressEvent → emit findNext → 上层槽函数（含 setSelection / ensureCursorVisible
/ scrollToCharOffset）→ keepFocus` 的串。其中任一帧 m_findEdit 的 `hasFocus()`
可能短暂返回 false——若此时 100ms 延迟检查到期，hideBar 被错误触发。

**取舍**：用户点击编辑器其他位置后搜索栏不再自动消失——但与"打开搜索栏直到主动
关闭"的直觉一致，Esc / 关闭按钮显式关闭即可，无心智负担。

### [INV-15] 点击 SearchBar 外部自动关闭（坐标判断）

SearchBar 在 `showSearch` / `showReplace` 末尾 `qApp->installEventFilter(this)`；
`hideBar` 顶部 `qApp->removeEventFilter(this)`。

`eventFilter` 增加 `QEvent::MouseButtonPress` 分支——比较 `me->globalPos()` 与
`QRect(mapToGlobal(QPoint(0,0)), size())`：
- 在范围内 → SearchBar 自己被点击（输入框 / 工具按钮 / 间隙）→ 不动作
- 范围外 → `QTimer::singleShot(0, ..., hideBar)` 延迟到下个事件循环关闭，不
  消费事件（让目标 widget 仍能正常处理点击）

**为何用坐标而非 focus**：FocusOut 在密集按钮交互场景下不可靠（INV-13 历史教训），
而 MouseButtonPress 是用户**真实操作**的同步事件——坐标判断结果确定，不会受
事件链时序影响。SearchBar 自绘按钮区域天然在 SearchBar widget 几何内，按钮点击
不会触发外部判定。

**与 INV-13 关系**：INV-13 禁止 FocusOut 延迟检查 hasFocus 自动隐藏；INV-15 是
"点击外部即关"的稳健替代——同一个用户需求（搜索栏在用户失去注意力时收纳），
不同的实现策略。

**hideBar 不可无条件 setFocus(parent)**：`SearchBar::hideBar` 末尾**仅当**焦点
仍在 m_findEdit / m_replaceEdit 时才 `parentWidget()->setFocus()`——Esc / 关闭按钮
关闭时焦点在搜索栏内部，需送回宿主；INV-15 点击外部触发时焦点已在新 widget
（用户单击的目标），不能再抢回——否则会出现"用户从 A 侧搜索栏切到 B 侧，点击
B 侧让 B 获焦 + A SearchBar.hideBar 抢回 A 焦点 + 用户 Ctrl+F 仍弹 A"的反向 bug
（2026-05-11 第六轮用户复测复现）。

### [INV-14] SearchBar 主题必须由宿主同步

宿主 Widget（EditorWidget / PreviewWidget）的 `setTheme(Theme)` **必须**调用
`m_searchBar->setTheme(theme)`——`SearchBar` 自身不订阅全局主题，它的 `Theme m_theme`
默认构造为浅色空主题，若宿主不主动同步，搜索栏的输入框背景 / 边框 / 文字色 / 按钮色
会与宿主和应用主题脱节，表现为"两侧搜索栏背景颜色不一致"或"暗色主题下出现一道
亮色搜索栏"。

历史教训（2026-05-11 第四轮）：
- `EditorWidget::setTheme` line 196 同步给 m_searchBar 没问题
- `PreviewWidget::setTheme` 历史上漏了这一行，导致预览侧搜索栏一直是 SearchBar
  默认浅色主题——之前 INV-9 互斥（一次只一侧可见）掩盖了对比；本轮第三轮把 Ctrl+F
  改"焦点优先 + 可见性兜底"路由后用户得以连续切换看两侧搜索栏，差异即可见

新增对称性约束：任何**复用 SearchBar 的新宿主** Widget 必须在自己的 setTheme 中
同步 m_searchBar->setTheme(theme)，建议 grep 检查项：

```
grep -n "m_searchBar->setTheme" src/**/*.cpp
# 应等于"复用 SearchBar 的 Widget 数量"
```

## 3. 接口

```cpp
class PreviewWidget : public QAbstractScrollArea {
public:
    // 弹出查找栏并聚焦输入框（仅查找，不显示替换行）
    void showSearchBar();

    // 关闭查找栏并清空搜索高亮
    void hideSearchBar();

    // 查询当前是否有搜索栏可见（供 MainWindow 决定路由）
    bool isSearchBarVisible() const;

private:
    SearchBar* m_searchBar;
    QThread m_searchThread;
    SearchWorker* m_searchWorker;
    QTimer m_searchDebounce;
    int m_searchRequestId = 0;
    QVector<QPair<int,int>> m_searchHits;  // (charOffset, length)
    int m_currentSearchIndex = -1;
    QString m_currentSearchText;

    // slots
    void onSearchTextChanged(const QString& text);
    void onSearchResultsReady(QVector<QPair<int,int>> matches, int requestId);
    void findNextHit();
    void findPrevHit();
    void scrollToCharOffset(int offset);
};
```

## 4. 验收条件

- **[T-1]** 预览区获焦后 Ctrl+F 弹出查找栏，输入文本即时高亮全部命中（300ms 防抖与
  编辑器侧一致）
- **[T-2]** F3 / Shift+F3 上下跳转，命中项滚动到可见区域
- **[T-3]** 关闭查找栏（Esc 或关闭按钮），搜索高亮立即消失
- **[T-4]** 大小写敏感、整词匹配开关切换正常工作
- **[T-5]** 搜索高亮与用户标记荧光笔颜色明显不同，重叠时搜索高亮可见
- **[T-6]** 演示模式 / 单预览模式下查找仍可用
- **[T-7]** 编辑器和预览同时打开时，Ctrl+F 按当前焦点路由到对应侧
- **[T-8]** 拖拽 splitter 期间 SearchBar 位置不抖动（与编辑器侧一致策略）
- **[T-9]** 中英文 i18n 字符串完整
- **[T-10]** 编辑器/预览查找栏互斥（INV-9）：
  - 在预览侧 Ctrl+F 打开搜索栏 → 切焦点到编辑器 → Ctrl+F → 预览搜索栏自动关闭，
    编辑器搜索栏打开（且预览搜索高亮一并消失）
  - 反方向同样成立（编辑器侧 Ctrl+F → 切预览 → Ctrl+F → 编辑器搜索栏关闭，预览栏打开）
- **[T-11]** keepFocus 一致性（INV-10）：预览侧搜索栏弹出后，鼠标点击工具按钮
  "上一个"/"下一个"，搜索框**不消失**，并正确跳转到对应命中（与编辑器侧行为一致）
- **[T-12]** 输入后立即回车（INV-11）：预览侧 Ctrl+F → 输入"关键词" → 立即按回车
  （< 100ms，先于 debounce 触发）→ 正确显示命中数并跳转到首个命中。等价测试：
  关闭后再次 Ctrl+F 重复——每次回车都能即时命中
- **[T-13]** 首次跳转按位置 + 后续按索引（INV-12）：
  - 滚动预览到文档中部 → Ctrl+F 搜索高频词 → 首次回车 → 跳转到第一个
    `y > 当前 scrollY` 的命中（不是第一项）
  - 此后**连续**按"下一个"或 F3 → 索引依次 +1，到末尾后回绕首项；不依赖当前 scrollY
  - 在 7/7（末项）按"上一个" → 6/7；**再**按"上一个" → 5/7（不能回到 7/7）；
    继续按到 1/7 后再按"上一个"回绕到 7/7
- **[T-14]** 禁止 FocusOut 自动隐藏（INV-13）：编辑器和预览搜索栏可见后
  **连续快速**点击"下一个"工具按钮 10 次（间隔 < 50ms），搜索栏始终保持可见；
  用户单击编辑器/预览的任意内容区域后，搜索栏仍**不会**消失——只有
  Esc / 关闭按钮 / Ctrl+F 切换到另一侧时才关闭
- **[T-15]** Ctrl+F 可见性兜底（INV-6 v2）：用户焦点在 TOC 面板 / 资源管理器 /
  其他非 editor/preview 区域时按 Ctrl+F——若 editor 可见 → 唤起 editor 搜索栏；
  若仅 preview 可见（纯预览模式）→ 唤起 preview 搜索栏（不再因焦点不在 editor/preview
  而 no-op）
- **[T-16]** 点击外部自动关闭（INV-15）：
  - 编辑器/预览搜索栏打开 → **点击编辑器/预览内容区域** → 搜索栏关闭，搜索高亮清除
  - 搜索栏打开 → **点击搜索栏自己的输入框/按钮/空白间隙** → 不关闭，正常交互
  - 搜索栏打开 → **点击工具栏 / 菜单栏 / TOC 面板** → 关闭
  - 搜索栏打开 → **连续快速点击"下一个" 30 次** → 不关闭（点击位置始终在 SearchBar 矩形内）
- **[T-17]** 单击切换区域后 Ctrl+F 正确路由（INV-6 setFocus 对称 + INV-15 不抢焦点）：
  - 编辑器 Ctrl+F → 单击预览（任意位置）→ Ctrl+F → 搜索栏正确切到预览区
  - 预览 Ctrl+F → 单击编辑器（任意位置）→ Ctrl+F → 搜索栏正确切到编辑器
  - 不依赖双击 / 选区操作来"激活"焦点转移

## 5. 已知限制

- 不支持替换（INV-2）
- 不支持正则（编辑器侧也未广泛使用，统一以"无正则"作为 V1 范围）——若 SearchBar
  正则开关已暴露则保留可用，但不在本 spec 验收范围
- char offset → Y 跳转精度到 LayoutBlock 级别，不到字符级（V1 简化）
